    Public exports: `lookupMojangSkin(username: string): Promise<MojangSkinResult>` and the `MojangSkinResult` interface (resolvedUsername, pngBytes, pngBase64, sha256, textureUrl, model).

    Constants: `const TIMEOUT_MS = 15_000;` — exactly 15 seconds per CONTEXT, no fudging.

    Implementation outline:

    1. Helper `fetchWithTimeout(url, ms)`: wraps `fetch(url, { signal: ac.signal, headers: { 'user-agent': 'sei-electron/0.1.0' } })` with an `AbortController` that fires after `ms`. Returns the Response. Clears the timer in finally.

    2. Helper `classify(err, stage)`: returns `new Error(`MOJANG_LOOKUP_FAILED: ${stage}: ${msg}`)`. Caller re-throws.

    3. Public `lookupMojangSkin(username)`:
       - Trim input. Throw `MOJANG_LOOKUP_FAILED: empty username` if empty. Reject anything not matching `/^[A-Za-z0-9_]{1,32}$/` with `MOJANG_LOOKUP_FAILED: invalid characters in username`.
       - **Step 1 (username → UUID):** `GET https://api.mojang.com/users/profiles/minecraft/${encodeURIComponent(trimmed)}`. Branches:
         * HTTP 204 or 404 → throw `MOJANG_LOOKUP_FAILED: no Minecraft account named ${trimmed}`
         * HTTP 429 → throw `MOJANG_LOOKUP_FAILED: Mojang rate-limited the lookup. Wait a minute and try again.`
         * !ok → throw `MOJANG_LOOKUP_FAILED: api.mojang.com responded ${status}`
         * 200 → parse json; require `typeof id === 'string' && typeof name === 'string'`; assign `uuid = id; canonicalName = name`.
       - **Step 2 (UUID → texture URL):** `GET https://sessionserver.mojang.com/session/minecraft/profile/${encodeURIComponent(uuid)}`. Branches:
         * 429 → throw `MOJANG_LOOKUP_FAILED: Mojang rate-limited the profile lookup. Wait a minute and try again.`
         * !ok → throw `MOJANG_LOOKUP_FAILED: sessionserver responded ${status}`
         * 200 → parse json; find `prop = properties.find(p => p.name === 'textures')`; throw `MOJANG_LOOKUP_FAILED: sessionserver returned no textures property` if missing or non-string `value`.
         * Decode: `decoded = JSON.parse(Buffer.from(tex.value, 'base64').toString('utf8'))`. Read `decoded.textures.SKIN.url`. Throw `MOJANG_LOOKUP_FAILED: profile has no skin set` when missing.
         * Read `decoded.textures.SKIN.metadata?.model`. If `=== 'slim'` set `model = 'slim'`; otherwise default to `'classic'`.
       - **Step 3 (PNG download):** `GET textureUrl`. Branches:
         * !ok → throw `MOJANG_LOOKUP_FAILED: textures.minecraft.net responded ${status}`
         * 200 → `pngBytes = Buffer.from(new Uint8Array(await r.arrayBuffer()))`. Verify PNG magic: `pngBytes[0..3] === 0x89 0x50 0x4E 0x47`. Throw `MOJANG_LOOKUP_FAILED: texture endpoint did not return a PNG` on mismatch.
       - Return `{ resolvedUsername, pngBytes, pngBase64: pngBytes.toString('base64'), sha256: crypto.createHash('sha256').update(pngBytes).digest('hex'), textureUrl, model }`.

    4. Every catch that did NOT originate from our own throw should re-throw via `classify(err, '<stage>')` to preserve the MOJANG_LOOKUP_FAILED prefix. Check `if (err instanceof Error && err.message.startsWith('MOJANG_LOOKUP_FAILED:')) throw err;` before each classify call so user-facing strings don't get double-prefixed.

    Three sequential awaits each carrying their own AbortController-backed 15s budget. Worst-case wall-clock = 45s; ok because the renderer just shows a "Searching skin..." indicator and can be cancelled by navigating away (Plan 05's UI cleanup unmounts the search field).

    Every throw starts with `MOJANG_LOOKUP_FAILED:` so the renderer's `classifyRendererError` and `ERROR_COPY[MOJANG_LOOKUP_FAILED]` both surface the same copy without needing a new heuristic. Plan 05's UI catches the rejected promise and shows `ERROR_COPY.MOJANG_LOOKUP_FAILED` directly.

    The 204/404 branch is the "no such user" case (RESEARCH §4 HTTP 204 trap) — treat as MOJANG_LOOKUP_FAILED with a specific suffix the UI can match for distinct copy.

    Do NOT cache the result — Plan 02's `applyPng` writes the PNG once the user clicks "Apply", and from then on the local file is the source of truth (no persistent dependency on Mojang per CONTEXT §decisions).

    Public export: `openSkinPicker(): Promise<SkinUploadResult | null>` where `SkinUploadResult = { pngBase64: string; sha256: string }`. Null when user cancels.

    Implementation outline:

    1. Resolve modal parent: `const win = BrowserWindow.getFocusedWindow() ?? BrowserWindow.getAllWindows()[0];`. The focused window is the modal parent; the fallback covers the rare case where focus is on a non-Electron app at click time.

    2. Call `dialog.showOpenDialog(win, { title: 'Choose a 64x64 Minecraft skin PNG', properties: ['openFile'], filters: [{ name: 'PNG Images', extensions: ['png'] }] })`. If `win` is undefined, call the no-parent overload `dialog.showOpenDialog({ ... })` with the same options.

    3. If `result.canceled || result.filePaths.length === 0`, return null. Otherwise read `bytes = await readFile(result.filePaths[0])`. On read error throw `SKIN_FILE_INVALID: could not read selected file: ${err.message}`.

    4. PNG magic + IHDR validation (identical to Plan 02 applyPng — copy/paste rather than import to keep skinUpload self-contained for the renderer file-dialog path):
       - `bytes.length >= 24`
       - `bytes[0..3] === 0x89 0x50 0x4E 0x47`
       - Parse width=`bytes.readUInt32BE(16)`, height=`bytes.readUInt32BE(20)`
       - Require `width === 64 && height === 64`
       - Throw `SKIN_FILE_INVALID: ...` with a specific suffix per failure (magic mismatch / dimension mismatch with actual w,h).

    5. Return `{ pngBase64: bytes.toString('base64'), sha256: crypto.createHash('sha256').update(bytes).digest('hex') }`.

    All throws prefix `SKIN_FILE_INVALID:` so the renderer's classifier picks them up consistently. The picker filters to `*.png` at the OS level, but we still validate magic + IHDR because OS filters can be bypassed and a `.png`-extensioned non-PNG can exist on disk.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Main process → Mojang public API | Outbound HTTPS to api.mojang.com + sessionserver.mojang.com + textures.minecraft.net. Untrusted bytes returned. All parsed defensively (typeof checks, magic-byte verification, dimension guard runs only in `applyPng` downstream because Mojang skins can be 64x32 OR 64x64; we accept ANY valid PNG here and let `applyPng` reject if not 64x64) |
| Main process → user filesystem | `dialog.showOpenDialog` returns an absolute path; main reads via `fs/promises.readFile`. Path is user-chosen so directory traversal is irrelevant (the user is choosing their own file) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-T2 | T (Tampering) | Mojang response chain | mitigate | Each step verifies the shape it expects (`typeof id === 'string'`, base64-decode + JSON.parse with try/catch, PNG magic check). Malformed responses throw `MOJANG_LOOKUP_FAILED:` before bytes are persisted. |
| T-09-I2 | I (Information Disclosure) | Mojang username lookup | accept | Submitting a username to Mojang's public API is the documented use case; no Sei-identifying info beyond the `user-agent: sei-electron/0.1.0` header. No keys, no tokens, no PII. |
| T-09-D2 | D (Denial of Service) | runaway Mojang fetch | mitigate | Each of the three sequential fetches has a 15s AbortController-backed timeout. Worst-case wall-clock 45s; renderer UI shows "Searching skin..." indicator. |
| T-09-T3 | T (Tampering) | user-chosen PNG via file dialog | mitigate | Magic-byte + IHDR width/height parse rejects non-PNG and non-64x64 at the `skinUpload` boundary. `applyPng` re-validates downstream so a bypass at this layer still cannot land an invalid PNG in `<userData>/skins/`. |
| T-09-S2 | S (Spoofing) | textures.minecraft.net hostname pinning | accept | A MITM attacker on the user's network could rewrite the texture URL during the sessionserver response. We do NOT pin certificates — accept the risk; the bytes still pass magic-byte validation, and the user can re-search if the skin looks wrong. Plan 09 documentation surfaces this in the security notes. |
</threat_model>
