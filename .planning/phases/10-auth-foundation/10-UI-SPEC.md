# Phase 10 — UI Design Contract

> Visual and interaction contract for the Auth Foundation phase. Generated by gsd-ui-researcher from established renderer design system (`src/renderer/src/styles/tokens.css`) + CONTEXT.md decisions D-01..D-14.

**Inheritance posture:** This is an established codebase. UI-SPEC ALIGNS with the v0.1.1 design system (CSS Modules + token-driven, sharp corners D-28, borderless underline inputs D-31) — it does **not** invent new tokens, fonts, or component primitives. Phase 10 adds *new screens/modals* that compose existing primitives (`Button`, `TextField`, `Banner`, `DeleteConfirmModal` pattern, `WizardStepShell`, scrim+modal scaffold from `SetupWizardModal`).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (CSS Modules + custom tokens) |
| Preset | not applicable (no shadcn — sharp-corner brand identity D-28 is intentional and incompatible with shadcn defaults) |
| Component library | none — bespoke primitives in `src/renderer/src/components/` |

| Corner radius | **0 everywhere** (D-28). Native input/button radii are reset in `global.css`. Do not introduce a single `border-radius > 0` in this phase. |

## Spacing Scale

Established token scale from `tokens.css` (8-point with 4pt half-step). Phase 10 reuses these — do not introduce new values.

> **Inheritance note:** 12px / 40px / 56px are intentional half-step (`--space-md`) and extended (`--space-2xl`, `--space-3xl`) values from the established 8-point-with-4pt-half-step system, inherited verbatim from `tokens.css`. They are not new Phase 10 introductions; they predate this phase and exist for footer-button rhythm, modal padding, and screen-section breathing room respectively.

| Token | Value | Usage in Phase 10 |
|-------|-------|-------------------|
| `--space-xs` | 4px | Icon-to-text gap inside Banner dismiss, focus outline offset |
| `--space-sm` | 8px | Compact element gaps (e.g., Sign In / Continue Locally button group inside modal footer) |
| `--space-md` | 12px | Footer button gaps (matches DeleteConfirmModal), Banner internal vertical padding |
| `--space-md-plus` | 16px | Body margin-bottom, section gaps, tile internal padding row |
| `--space-lg` | 24px | Modal body bottom margin, AuthChoice tile vertical padding |
| `--space-xl` | 32px | Modal padding (matches DeleteConfirmModal `.modal { padding: 32px }`) |
| `--space-2xl` | 40px | SetupWizardModal-style padding for SignInModal `padding: 24px 40px 40px` |
| `--space-3xl` | 56px | AuthChoice screen vertical rhythm between brand mark and tile pair |

**Exceptions:** None for Phase 10. All new spacing values MUST resolve to one of the tokens above.

## Typography

Exactly **four** font sizes for all Phase 10 surfaces. Role differentiation comes from weight, letter-spacing, text-transform, and color — never new sizes. Inherited from `tokens.css` and codified by existing components.

> **Inheritance note:** 14px and 15px form the intentional "fine-grain" tier — paragraph body (15px) sits 1px above helper / label / secondary text (14px). Both are inherited verbatim from existing components (`DeleteConfirmModal.module.css` body at 15px; `SettingsScreen.module.css`, `Banner.module.css`, `Button.module.css` secondary/label text at 14px). The 1px gap is deliberate density tuning — not an accidental near-duplicate.

| Size | Roles | Weight | Line Height | Letter Spacing | Transform / Color | Used For |
|------|-------|--------|-------------|----------------|-------------------|----------|
| 14px | Mono section label | 400 (mono) | 1.0 | 1.2px | `uppercase`, `var(--muted)`, `var(--mono)` font-family | Settings "ACCOUNT" section header (mono caps preserves the small-label feel previously held by 11px) |
| 14px | Secondary / helper / micro-copy | 400 | 1.45 | 0 | `var(--text-2)` or `var(--muted)` | Banner copy, modal small print, OAuth error one-liners, mode-toggle links, forgot-password link, framing micro-copy, footer micro-copy, helper text under Settings rows, auto-dismiss countdown caption, OR-divider word |
| 14px | Row label / form label | 600 | 1.4 | -0.1px | `var(--text)` | Settings row labels, form field labels (Email, Password) |
| 15px | Body | 400 | 1.5 | 0 | `var(--text-2)` (or `var(--text)` for emphasized body) | Modal body paragraphs (matches `DeleteConfirmModal .body`), tile descriptions, sub-line copy |
| 22px | Heading | 600 | 1.2 | -0.2px | `var(--text)` | Modal titles (matches `DeleteConfirmModal .title`), tile headings |
| 30px | Display (screen title) | 600 | 1.15 | -0.4px | `var(--text)` | AuthChoice screen title, Settings page title (matches `SettingsScreen.module.css .title`) |

**Distinct sizes:** 14px, 15px, 22px, 30px — **four total**.

**Notes:**
- No new font sizes may be introduced. If a Phase 10 surface needs a size not on this table, reuse the closest existing one.
- The 14px size carries three distinct roles via differentiation: (a) mono-uppercase 400 with 1.2px tracking for section labels — visually small-feeling despite the 14px box; (b) sans 400 for helper/secondary text; (c) sans 600 for labels. CSS Modules MUST express the difference via weight/transform/letter-spacing, not via a new size.
- `Press Start 2P` is reserved for: brand mark (`SeiPixelMark`) and any step indicators inside `WizardStepShell`. **Do not** use `--pixel` for body copy in Phase 10.

### Accent reserved for (explicit allowlist, Phase 10 only)

The peach (light) / blue (dark) accent in this phase appears on these elements **and nowhere else**:

1. **Focus rings** — `outline: 1.5px solid var(--accent)` on every focusable element (standard from `Button.module.css`).
2. **TextField active underline** — `border-bottom-color: var(--accent)` on `:focus` (standard from `TextField.module.css`).
3. **One primary CTA per modal** — `kind="accent"` Button on the modal's main affirmative action (the "Sign In" button inside SignInModal, the "Continue in your browser" button inside the OAuth interstitial). All other buttons are `kind="primary"`, `"ghost"`, or `"quiet"`.
4. **AuthChoice tile hover/focus tint** — `background: var(--accent-soft)` on the focused/hovered tile (light, semi-transparent — see Component Inventory below).
5. **"Verify your email" Banner background** — uses `var(--accent-soft)` via `Banner kind="warn"` (matches existing Banner `.warn` rule).

**Forbidden uses (will fail UI-check):** accent on resting-state buttons that aren't the page's primary CTA, accent on icons, accent on text, accent on the dismissal button of any modal, accent on inactive tabs.

### Destructive reserved for

- **Delete Account** button in Settings Account panel (`background: var(--red); color: var(--window)` — matches `DeleteConfirmModal.module.css .deleteBtn`).
- **Type-email-to-confirm input** when typed string is non-empty AND doesn't match (`border-bottom-color: var(--red)`, matches `TextField[aria-invalid='true']`).
- **OAuth error Banner** uses `kind="error"` (white-on-red).

No other destructive surfaces in Phase 10.

## Layout & Composition Rules

1. **AuthChoiceScreen sits inside `MacosWindow`** — no chromeless full-bleed. The window chrome and traffic-light buttons remain visible, identical to OnboardingScreen.
2. **Modal width is 460px** for every Phase 10 modal except the OAuth interstitial, which is also 460px. **Never 680px** (that width is reserved for `SetupWizardModal`). Rationale: auth dialogs are decisional, not exploratory.
3. **Scrim alpha 0.45** — every Phase 10 modal scrim uses `rgba(0, 0, 0, 0.45)` to match `DeleteConfirmModal` and `SetupWizardModal`. No custom scrim alphas.
4. **ESC closes** every Phase 10 modal except: (a) `DeleteAccountModal` while a delete request is in-flight, (b) `OAuthInterstitialModal` while the loopback callback is being exchanged (~1s window after the user returns).
5. **Click-outside closes** SignInModal and SignOutConfirmModal; does **not** close DeleteAccountModal (require explicit `Keep my account` — destructive-action friction) or OAuthInterstitialModal (require explicit `Cancel sign-in` — prevents accidental dismissal while user is in their browser).
6. **Focus management:** on modal open, focus the first interactive element (Email field in SignInModal, `Cancel sign-in` button in OAuthInterstitial, type-to-confirm input in DeleteAccountModal). On close, return focus to the trigger element.
7. **Banner stacking:** when both `VerifyEmailBanner` and `LinuxKeyringBanner` are showing, VerifyEmail renders **first** (top), Keyring **second**. Both stack above the main content with `border-bottom: 1px solid var(--border-strong)` between them (already in Banner CSS).
8. **Danger Zone visual treatment** (Settings → Account row 4): separated by `border-top: 1px solid var(--border-strong)` (heavier than the regular `.section` border-top of `--border`) + `padding-top: var(--space-lg)`. The label sets `color: var(--red)` inline. This is the established "GitHub Danger Zone" feel per CONTEXT.md `<specifics>`.

## Interaction Contracts

### AuthChoice → SignInModal vs OnboardingScreen routing
- Click **Sign In tile** → opens `SignInModal` overlaid on AuthChoice (AuthChoice stays mounted underneath, slightly visible through 0.45 scrim). `Back to Sei` returns to AuthChoice (the originating surface).
- Click **Continue Locally tile** → AuthChoice unmounts → `OnboardingScreen isReonboard={false}` renders. No modal animation; standard screen transition.

### Signed-in user reaches OnboardingScreen (D-03)
- `OnboardingScreen` accepts new prop `signedIn?: boolean`. When true, the 4-step flow becomes **3-step**: Welcome, Minecraft username, Preferred name — the Provider tiles step AND the API key step are both skipped. The cloud-AI placeholder ("Cloud AI — coming soon") appears as a small `--muted` line in the Welcome step body: `Cloud-hosted AI is launching in a future update. For now, you can paste your own provider key in Settings.`

### OAuth flow (D-05)
1. User clicks `Continue with Google` in SignInModal.
2. Main process opens loopback server + `shell.openExternal(authUrl)`.
3. SignInModal closes → `OAuthInterstitialModal` opens immediately with 60s countdown.
4. On loopback callback → modal shows brief 200ms `Signed in. One moment…` then closes; user lands on the feature they originally clicked (D-10) or on MainApp if from AuthChoice.
5. On `Cancel sign-in` → modal closes, main aborts loopback server, user returns to SignInModal (NOT AuthChoice — preserves the email they may have typed for fallback).
6. On 60s timeout → modal swaps body to "That took a little too long" error variant with **Try again** ghost Button + `Cancel sign-in` quiet Button.

### local → signed_in upgrade (D-10)
- Any cloud-gated CTA in `local` mode opens SignInModal **with framing micro-copy** above the title (`Sign in to browse public characters`, etc.).
- Dismissal (`Back to Sei`) returns the user to the originating feature surface — not to AuthChoice.
- On success: modal closes → app state transitions `local → signed_in` → the originating feature unlocks and is shown immediately. No full-screen flash, no toast, no "Welcome" interstitial.

### Sign-out while bot running (D-09)
- Click `Sign out` in Account panel → if bot status is `running` → SignOutConfirmModal with bot-running copy variant → Confirm → main process calls bot-supervisor stop → wait for clean stop → clear session → state transitions `signed_in → local` **without screen change** (current screen stays mounted; only Account panel disappears and any cloud-only inline CTAs flip to ghost buttons).
- Dismissal (`Stay signed in`) closes the modal and preserves the session (and the running bot, if applicable). No state transition.
- Bot not running → same flow with the alternate title copy; no stop step.

### Email verification (D-04)
- Verification does NOT block sign-in. After signup → user proceeds straight into the app with `signed_in && !email_verified`.
- VerifyEmailBanner appears at the top of every screen.
- Settings → Account → Email row shows `Resend verification` quiet button (rate-limited to 1/minute client-side — surfaces "Hold on — wait a minute…" copy on rapid re-clicks).
- (Cloud-write block — "verify first" modal — is **Phase 11's concern**; Phase 10 only ships the Banner.)

## Accessibility

- Every modal: `role="dialog" aria-modal="true" aria-labelledby="{title-id}"` (matches `DeleteConfirmModal` pattern).
- Banner: `role="alert"` (matches existing Banner).
- VerifyEmailBanner: `role="status"` (less interruptive than alert since it persists).
- AuthChoice tiles: rendered as `<button>` elements (NOT divs) — keyboard tab-reachable. Tab order: Sign In tile → Continue Locally tile → footer text (not focusable).
- OAuth countdown: rendered in an `aria-live="polite"` region so screen readers announce minute boundaries but not every second.
- Type-to-confirm input: `aria-describedby` pointing to the email-bolded paragraph so SR users hear the required string.
- Focus rings: NEVER suppressed via `outline: none` without a `:focus-visible` replacement (already standard from `Button.module.css`).
- Color contrast: All text uses `--text` or `--text-2` against `--window` / `--surface` — these pass WCAG AA 4.5:1 in both themes (verified against tokens.css hex values — `#1A1D24` on `#FDFEFF` = 18.4:1; `#4A4D55` on `#FDFEFF` = 9.3:1; dark theme `#E8EBF0` on `#191C21` = 14.1:1).
- Destructive `--red` on `--window`: `#C4523A` on `#FDFEFF` = 4.6:1 (AA passes); `#E07259` on `#191C21` = 5.9:1 (AA passes). Safe for button background with `--window` text.
