<objective>
Wire the rendered frame back to the LLM (VIS-02 delivery half) — the second-riskiest seam of the phase — AND close the explicit-vision routing gap (VIS-07/D-09): the ONE LLM turn that follows an explicit `visualize` must hit the proxy's per-hour vision cap path (`/vision/v1/messages`, built in 15-02), while every other turn (and BYOK/local entirely) stays on the default path.

`orchestrator.js`/`messageMappers.js` have NO image-content path today. The cross-provider-safe shape (15-RESEARCH.md Pattern 2 / Pitfall 4): the `visualize` tool_result stays a short TEXT string, and the image rides a FRESH `role:'user'` turn carrying a provider-neutral `{ type:'image', source:{ media_type, data } }` block that each mapper translates to its native form. Only Anthropic accepts an image inside a tool_result; OpenAI/Gemini/Ollama require a user-turn image — so the fresh-user-turn shape is the lowest common denominator across all four Phase-14 providers.

Purpose: This completes the explicit-`visualize` loop from 15-04 — the action renders, this plan delivers the frame to the model AND meters the explicit render against the per-hour cap. It keeps the seam single-sourced in `messageMappers.js` so the orchestrator stays provider-agnostic, and keeps the per-call routing single-sourced in `anthropicClient.call` so idle renders (15-07) and normal turns are unaffected.

Output: image-block translation cases in `messageMappers.js` (+ tests); a per-call `path` override on `anthropicClient.call`; a fresh-image-user-turn append in `orchestrator.js` after `visualize` settles that ALSO routes that one post-visualize turn to `/vision/v1/messages` in cloud mode; a `loop.js` passthrough test confirming the image block survives `buildAnthropicPayload` (no loop.js edit expected).
</objective>

| bot -> proxy path selection | The bot chooses which proxy path (/v1/messages vs /vision/v1/messages) a cloud turn hits; the cap is enforced server-side regardless |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-06-01 | Information disclosure | image egress to third-party LLM leaks player/HUD/other-player content (Pitfall 9) | mitigate | Render is bot-POV ONLY (no OS capture, no HUD, no player-monitor); frame-fidelity (no HUD/hand) set in the render path (15-01). This plan transmits the bot-POV frame as-is; confirm no other-player/HUD overlay is composited here. Privacy is satisfied by the bot-POV-only decision (Roadmap Locked Decision 1). |
| T-15-06-02 | Tampering | image placed on a tool/function-result message -> 400 or silent drop on OpenAI/Gemini/Ollama (Pitfall 4) | mitigate | Image rides a FRESH user turn (Task 3); tool_result stays text; test asserts image never lands on a tool/function-result message. |
| T-15-06-03 | DoS / cost | oversized base64 inflates every outbound request | mitigate | Image already downscaled ~256px q0.4 + size-capped in 15-04; this plan only moves the small buffer onto the user turn. Per-hour cap (15-02) bounds frequency. |
| T-15-06-04 | Elevation / bypass | a bot forges the vision path to escape the cap, OR forges it onto every turn to grief the user's cap | mitigate | The vision cap is server-authoritative in the proxy (15-02 vision_hourly bucket): hitting /vision/v1/messages can only CONSUME the cap, never bypass it — a bot routing explicit turns to /v1/messages instead just pays normal credits + RPM/TPM/$ (rateLimitGate), and routing extra turns to /vision only burns the user's OWN cap faster (bounded, self-limiting). The client cannot forge a vision-BYPASS because the cap lives behind verifyJwt on the proxy, not in the bot.
