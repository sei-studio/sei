<objective>
Build the foundations that do NOT depend on the render path: the vision config block (D-04), the custom line-of-sight helper (VIS-05), and the provider-capability surface (D-10/VIS-03) the renderer toggle needs. All three are pure schema/geometry/bridge work consumable by Waves 2-4, so they run in Wave 1 alongside the spike.

Purpose: The config block is the source of truth for every vision knob (D-05 — the Settings toggle writes through to it) AND the single enforcement point of VIS-06's 512x512 ceiling: `resolution_px` is hard-bounded `<=512` here, so 15-01/15-04's downscale/encode can never exceed it. The LOS helper is mandatory because raw `bot.world.raycast` misses fluids and never tests entity bounding boxes (Pitfall 10 / VIS-05) — it gates the idle auto-render (VIS-04). The capability surface closes D-10's renderer half: `capabilities.vision` lives in the bot utilityProcess and is invisible to the renderer today, so the Settings auto-render toggle (15-05) cannot truthfully disable itself for a non-VLM provider without it. Surfacing it now (in the same plan that already bridges config and owns botSupervisor) lets 15-05 wire the toggle's `disabled` state to a real signal instead of inferring from `ai_backend_kind` or deferring.

Output: `config.vision` block in the bot ConfigSchema (with the `<=512` `resolution_px` cap) + a `vision_auto_render` field in UserConfig (runtime-parse-asserted) + the botSupervisor bridge; `lineOfSight.js` exporting `hasClearLineOfSight(bot, targetEntity)` with fluid + entity-AABB occlusion and a 16-block range gate, plus a co-located test; a `vision:capability` IPC channel that pushes the active provider's `capabilities.vision` boolean bot->main->renderer into `useUiStore.visionCapable`.
</objective>
