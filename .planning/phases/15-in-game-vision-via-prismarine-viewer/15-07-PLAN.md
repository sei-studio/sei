Hook idle auto-render into the existing P3 idle tick (VIS-04). This is the last plan — it depends on the LOS helper (15-03), the visualize action (15-04), and the image-attach wiring (15-06) all landing first. On each `sei:idle` tick, gate on: `config.vision.auto_render` ON (default OFF), provider `capabilities.vision` true (D-10), owner within 16 blocks (VIS-04), and custom LOS clear (VIS-05). When all pass, render via `visualizeAction` with the idle flag (so D-02 dedupe applies) and attach the frame as a fresh user turn (reusing the 15-06 mechanism). Idle renders do NOT hit the proxy per-hour cap (D-09 — only explicit renders do).

It fails CLOSED at every gate, reuses the existing idle tick (no new timer — 15-PATTERNS.md "Idle (P3) tick reuse"), and the dedupe + default-OFF + cadence bounds keep cost in check (Pitfall 8). Explicit visualize (15-04) is ungated by proximity; the 16-block + LOS gate governs ONLY this idle path (D-08).

Output: `idleVisionGate.js` (`shouldAutoRender` — the composite gate) + test; an idle-tick hook in `orchestrator.js` that runs the gate and, when clear, renders + attaches the frame on the idle turn.
