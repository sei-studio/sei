```js
async function processNext() {
  const item = queue.shift()
  if (!item) {
    processing = false
    return
  }

  // Cancel current action if incoming has higher priority (lower number)
  if (currentAction && item.priority < currentAction.priority) {
    currentAction.controller.abort()
    currentAction = null
  }

  // Skip lower-priority events while a higher-priority action runs
  if (currentAction && item.priority > currentAction.priority) {

    queue.unshift(item)
    processing = false   // ← BUG: nothing schedules the drain after currentAction completes
    return

  currentAction = { controller, priority: item.priority }

  try {
    await onDispatch(item.event, item.data, controller.signal)
  } catch (err) { ... }
  finally {
    if (currentAction?.controller === controller) {
      currentAction = null
    }
  }

  if (queue.length > 0) {
    setImmediate(processNext)
  } else {
    processing = false
  }

```
WR-01 root: when the re-queue branch fires, `processing = false` and no further `enqueue()` schedules a drain — the re-queued item sits forever until the next event arrives. The fix is to leave `processing` TRUE on the re-queue branch (the in-flight action's finally will fall through to the trailing drain check), which means the early-return is the bug, not the unshift itself.

```js
    configurable: true,

```
WR-02 root: each call creates a fresh controller and overrides the loop's getter, but the EXTERNAL signal (FSM-supplied) bridge at line 663-665 is set ONLY for the original controller:
```js
const onExternalAbort = () => { try { loop.abortController.abort() } catch {} }
if (signal) {
  if (signal.aborted) onExternalAbort()
  else signal.addEventListener('abort', onExternalAbort, { once: true })

```
Since `onExternalAbort` references `loop.abortController` lazily (it dereferences the GETTER at call time), the bridge actually DOES route to whichever controller is current — IF the listener fires. But it's a `{ once: true }` listener; once consumed, subsequent external aborts won't re-trigger. So the bug is: after the FIRST external abort fires (consuming the listener), a SECOND external abort that arrives later in the loop's lifetime is not delivered to the new controller.

Fix: capture the external signal on the loop, and on every `replaceAbortController(loop)` call, re-bridge the external signal to the NEW controller (only if the external signal hasn't already aborted).
