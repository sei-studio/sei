## Parallel-Dig Cap Decision

**Cap = 1** (per plan quality gate "Plan 5 specifies exact parallel-dig cap value").

**Justification:**
- D-W-2: five identical dig calls on the same `{target:'#3'}` after the snapshot rotated (wood-findings line 244)
- D-H-5: seven-way dig storm during sand collection (hunt-sand-findings)
- RESEARCH A2 noted the regression on chopping-2-block-tree cases. **Accepted regression** — the LLM can re-issue dig next turn (≤2s round-trip vs. the visible disruption of 5–7 simultaneous identical digs).
- Abort message guides re-issue: `"aborted: only one dig per turn allowed; re-issue next turn or use {block:\"<name>\"} for repeat digs"`. The `{block:"<name>"}` hint is the canonical pattern from D-W-3 — repeated digs of the same block type auto-find the next nearest each call.

**3. [Rule 1 - Predicate input divergence] First-turn-say predicate uses loop._responsesReceived not loop.iterationCount**
- **Found during:** Task 1 implementation
- **Issue:** Plan's pseudocode reads `loop.iterationCount === 1` to identify the first iteration. Loop's `appendUserTurn` increments iterationCount per non-seed user turn. In the seed-loaded path (`sessionState && ownerStore && diary` present), the seed turn has `seed: true` and does NOT increment, so when the first response arrives `loop.iterationCount === 0`. In the fallback path (no memory layer wired), the first user turn IS counted, so when the first response arrives `loop.iterationCount === 1`. The plan's pseudocode would fire correctly only in the fallback path.
- **Fix:** Added `loop._responsesReceived` counter incremented immediately after `appendAssistant` (before tool dispatch). The pure predicate accepts an `iterationCount` parameter where `1` means "first response received from the LLM in this loop". Both paths converge on the same semantic.
- **Files affected:** `src/brain/orchestrator.js`
- **Verification:** Test case 4 ("Second iteration: NEVER reprompt") explicitly asserts the predicate returns false on iterationCount=2; runtime path passes `loop._responsesReceived` so semantics match.
- **Committed in:** `61ae82c`

**4. [Rule 1 - goTo range default condition broadened from `!args.range` to `range <= 1`]**
- **Found during:** Task 3 implementation
- **Issue:** Plan's pseudo: `if (!args.range && args.x !== undefined && isCoordsAtKnownPlayer(...))`. Zod's `.default(1)` fires when the LLM omits the field, making `!args.range` evaluate as `!1 === false` (the default value is 1, not 0). So the literal pseudo would never fire.
- **Fix:** Changed condition to `range <= 1 && isCoordsAtKnownPlayer(...)`. Captures both "LLM omitted range (Zod default 1 fired)" and "LLM explicitly passed range:0/1". The semantic intent — "when the LLM didn't deliberately request a close approach" — is preserved.
- **Files affected:** `src/adapter/minecraft/registry.js`
- **Verification:** Tested by code reading; D-H-9 case (LLM passes goTo to player coords without range) hits the bumped default.
- **Committed in:** `4f885db`

**5. [Rule 3 - Helper pre-stubbed in Task 1 commit, test added in Task 4 commit]**
- **Found during:** Task 4 setup
- **Issue:** Task 4 is `tdd="true"`, but `_advanceIterationCadence` and `SILENT_ITERATIONS_BEFORE_NUDGE` were already exported in Task 1's GREEN commit (`61ae82c`) because the `loop.iterationsSinceLastSay` and `loop._progressNudgeFired` initializations were placed in the same `handleDispatch` block as Task 1's `_firstTurnReprompted` initialization (single coherent loop-state block). Without that runtime wiring, the helper is a no-op; with it, the nudge fires per the unit-tested cadence.
- **Files affected:** Same as Task 4 — no extra changes.
- **Verification:** All 5 test cases pass; runtime nudge text appears in the next user turn's eventText after 4 silent iterations.
- **Committed in:** `61ae82c` (helper stub) + `63f071c` (runtime wiring + test)
