## Decisions Made

- **`reenqueue` callback for brain re-dispatch instead of bot.emit:** The orchestrator previously re-fired `sei:loop_terminal` and `sei:attacked` via `bot.emit(...)`, which leaked the bot reference into orchestrator.js and tied re-dispatch to mineflayer's EventEmitter. The new `reenqueue(event, data, priority?)` callback (injected by brain.start) routes the same events through the brain's priority queue with explicit priority assignment. Removes the last bot.* property access from the orchestrator and decouples re-dispatch from any specific event bus — works equally well with mineflayer today and Electron IPC in Phase 4.
- **Stop-verb fast-path deferred to Plan 03.1-03:** chat.js's pre-refactor fast path needed an orchestrator handle to (a) abort the active loop and (b) clear owner_goals without a Haiku round-trip. Exposing the orchestrator across the brain↔adapter seam would break the contract. Owner-chat preemption now runs through the brain priority queue (P1→P0 escalation when a non-P0 action is in flight); the visible regression is one extra Haiku round-trip on the literal "stop"/"halt"/etc. words. Documented as a deviation; restoring full fidelity (e.g. an adapter.cancelInflight() method or a brain-side stop-verb predicate) is in Plan 03.1-03's scope.
- **executeAction config shape:** Adapter's `executeAction(name, args, ctx)` receives a context object containing `signal` and `_goalStore`. The minecraft adapter folds these into the config-shaped 4th argument expected by existing behavior handlers (which read `config.pathfinder_timeout_ms`, `config._goalStore`, `config.signal`). It also unifies `adapter.minecraft.pathfinder_timeout_ms` and `adapter.minecraft.follow_range` back to the top-level keys behaviors expect, so the 12 minecraft-side action handlers don't need any changes — they continue to read flat config keys and the adapter does the lift.
- **sessionBotShim in brain/index.js:** sessionState.js still expects a `bot`-shaped argument with `.players`, `.username`, and `.once('playerJoined', ...)`. brain/index.js constructs a thin shim that proxies into `adapter.getKnownPlayers()` and `adapter.botUsername`, plus an in-memory `_onceJoinedListeners` queue drained when the adapter delivers a playerJoined event. Avoids a sessionState refactor in this plan; that refactor is in Plan 03.1-04 (memory).

**2. [Rule 3 - Blocking] Stop-verb fast path could not be preserved across the seam**
- **Found during:** Task 3 wiring
- **Issue:** chat.js (adapter behavior) had a stop-verb fast path that called `orchestrator.currentLoop?.abortController?.abort()` and `orchestrator.goals.remove(...)` synchronously. With the brain↔adapter seam in place, the orchestrator handle is no longer reachable from adapter behaviors. Two options were considered: (a) build a whole-cloth shim object exposing `currentLoop`/`goals`/`recordIncomingChat` (leaks brain internals across the seam, contradicting the plan's must-have truth #2), or (b) drop the fast path and rely on the brain priority queue's P1→P0 escalation for owner-chat preemption (one extra Haiku round-trip on `stop`).
- **Fix:** Took option (b). Pass `null` as the orchestrator argument to `startChat(bot, config, null)` — the chat behavior still does owner-detection / addressed / nearby filtering and emits `sei:chat_received`, which adapter/fsmWires translates to brain.onChat. Owner-chat is still preempted via the priority queue's promotion of `ownerSpoke=true` to P0 when a non-P0 action is in flight (see brain/fsm.js enqueue()).
- **Behavior change:** Pre-refactor: `stop` triggered an immediate body-cancel + chat ack + goal-clear with zero LLM cost. Post-refactor: `stop` runs through the queue → orchestrator → Haiku round-trip → `say("stopping.")` (with the abort still firing via the queue). Cost: one Haiku call per "stop" (~150ms + a few cents per session). Documented in Plan 03.1-03 scope; not a Plan 02 regression to fix.
- **Files modified:** src/index.js (passes null to startChat)
- **Verification:** Module graph still imports cleanly; brain priority queue tests P1→P0 escalation in fsm.js's enqueue() function.
- **Committed in:** `3bafb1c` (Task 3)
