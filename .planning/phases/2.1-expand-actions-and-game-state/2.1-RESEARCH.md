# Phase 2.1: Expand Actions & Game State - Research

**Researched:** 2026-04-25
**Domain:** mineflayer action surface expansion + world-state observation rendering for a two-layer LLM (Anthropic Haiku 4.5 personality + Ollama Qwen 2.5 movement)
**Confidence:** HIGH on mineflayer APIs (verified against installed `node_modules/mineflayer/docs/api.md`), HIGH on prior-art chest/dig/place patterns (verified against `../sui/mindcraft-0.1.4/src/agent/library/skills.js`), MEDIUM on token-cost estimates for the snapshot, MEDIUM on optimal handle TTL (low risk; easy to tune).

## Summary

This phase is mostly **glue work** over a known mineflayer surface. Every tier-1 action in D-22 maps to a single `bot.*` Promise call; the riskwork is correct timeout-wrapping (CLAUDE.md ADR #5), AbortSignal propagation through the FSM controller, and the targeting-resolver layer that turns `{block:"oak_log",maxDistance:32}` into a `Block` instance for `bot.dig` / `bot.placeBlock`. The snapshot composer is new but small: ~5 pure-function observers (vitals, inventory, blocks, entities, world) feeding a top-level renderer that produces a line-oriented text block with `#N` handles. Snapshot is injected **after** the cache breakpoint per D-27 so the cached prefix stays warm, and the capability paragraph + Minecraft primer + "still learning" trait extend the cached prefix per D-30/D-31/D-32.

**Primary recommendation:** Build `src/observers/` as five pure functions returning plain objects, plus one `composeSnapshot(bot, opts)` that owns the in-memory handle table. Add new actions to `createDefaultRegistry()` in `src/registry.js` following the `goTo` template (Zod schema → handler that resolves target → wraps mineflayer Promise in `Promise.race` against a wall-clock timeout → returns a deterministic *what*-only result string per D-35). Inject the snapshot text into `renderUserContext()` in `src/llm/orchestrator.js` (just before `Event:` lines), and extend `src/llm/anthropicClient.js#buildCachedSystem` to accept the new capability/primer/trait blocks as additional cached system text.

### Pattern 1: Tier-1 action handler skeleton (the load-bearing pattern)

Every new action follows this template — modeled after `src/behaviors/pathfind.js`:

```javascript
// src/behaviors/dig.js
// Source: synthesis of pathfind.js timeout pattern + mineflayer api.md L1906-1922
import { resolveBlock } from '../observers/targeting.js'

const DEFAULT_TIMEOUT_MS = 8000

export async function digAction(args, bot, config) {
  const signal = config?.signal
  if (signal?.aborted) return 'aborted'

  const block = await resolveBlock(args, bot)            // name | #N | coord → Block | null
  if (!block) return 'no target'                          // D-35: what-only

  if (!bot.canDigBlock(block)) return `cannot dig ${block.name}`  // [VERIFIED: api.md L1633]

  const timeoutMs = args.timeout_ms ?? config?.dig_timeout_ms ?? DEFAULT_TIMEOUT_MS

  const dig = bot.dig(block).then(() => `dug ${block.name}`)
                            .catch(() => `cannot dig ${block.name}`)
  const tmo = new Promise((r) => setTimeout(() => {
    try { bot.stopDigging() } catch {}                    // [VERIFIED: api.md L1924]
    r('timeout')
  }, timeoutMs))
  const onAbort = new Promise((r) => {
    if (!signal) return
    signal.addEventListener('abort', () => {
      try { bot.stopDigging() } catch {}
      r('aborted')
    }, { once: true })
  })

  return Promise.race([dig, tmo, onAbort])
}
```

### Pattern 2: Targeting resolver

```javascript
// src/observers/targeting.js
// Source: synthesis of mineflayer api.md (findBlock / findBlocks / nearestEntity) +
// mindcraft prior art (skills.js:444 getNearestBlock pattern) + CONTEXT D-25
import mcDataLib from 'minecraft-data'

let _handles = new Map()  // #N → { kind:'block'|'entity', pos|entityId, expiresAt }
const HANDLE_TTL_MS = 10_000

export function setHandles(entries) {
  _handles = new Map(entries)
}

export async function resolveBlock({ block, target, x, y, z, maxDistance = 32 }, bot) {
  // (a) explicit coord
  if (typeof x === 'number' && typeof y === 'number' && typeof z === 'number') {
    return bot.blockAt(new (await import('vec3')).default(x, y, z))
  }
  // (b) #N handle
  if (typeof target === 'string' && target.startsWith('#')) {
    const h = _handles.get(target)
    if (!h || h.expiresAt < Date.now()) return null  // stale → caller returns "stale target"
    if (h.kind !== 'block') return null
    return bot.blockAt(h.pos)
  }
  // (c) by name
  if (typeof block === 'string') {
    const mcData = mcDataLib(bot.version)
    const ids = mcData.blocksByName[block]
      ? [mcData.blocksByName[block].id]
      : []
    if (!ids.length) return null
    return bot.findBlock({ matching: ids, maxDistance })   // [VERIFIED: api.md L1629-1631]
  }
  return null
}

export function resolveEntity({ entity, target, entity_id }, bot) {
  if (entity_id != null) return bot.entities[entity_id] ?? null     // [VERIFIED: api.md L858-861]
  if (typeof target === 'string' && target.startsWith('#')) {
    const h = _handles.get(target)
    if (!h || h.expiresAt < Date.now() || h.kind !== 'entity') return null
    return bot.entities[h.entityId] ?? null
  }
  if (typeof entity === 'string') {
    return bot.nearestEntity(e => (e.name ?? '').toLowerCase() === entity.toLowerCase())
                                                                     // [VERIFIED: api.md L1655-1662]
  }
  return null
}
```

**Note on `minecraft-data`:** mineflayer already pulls this in transitively. If a direct import is undesirable, an alternative is to call `bot.findBlock({ matching: (b) => b.name === blockName, maxDistance })` — function-form `matching` is supported per api.md L1620.

### Pattern 3: Snapshot composer (D-26 / D-27)

```javascript
// src/observers/snapshot.js
// Source: synthesis of CONTEXT D-26 + mineflayer api.md state APIs +
// mindcraft world.js (../sui/mindcraft-0.1.4/src/agent/library/world.js) for the inventory-summary pattern
import { setHandles } from './targeting.js'

const HANDLE_TTL_MS = 10_000

export function composeSnapshot(bot, { goals, lastActionResult } = {}) {
  const lines = []
  const handles = []
  let n = 0

  // --- world ---
  const p = bot.entity.position                    // [VERIFIED: api.md L854]
  const biome = bot.world.getBiome(p)?.name ?? 'unknown'  // prismarine-biome
  const t = bot.time
  lines.push(`pos: ${p.x.toFixed(1)} ${p.y.toFixed(1)} ${p.z.toFixed(1)}`)
  lines.push(`biome: ${biome}  time: ${t.isDay ? 'day' : 'night'} (${t.timeOfDay} ticks)`)
                                                    // [VERIFIED: api.md L1076, L1090]

  // --- vitals ---
  lines.push(`hp: ${Math.round(bot.health)}/20  food: ${Math.round(bot.food)}/20  xp: lvl ${bot.experience.level}`)
                                                    // [VERIFIED: api.md L1014, L1024, L1028]
  if (bot.isSleeping) lines.push(`status: sleeping`)

  // --- holding ---
  const h = bot.heldItem                            // [VERIFIED: api.md L870-872]
  lines.push(`holding: ${h ? `${h.name}${h.maxDurability ? ` (durability ${h.maxDurability - (h.durabilityUsed ?? 0)}/${h.maxDurability})` : ''}` : 'empty'}`)

  // --- inventory (stack-summarized) ---
  const items = bot.inventory.items()               // [VERIFIED: api.md L1119-1121]
  const counts = {}
  for (const it of items) counts[it.name] = (counts[it.name] ?? 0) + it.count
  const invStr = Object.entries(counts).map(([k, v]) => `${k}×${v}`).join(' ') || 'empty'
  lines.push(`inventory: ${invStr}`)

  // --- nearby blocks (radius 16, top N closest) ---
  // findBlocks returns positions sorted closest-first; we re-render with #N handles.
  // mcdata mapping kept implicit by using function-form `matching`.
  // Targeting is "interesting blocks": logs, ores, beds, chests, crafting tables, furnaces, etc.
  const interestingBlock = (b) => INTERESTING_BLOCK_NAMES.has(b.name)
  const blockPositions = bot.findBlocks({ matching: interestingBlock, maxDistance: 16, count: 12 })
                                                    // [VERIFIED: api.md L1615-1627]
  let blockMore = 0
  if (blockPositions.length === 12) {
    // crude "+K more" hint — count once with a higher cap for the message only
    const all = bot.findBlocks({ matching: interestingBlock, maxDistance: 16, count: 64 })
    blockMore = Math.max(0, all.length - 12)
  }
  lines.push('nearby blocks:')
  for (const pos of blockPositions.slice(0, 8)) {            // 8 lines max for token budget
    n++
    const b = bot.blockAt(pos)
    if (!b) continue
    handles.push([`#${n}`, { kind: 'block', pos, expiresAt: Date.now() + HANDLE_TTL_MS }])
    lines.push(`  #${n} ${b.name} @${pos.x},${pos.y},${pos.z}`)
  }
  if (blockMore > 0) lines.push(`  +${blockMore} more`)        // D-29

  // --- nearby entities (radius 24, top N closest) ---
  const myPos = bot.entity.position
  const ents = Object.values(bot.entities)
    .filter(e => e !== bot.entity)
    .map(e => ({ e, d: e.position.distanceTo(myPos) }))
    .filter(({ d }) => d <= 24)
    .sort((a, b) => a.d - b.d)
  lines.push('nearby entities:')
  for (const { e } of ents.slice(0, 6)) {                       // 6 lines max
    n++
    handles.push([`#${n}`, { kind: 'entity', entityId: e.id, expiresAt: Date.now() + HANDLE_TTL_MS }])
    const label = e.username ?? e.name ?? `entity-${e.id}`
    lines.push(`  #${n} ${label} @${e.position.x.toFixed(1)},${e.position.y.toFixed(1)},${e.position.z.toFixed(1)}`)
  }
  if (ents.length > 6) lines.push(`  +${ents.length - 6} more`)

  // --- goals ---
  if (goals) {
    lines.push(`owner_goals: ${JSON.stringify(goals.owner_goals ?? [])}`)
    lines.push(`self_goals:  ${JSON.stringify(goals.self_goals ?? [])}`)
  }
  if (lastActionResult) lines.push(`last_action_result: ${lastActionResult}`)

  setHandles(handles)
  return lines.join('\n')
}
```

**Token-cost estimate (rough, [ASSUMED]):**
- vitals + world + holding: ~4 lines × ~12 tokens = ~50
- inventory: 1 line, depends on diversity; 30 distinct stacks ≈ ~80 tokens
- 8 block lines + 6 entity lines + 2 truncation hints + 2 headers ≈ ~120 tokens
- goals (2 lines) + last_action_result ≈ ~30 tokens
- **Total: ~280 tokens nominal, ~450 worst-case** — well under D-26's 500-soft-cap. If truncation cutoffs need tightening, drop to 6 blocks / 4 entities.

### Pattern 4: Snapshot handle TTL — recommendation

D-26's per-handle TTL is the cleaner option, but **per-snapshot replace is simpler and equivalent in practice** because the snapshot is recomputed every personality turn (D-27) and personality turns are < 10s apart in active play. Recommendation: **per-snapshot replace**, with each handle stamped with `expiresAt = now + 10s` only for the rare case where the LLM emits a tool_call with a stale `#N` from a previous snapshot (e.g., during a long movement chain). On lookup miss or expiry, the resolver returns `null` and the handler returns `"stale target"` per D-35; the personality LLM re-snapshots on the next turn naturally.

### Pattern 6: Container session lifecycle

`bot.openContainer(block)` returns a Promise that resolves to a `Container` instance (subclass of `windows.Window`). The container exposes `.deposit(itemType, metadata, count, nbt)`, `.withdraw(itemType, metadata, count, nbt)`, `.containerItems()`, and `.close()`. [VERIFIED: api.md L461-491, L2069-2077]

```javascript
// src/behaviors/container.js
import mcDataLib from 'minecraft-data'

const SESSION = { container: null, blockPos: null }

export async function openContainerAction(args, bot, config) {
  const signal = config?.signal
  if (signal?.aborted) return 'aborted'
  // close any prior session first
  if (SESSION.container) { try { await SESSION.container.close() } catch {} }

  const target = await resolveBlock(args, bot)
  if (!target) return 'no target'
  // pathfind close enough first — pattern from mindcraft skills.js:890 (goToPosition before openContainer)
  // (orchestrator may also have routed a goTo before this; we just verify proximity)
  if (bot.entity.position.distanceTo(target.position) > 4) return 'target out of reach'

  const timeoutMs = config?.openContainer_timeout_ms ?? 6000
  const open = bot.openContainer(target).then(c => { SESSION.container = c; SESSION.blockPos = target.position; return `opened ${target.name}` })
                                        .catch(() => `cannot open ${target.name}`)
  const tmo = new Promise(r => setTimeout(() => r('timeout'), timeoutMs))
  return Promise.race([open, tmo])
}

export async function depositItemAction({ item, count = 1 }, bot, config) {
  if (!SESSION.container) return 'no container open'
  const mcData = mcDataLib(bot.version)
  const id = mcData.itemsByName[item]?.id
  if (id == null) return `unknown item ${item}`
  const inv = bot.inventory.findInventoryItem(id, null, false)
  if (!inv) return `no ${item} to deposit`
  const tmo = new Promise(r => setTimeout(() => r('timeout'), config?.transfer_timeout_ms ?? 4000))
  const op  = SESSION.container.deposit(id, null, Math.min(count, inv.count)).then(() => `deposited ${count} ${item}`).catch(() => `deposit failed`)
  return Promise.race([op, tmo])
}

// withdrawItemAction is symmetric using SESSION.container.withdraw(id, null, count) and
// returns `"chest empty"` when containerItems().filter(name===item) is empty.
```

**Key:** the personality LLM is expected to chain `openContainer` → `depositItem`/`withdrawItem` → (implicit close on next non-container action or chain end). On chain end or new `openContainer`, close the prior session. **Edge case:** if FSM aborts mid-chain, ensure `SESSION.container.close()` runs in a `finally` to avoid leaving a phantom open window. The mineflayer `windowOpen`/`windowClose` events can verify this in tests. [VERIFIED: api.md L1484-1488]

### Anti-Patterns to Avoid

- **Letting the personality LLM see the action registry:** D-28 / D-33 / D-34. Personality tools remain `say`, `handOffToMovement`, `setGoals` (and now `look()`). Tier-1 actions are exposed ONLY to the movement LLM. The capability paragraph (D-30) is NL prose, not a tool list.
- **Causal reasoning in handlers:** D-35. Handlers MUST NOT return "you need a pickaxe" or "the chest is too far away" — return `"cannot dig stone"`, `"target out of reach"`. The personality LLM asks the owner per D-32.
- **Auto-equipping the right tool inside `dig`:** mindcraft does this via `bot.tool.equipForBlock(block)` (skills.js:480). **Don't.** Phase 2.1 explicitly defers crafting and tool-management heuristics. Failing with `"cannot dig stone"` is the desired loop closure.
- **Rendering the entire `bot.entities` map:** that's hundreds of entries on a populated server. Filter by distance ≤24 and cap at the truncation N.
- **Showing snapshot indices to the movement LLM:** `#N` lives entirely in the personality layer. The personality LLM resolves `#3` to a concrete name before handing off ("dig the closest birch_log within 16 blocks"), per D-34.
- **Mutating `bot.entity.{position,velocity}` to "fix" NaN values in any new behavior:** that's the substrate bug fixed in Phase 2 close-out (`02-03-SUMMARY.md` Substrate Fix B). New combat-adjacent code (e.g., `attackEntity`) must NOT touch `bot.entity` directly.
- **Calling `bot.dig` while a previous dig is unfinished:** `bot.dig` throws a fatal `diggingAborted` error. [VERIFIED: api.md L1922]. Either await the previous dig or call `bot.stopDigging()` first. The action handler's timeout race already covers this via `bot.stopDigging()` in the timeout branch.
- **Leaving a Container open after the chain ends:** see Pattern 6 — chest sessions need explicit `.close()` on abort/error.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Block name → block id resolution | Custom name table | `minecraft-data` (transitive via mineflayer) — `mcData.blocksByName[name].id`. Or pass function-form `matching` to `bot.findBlock`. | mcdata is already pulled in, version-correct (uses `bot.version`), maintained. |
| Distance-sorted nearby blocks | Manual scan + sort | `bot.findBlock`/`bot.findBlocks` — already returns closest-first | [VERIFIED: api.md L1627] |
| Distance-sorted nearby entities | Manual scan from a 3D index | `bot.entities` map + filter+sort by `pos.distanceTo` (~20 LOC) | Trivial; no need for a library. |
| Inventory stack summaries | Iterate raw slots | `bot.inventory.items()` + group by `.name` (~10 LOC) | Already a clean prismarine-windows abstraction. |
| Container deposit/withdraw | Manual slot drag | `container.deposit(id,null,count)` / `container.withdraw(id,null,count)` | [VERIFIED: api.md L465-481]. mindcraft uses the same API. |
| Biome lookup | Manual chunk parse | `bot.world.getBiome(pos).name` (prismarine-biome) | Standard mineflayer API. |
| AbortController/timeout pattern | Bespoke per-action plumbing | Reuse `pathfind.js` Promise.race template | Battle-tested in Phase 1 production code. |

**Key insight:** Every tier-1 action is one mineflayer Promise call; the engineering value-add is **uniform timeout+abort wrapping**, **deterministic result strings (D-35)**, and the **targeting resolver that turns LLM-friendly args into `Block`/`Entity` instances**. Resist the urge to add intelligence (auto-equip, auto-pathfind-before-action) inside handlers — D-35 says handlers stay dumb; the personality LLM orchestrates.

## Common Pitfalls

### Pitfall 1: Pathfinder-style silent hang on `bot.openContainer` / `bot.dig`
**What goes wrong:** Server packet loss → mineflayer waits forever for the windowOpen / blockBreak confirmation.
**Why it happens:** Same class of bug as the Phase 1 pathfinder hang — every server-confirmation operation is at risk.
**How to avoid:** ADR #5: every action handler wraps its mineflayer call in `Promise.race(call, timeout, onAbort)`. Suggested timeouts (Claude's discretion per CONTEXT): dig 8s, place 4s, equip 2s, attack 6s (per swing), consume 4s, lookAt 2s, drop 2s, activateItem 2s, sleep 12s, openContainer 6s, withdraw/deposit 4s.
**Warning signs:** Action handler never resolves; FSM `currentAction` stays held → next P0/P1 cancels via existing AbortController, but logs show actions completing only on cancel.

### Pitfall 2: `bot.dig` re-entry → fatal `diggingAborted`
**What goes wrong:** Two overlapping dig calls; mineflayer raises a fatal error and disconnects the bot.
**Why it happens:** [VERIFIED: api.md L1922] *"If you call bot.dig twice before the first dig is finished, you will get a fatal 'diggingAborted' error."*
**How to avoid:** In the `digAction` handler, check `bot.targetDigBlock != null` and either await its completion or call `bot.stopDigging()` first. The timeout race already calls `bot.stopDigging()` on the timeout branch.
**Warning signs:** Bot disconnects with `diggingAborted` in server log.

### Pitfall 3: Eat-while-moving silently fails
**What goes wrong:** `bot.consume()` fails or yields no food when the bot is moving (sprint/jumping interrupts the use animation).
**Why it happens:** Vanilla MC behavior — eating requires holding right-click without movement interruption.
**How to avoid:** Before calling `bot.consume()`, call `bot.clearControlStates()` (api.md L1824) — and document that the personality LLM should not chain "go to X and eat" in one hand-off; eat is its own intent. Result string on failure: `"could not eat"`.

### Pitfall 4: `bot.sleep(bedBlock)` rejects during the day
**What goes wrong:** Vanilla server rejects sleep before night; promise rejects.
**How to avoid:** Pre-check `bot.time.isDay === false` (api.md L1090). If day, return `"can't sleep during day"` without calling `bot.sleep`. The personality LLM may then ask the owner to wait or pivot to another goal per D-32.
**Warning signs:** Sleep failures clustered around `time < 12541` ticks.

### Pitfall 5: `bot.attack` distance and cooldown
**What goes wrong:** Bot swings at empty air; entity is out of reach.
**Why it happens:** Vanilla reach is ~3.0 blocks (survival). Also 1.9+ has an attack cooldown (~600ms for swords, longer for axes) where damage is reduced.
**How to avoid:** Pre-check distance (`entity.position.distanceTo(bot.entity.position) <= 3.5`). For multi-hit "kill this mob" intents, the personality LLM should issue `attackEntity({entity:"zombie"})` repeatedly (one attack per call) rather than a "fight this thing to death" intent. **Recommendation:** `attackEntity` action does ONE swing and returns. If user wants kill-loop semantics, that's a future phase (`mineflayer-pvp` plugin already loaded — but we're keeping it scripted P0-only per Phase 1).
**Warning signs:** Many `attackEntity` calls per chain → bot is in a dumb attack-loop; revisit handler design.

### Pitfall 6: Chest stays open after chain abort
**What goes wrong:** A `sei:attacked` P0 cancels mid-`depositItem` chain; container session is never `.close()`d; subsequent `openContainer` may fail or re-open the same window.
**How to avoid:** Wrap container session in a `try/finally` at the orchestrator OR maintain a top-level `currentContainer` that is force-closed on every fresh `sei:dispatch` chain start. mineflayer also auto-closes when the bot moves far away, but don't rely on it.
**Warning signs:** `windowOpen` / `windowClose` event imbalance.

### Pitfall 7: Large `bot.entities` map → snapshot blowup
**What goes wrong:** Server with 100+ mobs in render distance; snapshot inflates past 500-token soft cap.
**How to avoid:** Hard truncate to N=6 entities and N=8 blocks; emit `+K more` per D-29. Token-cost estimate above shows nominal ~280 tokens.

### Pitfall 8: Snapshot reflects PRE-action state in the same chain
**What goes wrong:** Personality LLM is shown a snapshot taken at chain start; midway, the bot has dug 3 logs; the LLM still thinks inventory is empty.
**Why it happens:** Snapshot is composed once per personality call (D-27). A multi-hop chain reuses the snapshot for that chain.
**How to avoid:** This is **by design**. D-36 says the next chain's snapshot reflects the new state. For mid-chain refresh, the personality LLM calls the `look()` tool, which is a registry action that does nothing but returns "ok refreshed" — the orchestrator detects the call and re-composes the snapshot for the next bump. Concretely: `look()` is on the **personality** registry (like `say` / `setGoals`), and its handler (in `orchestrator.js`) sets a flag `_snapshotDirty = true`; the next personality LLM call re-renders before sending.
**Warning signs:** LLM re-issues an already-completed action.

### Pitfall 9: Capability paragraph drift from actual registry
**What goes wrong:** D-30 paragraph says "Sei can mine and place blocks" but `dig` is missing from the registry, or vice versa. The "can't" list goes stale.
**How to avoid:** Lint test in `verify-phase2_1.js` structural mode: assert every capability mentioned in the paragraph maps to a registered action, and every "can't" item is NOT in `registry.list()`.
**Warning signs:** Adversarial prompt set fails because LLM tries to call a non-existent action.

### Pitfall 10: `cache_creation_input_tokens` stays 0 after Phase 2.1 additions
**What goes wrong:** Capability + primer + learning-line still don't push the prefix over Haiku 4.5's 4,096-token minimum (Phase 2 closed at 0). PERS-05 telemetry expectation is a real cache hit eventually.
**How to avoid:** Document in SUMMARY.md that this remains "optional future work" (Phase 2 already accepted this). Padding to 4k requires either expanding the system instructions block or expanding the primer. **Don't** pad with filler — pad with real content if/when desirable. Verifiable via `node scripts/verify-phase2_1.js --live` reading `usage.cache_creation_input_tokens` on the first call after restart.

## Open Questions

1. **Should `equip` accept item-name OR just inventory-slot index?**
   - What we know: `bot.equip(item, destination)` accepts `Item` instance OR numeric item id. [VERIFIED: api.md L1864-1877]
   - What's unclear: Movement LLM has only the personality's NL hand-off. If hand-off says "equip the diamond pickaxe", LLM passes `{item: "diamond_pickaxe", destination: "hand"}`. Resolver looks up `bot.inventory.findInventoryItem(name)`.
   - Recommendation: `equip({item: string, destination: 'hand'|'head'|'torso'|'legs'|'feet'|'off-hand'})`. Resolver uses `bot.inventory.findInventoryItem(itemId)` after mcdata name→id. Return `"no <item> in inventory"` if absent.

2. **Should `dropItem` map to `bot.tossStack(item)` or `bot.toss(itemType, metadata, count)`?**
   - What we know: `tossStack(item)` drops a whole stack; `toss(itemType, metadata, count)` drops a count. [VERIFIED: api.md L1885-1899]
   - Recommendation: One action `dropItem({item: string, count?: number})`. If `count` omitted → use `tossStack` (whole stack). If `count` given → use `toss(id, null, count)`.

3. **What should `consumeItem({item: "bread"})` do if the bot is already holding bread?**
   - Recommendation: Skip the equip step if `bot.heldItem?.name === item`. Saves a tick.

4. **How is "still learning" enforced beyond the prompt line?**
   - The trait IS the prompt line. No code enforces "ask the owner." The capability paragraph and primer set context; the line tilts behavior. Adversarial harness measures whether it works. If pass-rate is low after live test, options are: (a) strengthen the line, (b) move it earlier in the persona block (priority order), (c) repeat similar guidance in the system instructions.

5. **Should `look()` cost a hop?**
   - Recommendation: YES — it's a personality LLM call, and the hop counter increments per LLM call (D-11). This naturally discourages spam. If the LLM exhausts hops by snapshotting too much, the cap-hit handler fires and the chain ends with a persona-toned message.

6. **Snapshot in `--live` verification: how to assert correctness without a live MC server?**
   - Structural verification: stub `bot` with predictable state and assert `composeSnapshot()` text format (lines, handles, truncation).
   - Live verification: same as `verify-phase2.js --live` — use real Anthropic + Ollama, but mock the bot or run against the user's local MC server for full smoke.
