# Quick Task 260525-s09: IPC + Bot Safety Remediations Summary

Three audit findings closed from cluster E (IPC + bot-safety): one HIGH (`/`-prefix command injection in `adapter.chat`), one MEDIUM (`cancelSubscription` bypassing the `shell.openExternal` allowlist), and one LOW (`supabase/.temp/` not git-ignored). Two new test files added (23 cases), vitest baseline preserved (zero regressions).

## Findings Closed

### H4 (HIGH) — `adapter.chat` Minecraft command injection

**Attack:** The single-layer Haiku LLM can be prompt-injected by a player into emitting text like `/op MyName`. Pre-fix, `adapter.chat` forwarded that verbatim to `bot.chat`, which Minecraft interprets as a server command (operator command in the example) — turning a chat surface into a privilege-escalation primitive.

**Mitigation:** `src/bot/adapter/minecraft/index.js` — replaced `chat: (text) => { try { bot.chat(text) } catch {} }` with a leading-slash guard using `/^\s*\//`. Any string whose first non-whitespace character is `/` is DROPPED (not stripped-and-sent, not escaped). Dropping is the right choice because a stripped slash command would still leak the attacker's intended payload as visible chat noise.

**Tests:** 7 cases in `src/bot/adapter/minecraft/index.test.js` (TDD RED → GREEN):
- Forwards normal text → `bot.chat('hello world')` called once.
- Drops `/op MyName` → `bot.chat` not called.
- Drops `   /op MyName` (leading whitespace) → `bot.chat` not called.
- Drops `\t/help` (leading tab) → `bot.chat` not called.
- Forwards `say /help in chat` (slash NOT leading) → `bot.chat` called once with verbatim text.
- Forwards empty string `''` → `bot.chat('')` called once.
- Emits `console.warn` when dropping (asserted via `vi.spyOn`).

**Commits:** `92f0ec2` (RED), `27b9868` (GREEN).

### H5 (MEDIUM) — `cancelSubscription` `shell.openExternal` bypass

**Attack:** `src/main/ipc.ts` `app:open-external` enforced a strict https-host + mailto allowlist inline (`sei.gg`, `dmca.copyright.gov`, `sei.lemonsqueezy.com`). But `src/main/cloud/proxyClient.ts` `cancelSubscription` called `shell.openExternal(portalUrl)` DIRECTLY with the proxy-supplied URL — no validation. A compromised proxy or MITM could have redirected the user to an arbitrary URL (phishing, malicious download, `javascript:` if the OS handler permitted, etc.).

**Mitigation:** Extracted the allowlist + protocol gate into `src/main/lib/externalUrlValidator.ts` as the single source of truth. Exports:
- `assertSafeExternalUrl(url: string): void` — throws unless allowlisted.
- `ALLOWED_HTTPS_HOSTS` — readonly tuple of allowlisted https hostnames.
- `ALLOWED_MAILTO` — readonly tuple of allowlisted mailto path-strings.

Wired the helper into both call sites via lazy `await import(…)` (matches the pre-existing lazy `import('electron')` pattern, no startup-graph change):
1. `src/main/ipc.ts` `app:open-external` handler — inline allowlist removed entirely.
2. `src/main/cloud/proxyClient.ts` `cancelSubscription` — validation BEFORE `shell.openExternal(portalUrl)`. Reject case maps to `PROXY_NO_PORTAL_URL` (semantically "no usable portal URL") to keep the renderer error map unchanged.

Host comparison is exact-equality (`Array.prototype.includes`, NOT substring / `endsWith`) — so `evil.sei.lemonsqueezy.com.attacker.tld` is rejected even though it contains the allowlisted host as a substring.

**Tests:** 16 cases in `src/main/lib/externalUrlValidator.test.ts`:
- 5 allow-path: `https://sei.gg/legal`, `https://www.sei.gg/`, `https://dmca.copyright.gov/`, `https://sei.lemonsqueezy.com/billing/customer-portal/xyz`, `https://customers.lemonsqueezy.com/portal/abc123?signature=xyz`, `mailto:dmca@sei.app`.
- 8 reject-path: substring-bypass against `sei.lemonsqueezy.com` and `sei.gg`; `http://` (wrong protocol); `javascript:`, `file:///`, `data:` schemes; `mailto:attacker@evil.tld`; malformed `'not a url'`.
- 2 sanity: exported lists verbatim-equal the documented set.

Plus the existing `src/main/cloud/proxyClient.test.ts` `cancelSubscription` suite (which exercises the wired validator) continues to pass — the proxy-portal allow path goes through.

**Commit:** `968b924`.

#### Deviation (Rule 2 — auto-add missing critical functionality)

Added `customers.lemonsqueezy.com` to `ALLOWED_HTTPS_HOSTS`. The plan's spec'd allowlist was the verbatim ipc.ts list (`sei.lemonsqueezy.com` only), but the LS Subscriptions API returns `data.attributes.urls.customer_portal` URLs hosted at `customers.lemonsqueezy.com` (see `proxy/src/billing/customerPortal.ts:62-65` for the proxy-side fetch, and `src/main/cloud/proxyClient.test.ts:440` for the existing pre-s09 test that mocks this exact URL). Without this host the H5 validator would block the legitimate cancellation flow in production. Exposed immediately by the existing cancelSubscription test failing during baseline verification — fixed inline, validator test extended to cover the new host, allowlist provenance comment expanded to document the rationale.

### L1 (LOW) — Secrets-scan miss: `supabase/.temp/` untracked

**Risk:** The Supabase CLI creates `supabase/.temp/` for transient local-dev state (gotrue version pin, etc.). It surfaced in `git status` as `?? supabase/.temp/` and could be accidentally committed during a `git add -A`.

**Mitigation:** Appended a single block to `.gitignore` under the existing `# Sei proxy (Fly.io app)` section:

```
# Supabase CLI scratch dir — generated by `supabase` local dev, never for source control
supabase/.temp/
```

`git check-ignore supabase/.temp/` exits 0. `git status --short` no longer lists the path.

**Commit:** `3c2c85a`.
