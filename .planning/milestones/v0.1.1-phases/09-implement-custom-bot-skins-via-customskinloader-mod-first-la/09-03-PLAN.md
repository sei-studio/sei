Legacy 64×32 → 64×64 conversion (canonical algorithm — references: https://minecraft.wiki/w/Skin#Versions):
```
Modern 64×64 skin format has the legs (back+right) and arms (back+right) in the bottom half (y=32..63),
which the legacy 64×32 format didn't have. Convert by:
  1. Allocate a 64×64 RGBA buffer
  2. Copy the entire 64×32 source into rows 0..31 of the destination
  3. For rows 32..63, MIRROR the legacy leg/arm pixels:
     - Legacy right leg occupies (0..15, 16..31)  → modern right leg in (0..15, 16..31) (already copied)
     - Legacy right arm occupies (40..55, 16..31) → modern right arm in (40..55, 16..31) (already copied)
     - Modern left leg (16..31, 48..63) ← MIRRORED legacy right leg
     - Modern left arm (32..47, 48..63) ← MIRRORED legacy right arm
  4. Fill the rest of rows 32..63 with transparent pixels (0,0,0,0)
The mirroring is canonical because vanilla MC duplicates+flips the right limbs into the left slot.
```

    **1A — Create `src/main/skinImageUtil.ts` (new file). Public exports:**

    ```ts
    /** Parsed PNG header dims. */
    export interface PngHeader { width: number; height: number; bitDepth: number; colorType: number; }

    /** Reads the IHDR chunk that immediately follows the 8-byte PNG signature. Throws on non-PNG / malformed input. */
    export function parsePngIhdr(buffer: Buffer): PngHeader;

    /**
     * Normalize a Mojang skin PNG to 64×64. If the source is already 64×64, returns the input unchanged.
     * If the source is 64×32 (legacy format), upscales to 64×64 via the canonical legacy → modern conversion
     * (mirror right leg/arm into the left-side slots, fill remaining bottom-half pixels transparent).
     * Throws if dimensions are anything else.
     */
    export function normalize64x64(buffer: Buffer): Buffer;
    ```

    Implementation outline:

    1. `parsePngIhdr`: validate the 8-byte signature (`89 50 4E 47 0D 0A 1A 0A`). The IHDR chunk follows: 4-byte length (must be 13), 4-byte type (`IHDR`), then 13 bytes of data starting at offset 16. Read `width = readUInt32BE(16)`, `height = readUInt32BE(20)`, `bitDepth = buffer[24]`, `colorType = buffer[25]`. Return as `PngHeader`.

    2. `normalize64x64`:
       - Call `parsePngIhdr(buffer)`. If `width === 64 && height === 64`, return the input unchanged (zero-copy fast path — modern skins).
       - If `width === 64 && height === 32`, run the legacy-conversion algorithm. The simplest correct implementation is to decode the PNG to raw RGBA, build a 64×64 RGBA buffer, then re-encode. Use Node's built-in `zlib` for the inflate/deflate; PNG decoding for our specific case (we know it's a Mojang skin — bit depth 8, RGBA colortype 6) is simple:
         - Walk the chunks, find IDAT, concatenate IDAT bytes, `zlib.inflateSync` → scanline rows (each row: 1 filter byte + 64×4 pixel bytes for 64-px-wide rows).
         - Reverse the filter (most Mojang skins are filter type 0 = none, but handle types 1-4 per spec).
         - Allocate destination 64×64 RGBA buffer (`Buffer.alloc(64*64*4)`), all zeros (transparent).
         - Copy raw 64×32 source pixels into destination rows 0..31 (the entire legacy image fits in the top half).
         - Mirror the legacy right leg (source rect x=0..15, y=16..31, 16×16 pixels) into the modern left leg slot (dest x=16..31, y=48..63). For each source pixel (x, y) → dest pixel (31 - x + 16, y + 32). That horizontal flip is what makes the limbs face the right direction on the model.
         - Mirror the legacy right arm (source rect x=40..55, y=16..31, 16×16 pixels) into the modern left arm slot (dest x=32..47, y=48..63). For each source pixel (x, y) → dest pixel (55 - x + 32 - 40, y + 32) = `(87 - x, y + 32)`.
         - Re-encode the destination buffer as a 64×64 PNG using the same minimal-encoder pattern as `scripts/build-default-skins.mjs` (PNG signature + IHDR chunk + IDAT chunk via `zlib.deflateSync(rowsWithFilterBytes)` + IEND, all with CRC32 footers).
         - Return the encoded Buffer.
       - Otherwise (any other dimensions) throw `Error('normalize64x64: unsupported dimensions ' + width + '×' + height + '; expected 64×64 or 64×32')`.

    Notes:
    - The mirroring formulas are derived from the wiki diagrams: legacy skins only define the right side of the body; modern skins have explicit left-side textures that vanilla MC fills by mirroring the right side. We replicate vanilla's behavior so the bot renders correctly in the user's MC client.
    - For simplicity in Phase 9 v1, only handle bit depth 8, color type 6 (RGBA). Notch's skin (the canonical test case) is exactly this format. Throw a clear error for any other combination; the renderer surfaces it via `MOJANG_LOOKUP_FAILED: unsupported PNG format`.
    - Re-encoder is hand-rolled (~50 lines) — same pattern as `scripts/build-default-skins.mjs` Plan 01. Do NOT introduce `pngjs` as a runtime dep; we don't need general PNG support, just the one legacy-conversion case.

    **1B — Create `src/main/mojangSkinLookup.ts` with the implementation outlined below.**

    Public exports: `lookupMojangSkin(username: string): Promise<MojangSkinResult>` and the `MojangSkinResult` interface (resolvedUsername, pngBytes, pngBase64, sha256, textureUrl, model).

    Constants: `const TIMEOUT_MS = 15_000;` — exactly 15 seconds per CONTEXT, no fudging.

    Implementation outline:

    1. Helper `fetchWithTimeout(url, ms)`: wraps `fetch(url, { signal: ac.signal, headers: { 'user-agent': 'sei-electron/0.1.0' } })` with an `AbortController` that fires after `ms`. Returns the Response. Clears the timer in finally.

    2. Helper `classify(err, stage)`: returns `new Error(`MOJANG_LOOKUP_FAILED: ${stage}: ${msg}`)`. Caller re-throws.

    3. Public `lookupMojangSkin(username)`:
       - Trim input. Throw `MOJANG_LOOKUP_FAILED: empty username` if empty. Reject anything not matching `/^[A-Za-z0-9_]{1,32}$/` with `MOJANG_LOOKUP_FAILED: invalid characters in username`.
       - **Step 1 (username → UUID):** `GET https://api.mojang.com/users/profiles/minecraft/${encodeURIComponent(trimmed)}`. Branches:
         * HTTP 204 or 404 → throw `MOJANG_LOOKUP_FAILED: no Minecraft account named ${trimmed}`
         * HTTP 429 → throw `MOJANG_LOOKUP_FAILED: Mojang rate-limited the lookup. Wait a minute and try again.`
         * !ok → throw `MOJANG_LOOKUP_FAILED: api.mojang.com responded ${status}`
         * 200 → parse json; require `typeof id === 'string' && typeof name === 'string'`; assign `uuid = id; canonicalName = name`.
       - **Step 2 (UUID → texture URL):** `GET https://sessionserver.mojang.com/session/minecraft/profile/${encodeURIComponent(uuid)}`. Branches:
         * 429 → throw `MOJANG_LOOKUP_FAILED: Mojang rate-limited the profile lookup. Wait a minute and try again.`
         * !ok → throw `MOJANG_LOOKUP_FAILED: sessionserver responded ${status}`
         * 200 → parse json; find `prop = properties.find(p => p.name === 'textures')`; throw `MOJANG_LOOKUP_FAILED: sessionserver returned no textures property` if missing or non-string `value`.
         * Decode: `decoded = JSON.parse(Buffer.from(tex.value, 'base64').toString('utf8'))`. Read `decoded.textures.SKIN.url`. Throw `MOJANG_LOOKUP_FAILED: profile has no skin set` when missing.
         * Read `decoded.textures.SKIN.metadata?.model`. If `=== 'slim'` set `model = 'slim'`; otherwise default to `'classic'`.
       - **Step 3 (PNG download):** `GET textureUrl`. Branches:
         * !ok → throw `MOJANG_LOOKUP_FAILED: textures.minecraft.net responded ${status}`
         * 200 → `rawPngBytes = Buffer.from(new Uint8Array(await r.arrayBuffer()))`. Verify PNG magic: `rawPngBytes[0..3] === 0x89 0x50 0x4E 0x47`. Throw `MOJANG_LOOKUP_FAILED: texture endpoint did not return a PNG` on mismatch.
       - **Step 4 (legacy normalization, WARNING 8):** Call `normalize64x64(rawPngBytes)` from skinImageUtil. Catch errors from the normalizer and re-throw with `MOJANG_LOOKUP_FAILED: ${err.message}` prefix. Use the returned (possibly-converted) buffer as `pngBytes` for the rest of the function.
       - Return `{ resolvedUsername, pngBytes, pngBase64: pngBytes.toString('base64'), sha256: crypto.createHash('sha256').update(pngBytes).digest('hex'), textureUrl, model }`.

    4. Every catch that did NOT originate from our own throw should re-throw via `classify(err, '<stage>')` to preserve the MOJANG_LOOKUP_FAILED prefix. Check `if (err instanceof Error && err.message.startsWith('MOJANG_LOOKUP_FAILED:')) throw err;` before each classify call so user-facing strings don't get double-prefixed.

    Three sequential awaits each carrying their own AbortController-backed 15s budget. Worst-case wall-clock = 45s; ok because the renderer just shows a "Searching skin..." indicator and can be cancelled by navigating away (Plan 06's UI cleanup unmounts the search field).

    Every throw starts with `MOJANG_LOOKUP_FAILED:` so the renderer's `classifyRendererError` and `ERROR_COPY[MOJANG_LOOKUP_FAILED]` both surface the same copy without needing a new heuristic. Plan 06's UI catches the rejected promise and shows `ERROR_COPY.MOJANG_LOOKUP_FAILED` directly.

    The 204/404 branch is the "no such user" case (RESEARCH §4 HTTP 204 trap) — treat as MOJANG_LOOKUP_FAILED with a specific suffix the UI can match for distinct copy.

    Do NOT cache the result — Plan 02's `applyPng` writes the PNG once the user clicks "Apply", and from then on the local file is the source of truth (no persistent dependency on Mojang per CONTEXT §decisions).
