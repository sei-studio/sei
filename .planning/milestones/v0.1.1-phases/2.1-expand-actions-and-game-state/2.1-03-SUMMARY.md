# Phase 2.1 Plan 03: Orchestrator Integration & Adversarial Verification — Summary

**One-liner:** Wired Plans 01/02's substrate into `orchestrator.js` — composeSnapshot now lands in every personality user message, the cached system prefix carries the new capability/primer/learning blocks, `look()` joins the personality tool list as a no-op snapshot-refresh signal, `lastActionResult` is fed back per D-35, and `closeContainerSession()` is invoked on every chain exit path (start, abort, hop-cap, error). Shipped `scripts/verify-phase2_1.js` with 31 structural assertions (default mode) plus an opt-in `--live` adversarial replay against Anthropic for ROADMAP §2.1 SC-3.

## What Was Built

### orchestrator.js wiring (Task 1)

- **Imports** consolidated: `composeSnapshot`, `capabilityParagraph`, `minecraftPrimer`, `stillLearningLine`, `closeContainerSession` added; `renderPersona`/`capHitLine` import line extended (no duplicates).
- **`personalityTools`** gains a 4th entry `look` with empty `input_schema` and the description from `ACTION_DESCRIPTIONS.look` ("Refresh your world snapshot…"). `ACTION_DESCRIPTIONS` extended with the matching `look:` entry.
- **`rebuildPersonalitySystem()`** now calls `anthropic.buildCachedSystem(SYSTEM_INSTRUCTIONS, renderPersona(...), capabilityParagraph(), minecraftPrimer(), stillLearningLine(), personalityTools)` — 6 args.
- **`renderUserContext(event, data, goalsSnapshot)`** rewritten: prepends a `World snapshot:` block (output of `composeSnapshot(bot, { goals: goalsSnapshot, lastActionResult })`) to the existing event/data lines. Snapshot lands AFTER the cache breakpoint (which sits on the tool block in the system prefix), so re-rendering every turn does not invalidate the cached prefix per D-27. Wrapped in try/catch that falls back to `'(snapshot unavailable)'` so a transient observer error never breaks dispatch.
- **`lastActionResult`** declared as orchestrator-scoped `let`. Updated after every successful or failed `registry.execute()` in both the personality `setGoals` loop and the movement tool-call loop. String results pass through; object-shape `{ ok }` results are stamped as `${name}:ok|fail`; thrown errors stamp `${name} error`.
- **`look()` handling** in `handleDispatch`: filtered alongside `say`/`setGoals`/`handOffToMovement`. Treated as a no-op — chain tracker has already counted the personality hop, snapshot will recompose on the next dispatch's `renderUserContext` call. If `look()` is the sole tool emitted (no handoff), `lastActionResult` is stamped `'looked'` and the chain terminates normally via the existing no-handoff branch (`chains.end(chainId); return`).
- **Container cleanup** (Pitfall 6) — `closeContainerSession()` invoked from:
  1. Chain start, **only on fresh chains** (skipped for FSM-driven continuations to avoid closing a session the user is mid-deposit on).
  2. All three `signal.aborted` early-return branches (pre-personality, pre-movement, post-movement, and inside the movement-tool loop).
  3. The `HopCapHit` catch arm.
  4. The `AbortError` catch arm.
  5. The generic dispatch-error catch arm.
  Each call wrapped `try { await closeContainerSession() } catch {}` so cleanup never throws.

What was deliberately **not** changed (per the plan's negative requirements):
- `MOVEMENT_SYSTEM`, `callMovement`, `movementToolsFor` — the movement LLM does not receive the snapshot (D-28).
- The `cache_control` breakpoint placement — still on the tool block inside `buildCachedSystem`.
- Any existing chain-tracker / hop-cap / debouncer / rate-bucket / circuit-breaker logic.

## Key Decisions Made

- **Chain-start `closeContainerSession()` is gated on `!isContinuation`** — FSM-driven continuations (e.g. an `openContainer` completion event re-dispatching the chain) must NOT close the session that just opened. Only fresh chains pay the safety-close cost.
- **`look()` alone ends the chain** rather than triggering an internal re-call — keeps the chain tracker simple and gives the FSM/event loop a chance to fire other dispatches before the LLM "thinks again". Snapshot freshness is guaranteed by the per-turn `composeSnapshot` call, not by an in-loop refresh.
- **Snapshot composition wrapped in try/catch** with `'(snapshot unavailable)'` fallback — observer modules touch a lot of mineflayer surface, and a transient `bot.entity` flap should not break the personality dispatch path.
- **Verifier uses paren-balanced arg counting** rather than a brittle regex — survives multi-line `buildCachedSystem` calls and trailing commas.
- **LIVE mode bundles its own minimal `personalityTools` + system prompt** rather than importing from `orchestrator.js`. Keeps the live test independent of orchestrator factory wiring (which expects a bot/registry/config).
