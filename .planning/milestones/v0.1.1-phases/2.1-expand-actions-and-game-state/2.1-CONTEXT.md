# Phase 2.1: Expand Actions & Game State - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Broaden the closed Zod action registry from `goTo`/`setGoals` to a tier-1 set of mineflayer primitives so Sei can actually play (mine, build, fight, eat, sleep, manage chests). Inject a compact text-rendered world snapshot (position, biome, time, vitals, inventory, nearby blocks/entities, goals) into the personality LLM context every turn so it's grounded in the present. Add a short capability paragraph and a "still learning, asks the human" persona trait to the cached prefix so Sei stops requesting impossibilities and asks the owner when ambiguous.

Out of scope: crafting (its own phase), vision/screenshot (v2 — VIS-01–04), SQLite persistence (Phase 3), Electron GUI (Phase 4). Hot-reload of persona/capability prompt (Phase 4).

</domain>

<decisions>
## Implementation Decisions

### Action Set (Tier 1)
- **D-22:** Tier-1 actions to register: `dig`, `placeBlock`, `equip`, `attackEntity`, `consumeItem`, `lookAt`, `dropItem`, `activateItem`, `sleep`, `openContainer`, `withdrawItem`, `depositItem`. Plus `look()` for on-demand snapshot refresh (see D-26).
- **D-23:** **Crafting deferred** to its own future phase — it needs recipe lookup, crafting-table pathing, and inventory shuffling, which is a phase, not an action.
- **D-24:** All new actions follow the Phase 2 contract: closed Zod schema in the registry, timeout-wrapped per CLAUDE.md ADR #5, AbortController-cancellable, full registry sent to movement LLM each call (D-03).

### Targeting Model
- **D-25:** Use mineflayer's built-in resolvers: `bot.findBlock` / `bot.findBlocks` for blocks, `bot.nearestEntity` for entities. Do not reinvent.
- Layered targeting:
  - **Default — by name:** movement LLM passes `{block:"oak_log", maxDistance:32}` → resolver returns nearest match.
  - **Disambiguation — by snapshot index:** `{target:"#3"}` when the snapshot lists ≥2 of the same name and the personality LLM has already chosen one. Snapshot handles have ~10s TTL; stale handles return a clean `"stale target"` error and the LLM re-snapshots.
  - **Escape hatch — by coordinate:** for cases like "place a torch *here*" or rare precision needs.

### World Snapshot (Personality LLM only)
- **D-26:** Compact line-oriented snapshot, soft cap **~500 tokens**, contents:
  - `pos`, `biome`, `time` (day/night + ticks)
  - `health`, `food`, `xp`
  - `holding` (item + durability)
  - `inventory` (stack-summarized, e.g. `oak_log×12`)
  - `nearby blocks` (radius 16, numbered `#N` handles with coords)
  - `nearby entities` (radius 24, numbered `#N` handles with coords)
  - `goals` (owner + self lists from the Phase 2 goal store)
- **D-27:** **Cadence — every personality turn**, injected *after* the cache breakpoint (cache stays warm). No delta-only mode. Plus a `look()` registry action for on-demand refresh.
- **D-28:** Snapshot is **personality-only**. Movement LLM never sees it. Indices live entirely in the personality layer.
- **D-29:** Truncation strategy: if radius scans return >N entries, keep the closest N and append `+K more` so the LLM knows the world is bigger than the slice. Specific N value is Claude's discretion (target the 500-token soft cap).

### Capability Overview & Persona Trait (Cached Prefix)
- **D-30:** Capability overview is a **single NL paragraph** (~150 tokens) added to the cached prefix, listing what Sei can do in prose with an explicit inline "can't" list (no crafting, riding, enchanting, brewing, redstone). Not a structured list — keeps the model on-character.
- **D-31:** **Compact Minecraft primer** (~300–500 tokens) added to the cached prefix: wood types per biome, common mob threats, basic tool→block matrix, day/night basics. Big enough to ground "be specific" hand-offs and prevent ambiguous intent.
- **D-32:** **New persona trait — "still learning, asks the human."** One short line added to the cached persona prefix:

  > "You're new to this world and still learning. When something doesn't work and you're not sure why, or when a request is ambiguous, ask your owner — don't guess and don't pretend to know. Asking is on-character, not a failure."

  Unifies three behaviors: asking when intent is ambiguous (which wood?), declining impossibilities in-character, and asking when a *what*-only failure result needs a *why*.

### Movement LLM Contract (No Bloat)
- **D-33:** Movement LLM's runtime job stays minimal: **NL intent → tool_call.** No prose generation, no snapshot interpretation, no result paraphrasing. Runtime prompt = cached tool definitions + one NL sentence.
- **D-34:** Personality LLM owns target selection (by name, with optional `#N` from snapshot). Hand-off sentences are pre-resolved to be specific — "dig the closest birch_log within 16 blocks", not "get wood".

### Failure Feedback
- **D-35:** Action handlers return **deterministic, what-only result strings**, surfaced to the personality LLM on its next turn as `last_action_result`. Examples: `"dug oak_log"`, `"chest empty"`, `"cannot dig stone"`, `"couldn't reach target"`, `"inventory full"`. **No causal reasoning in the handler** (no "need a pickaxe") — that's hard to compute correctly and the personality should ask the owner instead per D-32.
- **D-36:** Snapshot also reflects new state on the next turn (belt-and-suspenders). Movement LLM never produces prose.

### Verification
- **D-37:** Verify harness gains an **adversarial prompt set** (~6–10 prompts: "craft me a sword", "ride that pig", "enchant my pickaxe", "brew a potion", etc.) that asserts Sei declines in-character and either offers an alternative or asks the owner. Covers ROADMAP success criterion #3 for this phase.

### Claude's Discretion
- Exact Zod shapes for each new action (follow Phase 2 patterns)
- Specific timeout values per action (reasonable defaults: dig 8s, place 4s, attackEntity 6s, sleep 12s, openContainer 6s, transfer 4s)
- Truncation cutoffs for nearby-blocks/entities lists (target 500-token snapshot soft cap)
- Internal layout of `observers/` files within `src/`
- Snapshot handle TTL implementation (per-handle expiry vs. per-snapshot replace)
- Error string wording (so long as it stays *what*-only, deterministic, and short)
- Capability paragraph and Minecraft primer exact wording (constraints captured in D-30/D-31/D-32)

</decisions>

<specifics>
## Specific Ideas

- The "still learning, asks the human" trait (D-32) is the load-bearing personality move of this phase. It converts every gap (impossibility, ambiguity, unexplained failure) into in-character dialogue. Without it, Sei either guesses (breaks immersion) or gets stuck.
- The snapshot's `#N` handles are the bridge between Haiku's good reasoning and Qwen's narrow tool-calling: Haiku resolves "the closest birch" to `#3` in the snapshot, Qwen just emits `dig({block:"birch_log"})` (or `{target:"#3"}` if disambiguating). Qwen never sees the snapshot.
- Capability paragraph + Minecraft primer + learning-trait line all extend the cached prefix together — the cost is paid once per session per persona/registry/primer change.

</specifics>
