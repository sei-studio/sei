   ```js
     lastActionResult = `${call.name} error`
     logger.warn(...)

   ```

   ```js

     const snapshot = composeSnapshot(bot, { goals: goalsSnapshot, lastActionResult })
     return [
       'World snapshot:',
       snapshot,
       '',
       `Event: ${event}`,
       `Data: ${JSON.stringify(data ?? {})}`,
     ].join('\n')
   }
   ```
   The snapshot lands in the user message (which sits AFTER the cached system blocks in the API call), so re-rendering every turn does NOT invalidate the cached prefix per D-27.

6. **look() handling in handleDispatch** — after destructuring `personalityResp.toolUses`, branch on `look`:

   ```js
   // so simply receiving this tool_call signals "the LLM wants another think with fresh eyes".
   // No action required — the chain tracker has already counted this personality hop.
   ```

   - Wrap each call in `try { await closeContainerSession() } catch {}` so cleanup never throws.

8. **Snapshot dirty flag** — NOT needed. The snapshot is recomputed every personality call by step 5. Per 2.1-RESEARCH.md, this is the agreed simplification: `look()` is meaningful as an LLM intent ("I want to look again") but operationally is a no-op because the next dispatch will re-render anyway.

Do NOT add the snapshot to the movement LLM's input (D-28: snapshot is personality-only).

Do NOT remove or rename existing personalityTools entries.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Snapshot text -> Anthropic user message | Crosses from local bot state into Anthropic API. Already inside Phase 2 trust assumption. |
| look() tool_call -> orchestrator | Untrusted personality LLM call; no-op handler so no attack surface. |
| chain end / abort -> closeContainerSession | Internal invariant; failure is already swallowed via try/catch. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2.1.3-01 | Information Disclosure | snapshot in user message | accept | Snapshot text contains only public game-state info; same trust boundary as Phase 2 user-message content. No new disclosure. |
| T-2.1.3-02 | Tampering | container session leak across chains | mitigate | closeContainerSession called on chain start, chain end, and abort branches. Verify-phase2_1.js doesn't directly check this; Plan 02 task 2 handler-level cleanup is the second layer of defense. |
| T-2.1.3-03 | Denial of Service | look() loop | mitigate | look() consumes a hop in chains.increment, so hop cap (5) bounds any LLM "keep looking" loop. |
| T-2.1.3-04 | Spoofing | adversarial prompt set | mitigate | D-37 verification harness asserts personality LLM declines/asks rather than fabricating tool_calls for impossible capabilities. |
| T-2.1.3-05 | Repudiation | last_action_result fed back to LLM | accept | Result strings are deterministic and non-causal (D-35). Cannot be used to leak handler state beyond what the LLM already sees. |
</threat_model>
