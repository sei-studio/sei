# Phase 03.1 Plan 09: Minecraft Adapter Behaviors (Gap Closure) Summary

Closed five Minecraft adapter and pathfind gaps surfaced in the Phase 03.1 verification: dropItem now aggregates across multiple inventory slots (so a 1+9 oak_log split actually drops 10), attackEntity refuses item-class / orb / global entities before any swing, stopFollow clears the active target so the snapshot's follow_target line reads `(none)` after unfollow, the orchestrator now injects a one-shot Pathfinder-rule nudge on the second identical cant_reach so the LLM escalates to ask-for-help instead of silently retrying, and the adapter exposes an idempotent `detach()` invoked from the boot composer's `onEnd` and `stop()` to remove the bot-listener leak window across reconnects.

## Coverage

This plan addresses 5 of 17 Bucket A items from the Phase 03.1 verification report:

- **NEW-W-A** (#7) — dropItem(count=10) actually drops 9 (off-by-one between hand-slot and inventory accounting)
- **D-H-15** (#8) — attackEntity on item entities (dropped items can't die, burns iterations)
- **D-H-16** (#9) — stale follow_target in snapshot persists after explicit unfollow
- **D-W-7** (#10) — closest-distance hint visible but bot does NOT invoke "ask for help" rule on repeated cant_reach
- **WR-07** (#16) — adapter listeners not cleaned on reconnect (OLD bot's listeners only GC-eligible on closure release)

## Tasks Executed

### Task 1 (TDD): dropItem multi-slot accounting + attackEntity item-class refusal (NEW-W-A, D-H-15)

**RED commit:** `af37e5c` — `test(03.1-09): add failing tests for dropItem multi-slot + attack item-refuse`
**GREEN commit:** `8bc9d57` — `feat(03.1-09): dropItem multi-slot accounting + attackEntity item-class refusal`

**RED phase** — both new test runners failed as expected:
- `dropItemCount` T1: `dropped 1 oak_log` (expected `dropped 10 oak_log`) — proved the off-by-one on split slots.
- `attackItemRefuse` T5: `attack failed (item): live.position.offset is not a function` — proved the item-class refusal branch did not exist (control flow fell through to the bot.attack path which then crashed on the test's mock entity shape).

**GREEN phase** — two atomic file changes:

1. **`src/adapter/minecraft/behaviors/drop.js`** — replaced `find(...)` single-slot accounting with `filter(...)` + `totalAvailable` aggregation. Walks matching slots oldest→newest, issues sequential `bot.toss(slot.type, slot.metadata ?? null, fromThisSlot)` calls until the running total hits `toDrop = min(requested, totalAvailable)`. Each toss races against a shrinking timeout budget (`tmoBudget = max(50, timeoutMs - elapsed)`) and the abort signal — preserves the wall-clock cap across N tosses. On undercount returns `dropped N item (only N available)`; on partial-success-then-failure returns `dropped N item then <reason>`.

2. **`src/adapter/minecraft/behaviors/attack.js`** — added three refusal branches between the existing player-refusal and the resolve-name path:
   - `entity.type === 'object' || entity.name === 'item'` → `cannot attack <label> (dropped item entity)` (covers mineflayer versions where dropped items might not report `type==='object'`)
   - `entity.type === 'orb'` → `cannot attack xp orb`
   - `entity.type === 'global'` → `cannot attack global entity`

**Verification:** `node scripts/test-dropItemCount.mjs` → `dropItemCount: all cases passed`. `node scripts/test-attackItemRefuse.mjs` → `attackItemRefuse: all cases passed`. All 4 acceptance grep gates pass (totalAvailable=3, "(dropped item entity)"=1, "cannot attack xp orb"=1, OLD `Math.min(count, invItem.count)` line gone).

## Behavioral Impact (qualitative)

After this plan, in the next live replay we expect:

1. **`drop 10 oak_log` actually drops 10** — split-stack inventory is no longer an off-by-one trap. If the user only has 7 across multiple slots, the bot replies "dropped 7 oak_log (only 7 available)" instead of silently dropping fewer than asked.
2. **No more `attackEntity` swings on dropped items / xp orbs / projectiles** — the bot returns `cannot attack item (dropped item entity)` or `cannot attack xp orb` immediately, the LLM gets a clear refusal, and the iteration moves on instead of burning swings.
3. **After `unfollow`, the snapshot's `follow_target` reads `(none)`** — no more stale `SSk1tz` lingering in seeds where the model thinks it's still following the owner.
4. **On the second `cant_reach` to the same destination, the bot escalates** — post-replay logs will show `[cant_reach 2× to (X,Y,Z) range=R — per the Pathfinder rule, do NOT retry...]` injected into the next user turn, and the model will say something like "stuck on the path, can you come closer?" instead of repeating the same goTo. The 6-consecutive-cant_reach pattern observed against moving SSk1tz in the verification log should not recur.
5. **No more bot-listener leaks across reconnects** — `_adapter.detach()` runs in `onEnd` before `_bot = null`, so the OLD bot's wireBotEvents listeners are explicitly disposed instead of relying on closure GC.

These claims are forward-looking; verification requires the next live replay session under similar conditions to wood-postfix / hunt+sand-postfix.
