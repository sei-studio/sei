## Summary

Re-review covering the gap-closure changes from plans 03.1-07 / 08 / 09 / 10
(commits roughly 607607f..b335def). The previous 03.1-REVIEW.md (commit
3bc6fa8) covered the pre-gap-closure state and is replaced here.

The gap-closure work largely lands cleanly. The `fileLock` primitive (Plan
03.1-08 WR-06) is well-structured, the FSM drain fix (WR-01) is correct and
covered by a focused test, and the file-lock + sync-throw tests give good
regression coverage. The Plan 03.1-09 dropItem multi-slot aggregation fix is
correct and tested.

The most material issues found:

1. **WR-04 preserved-interrupt re-enqueue carries the wrong `username`.** The
   chat-arrival branch at orchestrator.js:618 only stores `chatText` on
   `pendingInterrupt`; the attack branch at line 641 then reads
   `pendingInterrupt.who` (always undefined) and falls back to `data?.who`
   from the *attack* event's data (which only carries `attacker` /
   `attackerLabel`, never `who`). The re-enqueued chat event therefore always
   has `username: 'owner'` literal — losing the actual chatter's identity for
   `pushOwner`, the seed-turn `who:` label, and the dedup signature.
2. **`loopHasAffect` regex matches against full user-text blocks including
   snapshot and event boilerplate.** Words like `please`, `love`, `i am`
   appearing in arbitrary owner chat (or, in some cases, snapshot text or
   non-owner chat embedded in the `event` block) will over-trigger diary
   writes. By design the comment claims "we err on the side of recall," but
   the regex is broad enough that the documented invariant ("safe against
   surrounding boilerplate") does not hold across all snapshot variants.

The findings below are graded by severity. No CRITICAL findings (no security
vulnerabilities, data-loss, or crash paths discovered in the changed surface).

## Warnings

### WR-01: WR-04 preserved-interrupt re-enqueue uses wrong `username`

**File:** `src/brain/orchestrator.js:615-647, 844-852`
**Issue:** The chat-arrival branch stores only `{ chatText }`:

```js
// line 618
pendingInterrupt = { chatText: String(chatText) }
```

Later, when `sei:attacked` arrives mid-loop, the attack branch reads
`pendingInterrupt.who`:

```js
// line 640-641
const preservedInterrupt = pendingInterrupt
  ? { chatText: pendingInterrupt.chatText, who: pendingInterrupt.who ?? data?.who ?? 'owner' }
  : null
```

`pendingInterrupt.who` is always `undefined` (never written). `data` here is
the **attack** event's payload, which carries `attacker` / `attackerLabel` /
`attackerKind` but never `who` or `username`. Therefore `data?.who` is also
always `undefined`. The fallback chain collapses to the literal string
`'owner'`. When the chat is re-enqueued at line 847:

```js
reenqueue('sei:chat_received', {
  username: pa.preservedInterrupt.who,  // always 'owner'
  text: pa.preservedInterrupt.chatText,
  ...
})
```

…the brain's `onChat` handler (`src/brain/index.js:194`) then calls
`orchestrator.recordIncomingChat('owner', text)` — which stores the line in
`convoMemory.recentChat.pushOwner` with `who='owner'`, mis-attributing it in
the next loop's `recent_owner_chat` block. The dedup signature
(`${who}:${chatText}:${bucket}`) also uses the literal `'owner'`, defeating
the dedup for any second arrival.

This is masked by the fact that the bot's `owner_username` is rarely the
literal string `'owner'`, so the LLM sees a foreign speaker name in the seed
turn. Verification trails will show the wrong who in `recent_owner_chat`.

**Fix:**
```js
// orchestrator.js:608-618 — capture username at chat-arrival time
const who = data?.username ?? data?.who ?? 'owner'
const chatText = (data && (data.text ?? data.message)) ?? JSON.stringify(data ?? {})
const sig = `${who}:${chatText}:${Math.floor((data?.ts ?? Date.now()) / 500)}`
if (shouldPreserveInterrupt(currentLoop, sig)) {
  pendingInterrupt = { chatText: String(chatText), who }   // ← add who
  try { currentLoop.abortController.abort() } catch {}
}
```
Then line 641 simplifies to `who: pendingInterrupt.who` (no fallback to
attack-event data needed).

### WR-02: `loopHasAffect` regex matches against snapshot/event boilerplate, not just owner chat

**File:** `src/brain/sessionState.js:93-120`
**Issue:** The comment block claims:

> Owner-chat is delivered via the seed-turn `event` block on first turn (e.g.
> `Event: sei:chat_received\nData: text: thanks`), and via `PLAYER INTERRUPT:
> <text>` on subsequent turns. The regex below catches both shapes — the
> keywords are rare enough in the surrounding boilerplate (Event, PLAYER
> INTERRUPT, snapshot text) to be safely matched against the full block text.

This invariant does not hold. The regex runs against EVERY user-role text
block, including the `snapshot` block, which carries free-form world prose
(biome names, structure names, mob lists, etc.), and the `event` block,
which embeds verbatim chat text from `formatEventData(event, data)` —
including non-owner chat through the `text: ${data.text}` branch.

The regex is broad:

```js
/\b(good job|thanks|thank you|nice work|love|appreciate|sorry|please|name is|call me|i am)\b/i
```

Realistic collisions:
- A non-owner player saying "thanks" in chat triggers `batchHasAffect = true`
  and a diary flush — confabulating an "affect" entry from a bystander.
- `\bplease\b` matches verbatim if any player says it.
- `\bi am\b` matches owner self-narration ("i am hungry") — possibly fine,
  but the gate is wider than the intent ("praise / preference / name").

**Fix:** narrow the match surface to extracted chat payloads only:

```js
function extractChatPayload(text) {
  // Owner-chat seed-turn shape: `Event: ...\nData: text: <body>${addendum}`
  const m1 = text.match(/Data:\s*text:\s*([^\n]+)/)
  if (m1) return m1[1]
  // Subsequent-turn shape: `PLAYER INTERRUPT: <body>`
  const m2 = text.match(/^PLAYER INTERRUPT:\s*(.+)$/m)
  if (m2) return m2[1]
  return null
}
// inside loopHasAffect user-block branch:
const chat = extractChatPayload(t)
if (chat && AFFECT_RE.test(chat)) return true
```

Add a unit test that asserts `loopHasAffect` returns `false` for a snapshot
block containing the substrings "love" or "please."

### WR-03: Stored `pendingInterrupt` shape is missing `ts` for dedup-bucket invariance across attack preservation

**File:** `src/brain/orchestrator.js:608-619, 640-647`
**Issue:** The dedup-bucket signature is
`${who}:${chatText}:${Math.floor((data?.ts ?? Date.now()) / 500)}`. When an
attack preempts an interrupt and we re-enqueue the chat at line 847, the
re-enqueue does NOT pass through a `ts` field. The brain layer's `onChat`
(index.js:189-206) similarly does not stamp `ts`. The next time this chat
dispatch arrives at `handleDispatch`, `data?.ts` is undefined and `Date.now()`
is sampled fresh — so a duplicate **arriving in the same 500ms bucket as the
original** will not dedup against itself across the attack-preserve path. Net
effect: the chat may be processed twice if the adapter re-emits within the
bucket (rare, but possible during reconnect chat-replay).

**Fix:** at re-enqueue (orchestrator.js:846), include the original `ts`:

```js
reenqueue('sei:chat_received', {
  username: pa.preservedInterrupt.who,
  text: pa.preservedInterrupt.chatText,
  message: pa.preservedInterrupt.chatText,
  ownerSpoke: true,
  ts: pa.preservedInterrupt.ts ?? Date.now(),
}, 1)
```

…and capture `ts` at the original chat-arrival site (line 618) into
`pendingInterrupt`.

### WR-04: `bridgeExternalAbort` early-return on already-aborted external signal does not record listener; `replaceAbortController` instantly aborts the new controller without graceful-close path

**File:** `src/brain/orchestrator.js:1269-1283, 1294-1306`
**Issue:** When `bridgeExternalAbort` is invoked from `replaceAbortController`
and the external signal is **already aborted**, the function aborts the
freshly-created internal controller and returns WITHOUT installing or
recording a listener:

```js
if (ext.aborted) {
  try { loop.abortController.abort() } catch {}
  return
}
```

This is fine on its own, but the call sequence is:
1. abort fires → catch arm runs
2. `replaceAbortController` creates `fresh = new AbortController()`
3. `bridgeExternalAbort(loop)` runs against `fresh`; sees `ext.aborted`
   → aborts `fresh` immediately
4. Next `while` iteration: `signal.aborted === true` → throws AbortError
   → catch arm runs again → `repairAfterAbort` synthesizes ANOTHER PLAYER
   INTERRUPT user turn (or no-op since `pendingInterrupt` was cleared at
   the previous catch arm at line 1203).

In the second pass, `repairAfterAbort` (line 1325) finds the last
assistant turn — which is now an empty assistant turn (no tool_uses since
last loop just appended an interrupt-response cycle). `aborted.length === 0`
so it falls through to `appendUserTurn(...)` with another `PLAYER INTERRUPT`
text-only turn. We've now appended **two** PLAYER INTERRUPT user turns in
a row with no intervening assistant response, which Anthropic will accept
but creates a misshapen conversation history. There's also no progress-
making out of this — the loop will keep trying until `iteration_cap`.

In practice this is gated by the FSM clearing the external signal between
preemptions, so the re-entry is rare. But the test
(`scripts/test-attackInterruptRace.mjs`) covers `_setAbortController` only
in isolation, not the full bridge re-entry, and the failure mode is silent
(no thrown error, just a degenerate conversation).

**Fix:** if `replaceAbortController` is asked to install a fresh controller
and the external signal is already aborted, route through `gracefulCapClose`
(or a new `gracefulExternalAbortClose`) instead of producing a controller
born aborted. Alternatively, document the contract clearly and add a unit
test that drives the re-entry against a real loop.

### WR-05: `attackInterruptRace.mjs` test does not actually exercise WR-04

**File:** `scripts/test-attackInterruptRace.mjs:1-57`
**Issue:** The file's own header acknowledges this:

```js
// WR-04's full end-to-end flow is grep-verified + live replay;
// the unit-level proof here is that loop's abortController getter returns
// the swapped instance after _setAbortController.
```

…but WR-04 is the change with the highest blast radius (preserved interrupt
re-enqueue including `username`, `text`, `ownerSpoke`, ordering relative to
P0 attack). The "grep gates and live replay" approach gives qualitative
confidence but does NOT detect:
- finding WR-01 above (`username` always 'owner'),
- finding WR-03 above (no `ts` propagation),
- ordering regressions if FSM priority constants change.

A pure unit test could spy on `reenqueue` and assert the call shape:

```js
const calls = []
const reenqueue = (event, data, priority) => calls.push({ event, data, priority })
// ... drive handleDispatch with a chat then attack ...
assert.equal(calls[0].event, 'sei:attacked')
assert.equal(calls[1].event, 'sei:chat_received')
assert.equal(calls[1].data.username, /* the original chatter, NOT 'owner' */)
assert.equal(calls[1].data.text, /* original text */)
assert.equal(calls[1].priority, 1) // P1
```

Without this, regressions in WR-04 will not surface in CI — they will only
show up as misattribution in `recent_owner_chat` blocks. The file is also
named `attackInterruptRace.mjs` but tests `loop._setAbortController` only;
the filename is misleading.

**Fix:** extend the test to spy on `reenqueue` and verify the WR-04 contract,
or rename the file to reflect what it actually tests
(e.g. `test-loopAbortControllerSwap.mjs`).

### IN-04: `affectLog` factory and standalone `readAffectFull` create dual access paths to AFFECT.md

### IN-05: `processNext` re-queue branch uses `queue.unshift` after a previously sorted insert — order may flip relative to first-arrival for same-priority items

**File:** `src/brain/fsm.js:114-127`
**Issue:** When a lower-priority item is dequeued during a higher-priority
hold, it's re-pushed via `queue.unshift(item)`. But `enqueue()` always sorts
the queue. The re-queued item sits at index 0 until the next `enqueue`
re-sorts. If two items at the same priority compete after a hold, the order
may flip relative to first-arrival.

This is unlikely to surface in practice (the reachable priorities are
distinct enough today), but the dispatch order between two same-priority
items can flip after a hold. WR-01's test (`test-fsmQueueDrain.mjs`) does
not cover the same-priority case.

**Fix:** prefer re-`enqueue` of the item rather than `unshift`, so the sort
restores deterministic order. Or use a stable sort and document FIFO within
priority.
