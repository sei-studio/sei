# Phase 03.1: Behavior polish and AI/game decoupling refactor — Research

**Researched:** 2026-05-06
**Domain:** LLM-driven game-companion architecture; prompt engineering; behavior polish
**Confidence:** HIGH for code-shape findings (read directly), MEDIUM for some token-trim recommendations (estimated, not measured live)

## Summary

This phase has three interlocking pieces:

1. **Behavior polish** — 37 cataloged defects across 4 logs cluster into ~8 themes. The biggest classes are (a) say()/think discipline, (b) parallel-action storms with stale `#N` indices, (c) pathfinder dead-loops with no fallback, (d) the diary write-side gating bug, and (e) tone (no "we" framing, all punctuation present).
2. **brain/ vs adapter/minecraft/ refactor** — most of `src/llm/`, `src/memory/`, and the FSM pieces are already game-agnostic in spirit. The seam is small: the brain dispatches **action calls** (string name + JSON args) and consumes a **world-summary text** plus a **typed event stream**. Only `bot.js`, `registry.js`, all of `src/behaviors/`, and the observer modules need to move. `orchestrator.js` is the big surprise — it currently imports follow.js and container.js directly, which is the only mineflayer leak in the brain layer.
3. **Token efficiency** — current cached prefix is ~5,300 tokens (system + persona + capability + primer + tools), per-turn dynamic blocks add ~600–4,000 tokens. The biggest non-cached cost is repeated tool_result blobs from parallel-dig storms. Fixing the `#N` index instability removes most of this *for free*.

**Primary recommendation:** Refactor first (3 plans), then run a prompt-tone wave (1 plan), then a memory-write-side wave (1 plan), then a defect-cleanup wave (1 plan). The refactor must precede prompt edits because the persona prompt is moving files. The memory-write-side fix is the highest-value behavior change in the entire phase — it's the only one that addresses "feels like a long-time friend".

## Project Constraints (from CLAUDE.md)

These directives must be honored by all plans:

- **Three-process Electron** is the long-term architecture. Mineflayer must stay in utilityProcess. The brain/adapter split should not break this — adapter still imports mineflayer, brain doesn't.
- **Closed action registry** — LLM never generates code or coordinates. The refactor must preserve this; brain → adapter calls cross the seam as `{name, args}` JSON, not closures.
- **Single Haiku call per iteration** combines reasoning + dispatch. No second layer.
- **Event-sourced FSM with priority queue, single outstanding action, AbortController.** Refactor must preserve P0/P1/P2/P2.5/P3 semantics.
- **Every external call has a timeout** — pathfinder, Anthropic. Don't introduce new untimed calls.
- **Cache breakpoints respected** — prompt edits MUST NOT change byte content of cached blocks (system instructions, persona, capability, primer, tool list) without intentional cost. The current cache_control marker sits on the LAST tool entry (260502-h6i).
- **GSD workflow** — plans cite log line numbers, commit planning docs alongside code, update STATE.md after phase complete.

## Defect taxonomy — 37 findings grouped into 8 themes

All defect IDs reference the four log-analysis files in `log-analysis/`. Total: 10 wood + 10 explore + 11 hunt-sand + 6 memory = **37**.

### Theme 1 — Say/think discipline (D-1, D-3 partial)
- **Defects:** D-W-5 (text contains player-prose, only first line said), D-E-2 (text→say verbatim copy), D-H-1 (first-turn say omitted on action), D-H-2 (text addresses player in 2nd person), D-H-10 (empty text on action turns), D-H-11 (26s of silent combat).
- **Root cause:** First-turn say() rule is advisory in the prompt, not enforced. When the LLM reaches for a tool action it treats prefatory `text` as fulfilling the speech rule.
- **Files:** `src/llm/orchestrator.js` (the per-loop seed assembly + the runIterations top-of-loop). System prompt lines 23–24 attempt the rule but fail.
- **Scope:** ~50 lines: orchestrator-level enforcement (intercept first iteration response if no `say` and no prior say in this loop) + a 1-line prompt strengthening.

### Theme 2 — Tone (D-2, D-7)
- **Defects:** D-E-3 (you/me dominant, punctuation everywhere), D-H-3 (only 1 "we" line in entire session), D-H-4 (punctuation everywhere despite preference).
- **Root cause:** No pronoun rule in persona prompt; no punctuation-strip post-processor.
- **Files:** `src/llm/persona.js` (TONE_LINES — add we-framing line), `src/llm/orchestrator.js` lines 643–648 (the say() handler).
- **Scope:** ~20 lines: one regex post-processor + one prompt edit. Cache impact: the prompt edit changes a cached block — accept the one-time bust.

### Theme 3 — Parallel action storms + stale `#N` indices (D-W-2, D-W-3, D-W-6, D-H-5, D-H-6, D-H-7)
- **Defects:** Wood D-W-2 (10-call dig batch wastes 9 on stale targets), D-W-3 (5 identical out-of-range retries), D-W-6 (`#1` = sand on turn 1, `#1` = jungle_log on turn 2), Hunt D-H-5 (7-way dig barrage with guaranteed abort), D-H-6 (follow returns "target gone" because attackEntity ran first in the same batch), D-H-7 (snapshot `#12` from pre-loop snapshot doesn't match in-loop snapshot's `#4`).
- **Root cause:** (a) `#N` indices are recomputed every snapshot by distance — they are turn-local but the prompt invites batched use. (b) The action registry permits unlimited parallel calls of the same type. (c) parallel `follow + attackEntity` has an ordering race.
- **Files:** `src/observers/snapshot.js` (handle minting), `src/observers/targeting.js` (handle table), `src/llm/orchestrator.js` (parallel-call dispatch loop, lines 640–679 — currently runs them sequentially `for` not `Promise.all`, but the LLM **emits** them as a batch and the prompt should discourage that).
- **Scope:** Multiple options, recommend (architectural):
  - **Snapshot side:** keep `#N` as numeric indices but **scope them per-snapshot** (which they already are — TTL is 10s) and add a clear prompt rule: "`#N` references are valid only for the snapshot that listed them; after any movement or dig, indices may have re-rotated. Prefer `{block: "<name>"}` for repeated digs of the same resource type."
  - **Registry side:** add a server-side "dig N of `<block>`" loop primitive (single-call batch) so the LLM stops simulating it with parallel calls. Or (simpler): cap parallel dig calls to 1 in the runIterations dispatch — if more than 1 dig is emitted, only run the first and synthesize "ignored: only one dig per turn allowed; subsequent block names will be reachable next turn" results for the rest. This is forceful but log-driven and cheap.
  - **follow+attackEntity:** the prompt already says "follow first, then attackEntity" — the bug is the *parallel* emission. Easiest fix: make follow a no-op (return "already following") when the same target was just resolved in this loop's prior assistant turn, and tell the prompt that in hunting `attackEntity` alone is enough since combat reflex auto-pursues moving mobs (per `combat.js` startAttacking).

### Theme 4 — Pathfinder dead-loops + no fallback (D-E-1, D-E-5, D-W-3, D-W-7, D-H-9)
- **Defects:** D-E-1 (9 consecutive cant_reach with no escape), D-E-5 (owner says "come here", bot stuck in water pit, gives up by chatting), D-W-3 (dig `maxDistance:32` but server caps at 4.5m — no auto-pathfind-to-range), D-W-7 (goTo to walking owner stale waypoint), D-H-9 (cant_reach fixed only by adding `range:3` next turn).
- **Root cause:** No fallback strategy when goTo fails repeatedly. Each turn picks a fresh coordinate. There's no "I tried X 3 times, escalate to Y" logic. dig's `maxDistance` parameter advertises a search radius, not a reach radius — the LLM reads it as "I can dig from 32m away".
- **Files:** `src/behaviors/pathfind.js`, `src/behaviors/dig.js`, the system prompt's hunting/digging rules.
- **Scope:**
  - Default `range: 2` for goTo when target is a player (D-H-9 is a 1-line config fix).
  - On goTo cant_reach, the action result already says "cant_reach" — extend with current-distance hint so the LLM knows whether it was 1m off or 30m off: `cant_reach (closest=8.4m to target)`.
  - Don't add bot-side auto-bridge/dig — that's complex and risky. Instead make the prompt rule more concrete: "if `cant_reach` fires twice in a row at the same coords, ask the owner for help or pick a different goal." The 13-line system prompt already has rule for this (orchestrator.js line 32) — strengthen it from "2+ failures" to "2 cant_reach in a row".
  - dig `maxDistance` description rewrite: clarify it's the **search radius** for finding the named block; the actual swing reach is fixed at 4.5m and the bot pathfinds into reach automatically.

### Theme 5 — Memory write-side gating (D-M-1, D-M-2, D-M-4, D-M-5, D-M-6)
- **Defects:** D-M-1 (praise/affect events never persisted because MUTATING_ACTIONS gate), D-M-2 (entries are prose blobs, not retrievable facts), D-M-4 (OWNER.md `## Notes` never written by the runtime), D-M-5 (confidently denies past praise instead of hedging), D-M-6 (loopHistory crowds out diary at compose time).
- **Root cause:** `sessionState.js:36-39, 285-333` — diary writes are gated on world-mutating actions only. Any session that's just chat (the sessions where praise actually happens) leaves zero diary trace. Plus diary entries are unstructured prose — even if praise *were* logged, recall queries can't surface it.
- **Files:** `src/llm/sessionState.js`, `src/memory/diary.js`, `src/memory/owner.js`, `src/llm/compaction.js`, `src/llm/orchestrator.js` (system prompt + tool list).
- **Scope:** This is the highest-impact item in the phase. Architectural — see "Memory write-side fix design" section below.

### Theme 6 — Diary content quality (D-W-9, D-E-10)
- **Defects:** D-W-9 (diary self-narration is florid prose: "quiet moments where we both just stood there in the dirt together"), D-E-10 (diary loaded every turn at ~5KB cached but doesn't change behavior).
- **Root cause:** Compaction prompt encourages narrative/sentiment without a length cap or tone constraint. Diary is stored as flavor text rather than an instruction-shaped fact ledger.
- **Files:** `src/llm/compaction.js` (SUMMARY_PROMPT_INTRO).
- **Scope:** ~15 lines — rewrite the compaction prompt for tone (no purple prose, no metaphors, declarative sentences only), add a hard length cap (e.g., 3 sentences, ≤ 80 words), and explicitly mention "include any praise, gratitude, or owner-stated preferences VERBATIM if observed in this batch".

### Theme 7 — Idle behavior shape (D-E-8, D-W-1, D-W-10)
- **Defects:** D-E-8 (idle tick produces another goTo+chat instead of standing/observing), D-W-1 (loop_end re-fires fresh loop and bot self-assigns sand-gathering with no instruction), D-W-10 (drops 10 sand silently when priorities shift).
- **Root cause:** Idle and loop_end seed addenda say "you are a peer, pick something" — which the model interprets as "do something" rather than "watch / comment / wait". loop_end specifically nudges toward action ("decide a follow-up sub-goal") rather than equilibrium.
- **Files:** `src/llm/orchestrator.js` lines 418–432 (event-specific seed addenda).
- **Scope:** ~20 lines — split idle from loop_end semantically. idle = "you've been quiet for a minute. observe something, comment if natural, otherwise stay silent — silence is fine, no tools required". loop_end = "you finished a task. settle, no need to start a new one unless an explicit goal is pending". Plus: explicitly forbid auto-starting world-mutating actions on idle / loop_end without an owner_goal or self_goal that demands it. **D-W-10 fix:** add a rule that destructive inventory ops (dropItem ≥ 4 of anything visible to owner, depositItem to non-owner-context) must be paired with a `say()` first.

### Theme 8 — Loop control / interrupt edge cases (D-W-8, D-H-8, D-W-4)
- **Defects:** D-W-8 (P0 attack preempts a queued chat ack — owner's "hello?" gets ghosted), D-H-8 (PLAYER INTERRUPT injected twice for one chat — wasted LLM call), D-W-4 (re-narrating inventory every turn).
- **Root cause:** D-W-8 — when sei:attacked re-fires after current loop's finally, any unspoken queued say() is lost. D-H-8 — race between in-flight haiku call and the interrupt-injection: both preserve, only one inlines the chat text. D-W-4 — system prompt line 24 says "do NOT restate inventory" but the model does it anyway because it has no other lead-in for thinking.
- **Files:** `src/llm/orchestrator.js`.
- **Scope:** D-W-8 is real but small (~10 lines — track unspoken acknowledge state across attack-rerun). D-H-8 is a known-but-rare race; investigate but don't block on it. D-W-4 is partly prompt-tweak, partly already-tried; the better fix is making `text` count as private reasoning more strictly via the orchestrator (skip empty-text turns from the convoMemory self-buffer entirely so the model doesn't see itself restating inventory yesterday).

### Brain ↔ Adapter interface contract

The brain depends on this interface (and nothing else mineflayer-shaped):

```js
// src/brain/types.js (JSDoc/conceptual; no Zod yet)

/**
 * @typedef {Object} Adapter
 *
 * Action surface — brain calls these by name+args (closed registry):
 * @property {() => string[]}                     listActions()
 * @property {(name) => ZodSchema|null}           getActionSchema(name)
 * @property {(name) => string}                   getActionDescription(name)
 * @property {(name, args, ctx) => Promise<string>}
 *                                                executeAction(name, args, { signal, goalStore })
 *
 * World perception — brain consumes plain text it doesn't have to understand:
 * @property {() => SnapshotComposer}             createSnapshotComposer()
 *   // SnapshotComposer = { next({goals, lastActionResult, inFlight}): string, reset() }
 * @property {() => string}                       worldPrimer()
 *   // The "Quick world primer" content; replaces the hardcoded MINECRAFT_PRIMER
 *
 * Session lifecycle — brain hooks these to drive memory:
 * @property {(handlers: AdapterHandlers) => void} attach(handlers)
 *   // AdapterHandlers = {
 *   //   onPlayerJoined(player), onPlayerLeft(player),
 *   //   onChat({username, text, ownerSpoke, addressed, nearby}),
 *   //   onAttacked({attacker, attackerLabel, attackerKind}),
 *   //   onSpawn(),
 *   // }
 *
 * Effects the brain commands but can't synthesize:
 * @property {(text: string) => void}             chat(text)        // adapter does say()
 * @property {(fn) => void}                       setInflightProvider(fn)
 * @property {() => Promise<void>}                closeAnySessions() // container, etc.
 *
 * Optional capabilities (brain checks before using):
 * @property {boolean}                            supportsAutoEat
 * @property {boolean}                            supportsFollow
 *
 * Identity:
 * @property {string}                             botUsername       // for self-message filtering
 * @property {() => Object<string, Player>}       getKnownPlayers()
 */
```

`Player` is generic: `{ username: string, uuid: string }`. The brain never sees a mineflayer Entity.

The `executeAction` return is **always a string** describing the result (the existing convention — "dug oak_log", "out of range (5.4m, need ≤4.5)", "stale target", etc.). This is already the codebase convention (see all of `behaviors/`); the refactor just makes it the documented contract.

The `Snapshot` is also always a plain string. The format is currently a multi-line `key: value` document with `nearby blocks: ...`, `nearby entities: ...`, `recent_events: ...`, etc. The brain reads it byte-for-byte into the seed user turn. The adapter owns the format. Recommend renaming `composeSnapshot` → `composeWorldSummary` and the snapshot block name → `world_summary` for game-neutral naming.

A swappable Stardew/Roblox/Wuthering-Waves adapter would implement this same interface: it ships its own behaviors (e.g., `harvest`, `gift`, `chop`), its own world-summary composer, its own primer. The brain code path doesn't care.

## Token-efficiency audit

### Current cached prefix (~5,300 tokens, byte-stable)

Per `anthropicClient.js:79-90`, the cached system blocks are:

1. `SYSTEM_INSTRUCTIONS` (orchestrator.js:19-38) — ~600 tokens. 18 numbered rules. Some redundancy:
   - "Use `say` frequently" (line 23) and "say() cadence: REQUIRED on FIRST and LAST" (line 24) overlap
   - "Movement rule: AT MOST ONE TYPE" (line 27) and "In-flight rule" (line 28) are both about not stepping on yourself
   - Could trim ~150 tokens by merging.
2. `renderPersona(persona) + '\n' + STILL_LEARNING_LINE` — ~80 tokens
3. `CAPABILITY_PARAGRAPH` — ~150 tokens (lines 43-53 of persona.js)
4. `MINECRAFT_PRIMER` — ~430 tokens (lines 55-74 of persona.js). Largest cached block.
5. Tool list — ~500 tokens with cache_control marker

**Refactor opportunity:** moving the primer to adapter-supplied makes its content a per-adapter choice. For minecraft we keep the existing primer (it's well-tuned). The byte-stability requirement still holds.

### Per-turn dynamic blocks (NOT cached, billed every turn)

Per orchestrator.js:115-144 (`composeSeedBlocks`) — first user turn of every Loop:

| Block | Bytes | Tokens | Cost driver |
|-------|-------|--------|-------------|
| seed_owner | <1024 (config cap) | ~200 | mostly empty in current state (D-M-4) |
| seed_diary | up to 3072 (config cap) | ~768 | the D-W-9 purple-prose problem |
| recent_loop_history | ~600 (20 entries × ~30 chars) | ~150 | from convoMemory loopHistory |
| recent_owner_chat | ~600 (10 entries × ~60 chars) | ~150 | from convoMemory recentChat |
| your_recent_messages | ~600 | ~150 | from convoMemory recentChat self side |
| event | ~80 + addendum (~200) | ~70 | per-event customized |
| snapshot | 1.5–3 KB | ~500 | the `nearby blocks` + `around feet` + entities + `recent_events` |

**Total dynamic seed:** ~2,000 tokens uncached on every fresh Loop's first turn.

Plus per-iteration tool_result blobs grow as the loop runs — wood-findings reports cache_creation spike of 3,831 tokens at line 203 due to a 10-call dig batch's tool_result array being re-prefixed. **This is the single biggest waste in the current logs and goes away with theme-3 fixes.**

### Recommended trims

- **Trim 1 (free):** Cap parallel dig calls at 1 (theme 3) → kills the tool_result blob storm. Estimated 500–3000 token saving per dig-heavy turn.
- **Trim 2 (cheap):** Compaction prompt rewrite (theme 6) → diary entries shrink from ~600 B prose paragraphs to ~250 B fact blocks. Saves ~400 tokens of cached seed_diary (cache invalidates once on the prompt rewrite, then is rebuilt).
- **Trim 3 (cache cost):** SYSTEM_INSTRUCTIONS merge of overlapping rules → save ~150 tokens. ONE-TIME cache bust required.
- **Trim 4 (zero risk):** loopHistory is currently 20 entries; the snapshot itself shows recent_events deltas, and per D-M-6 the loopHistory crowds out diary content. Cap at 10 entries → save ~75 tokens.
- **Trim 5 (touchy):** MINECRAFT_PRIMER is 430 tokens. Some content (e.g., the full mob behavior list) is rarely actionable. Evaluate trimming primer to ~250 tokens. **Defer** — high risk of degrading "knows what zombies do at night" behavior. Surface as a discussion item per D-6.

## Memory write-side fix design (D-4 / Theme 5)

This is the single most important behavioral change in the phase. The bug: `sessionState.js:36-39, 285-333` gates diary writes on `MUTATING_ACTIONS = {dig, placeBlock, attackEntity, dropItem, ...}`. Any pure-chat session — *exactly the sessions where social moments happen* — leaves zero diary trace.

### Recommended design

Three changes, sequenced:

**Change 1 — Decouple diary triggers from world mutation.**

Replace the binary `MUTATING_ACTIONS` gate with a **salience-tagged write** model:

- The compactor's summary call (`compaction.js:summarizeLoopBatch`) already runs once per loop-batch cadence (currently gated to idle events). Lift the cadence gate from "loops with mutation" to **"loops with mutation OR loops with affect"**.
- "Affect" means: any loop containing owner chat that matches simple keywords (`good job|thanks|thank you|nice|love|appreciate|sorry|please|name is|call me|i am`), OR any loop where the owner asked a recall question, OR any loop where the loop's say() output went above some sentiment threshold (skip the LLM call for sentiment — keyword matching is enough for v1).
- Add a separate, smaller flag `loopHasAffect` walking the same loopMessages array as `loopHasMutation`.

**Change 2 — Add a `noteToSelf` tool.**

Add a new personality tool (alongside `say` and `setGoals`) named `noteToSelf` with this signature:

```js
{
  name: 'noteToSelf',
  description: 'Privately record a moment worth remembering across sessions: praise from owner, an inside joke, a stated preference, a name they revealed, a milestone reached. Will be written to your diary. Use sparingly — only for things you would want to remember weeks later.',
  input_schema: {
    type: 'object',
    properties: {
      kind: { type: 'string', enum: ['praise', 'preference', 'name', 'milestone', 'moment'] },
      summary: { type: 'string' },  // 1-2 sentences, the bot's words
    },
    required: ['kind', 'summary'],
  },
}
```

In the orchestrator, `noteToSelf` calls go straight to a small `affectLog` write (separate file `memory/AFFECT.md`, append-only) AND mark the current loop as `hasAffect=true` so the loop-batch summary fires. This bypasses the mutation gate cleanly.

`memory/AFFECT.md` is loaded **in full** into every seed turn (it's tiny by construction — a session yields 0–2 lines). This is option 3 from memory-findings — "dedicated affect log" — and it's the cheapest of the five options. No embeddings, no scoring, no truncation logic.

**Change 3 — Update OWNER.md notes from `noteToSelf({kind:"name"|"preference"})`.**

When `noteToSelf({kind: 'name', summary: 'Their name is Shawn'})` fires, also update `OWNER.md` `preferred_name` field. When `kind: 'preference'`, append to OWNER.md `## Notes` (with timestamp). This fixes D-M-4.

### File edits required

- `src/llm/sessionState.js` — add `loopHasAffect()` walker, OR-gate it with `loopHasMutation()`
- `src/llm/orchestrator.js` — register `noteToSelf` in personalityTools, route its execution to a new `affectLog.append()` call, mark the current loop affect-positive
- `src/memory/affectLog.js` (NEW) — small append-only log with full-load read
- `src/llm/orchestrator.js` `composeSeedBlocks` — add `affect_log` block before/after `seed_diary`
- `src/memory/owner.js` — add `setPreferredName(path, name)` and `appendNote(path, note)` helpers
- `src/llm/persona.js` or system prompt — instruct the bot when to call `noteToSelf` (keep this to ~3 sentences)

Estimated scope: ~150 lines of new code, ~50 lines of edits across 4 files. Cache breakpoint impact: tool list changes (system tool descriptions), so this is a one-time cache bust. Group with the other system-prompt-edit work.

### What we're NOT doing (deferred)

- Embedding similarity over diary entries (option 4 from memory-findings) — too heavy for v1.
- Topic-keyword retrieval at seed time (option 2) — useful but the structural fix above does most of the work; reconsider if affect-log + un-gated diary aren't enough.
- LLM-directed re-summarization (option 5) — extra Anthropic cost, deferred.

## First-turn say() acknowledgement design (D-1 / Theme 1)

Two options, both viable:

### Option A — Soft prompt (current approach, strengthened)

Edit orchestrator.js line 24:

```
say() cadence: REQUIRED on the FIRST turn of any new owner-triggered loop.
The first say() must ACKNOWLEDGE THE REQUEST — name what you're undertaking
("alright getting sand", "going hunting", "dropping the oak"). A `text` block
about the task does NOT count. If you're starting an action, the say() comes
in the SAME tool batch as the action call (e.g., `say + follow + attackEntity`).
```

This is what 260505-twx tried; it's been proven insufficient (hunt-sand-findings line 12, 19, 35). Soft alone isn't enough.

### Option B — Hard enforcement at orchestrator level (recommended)

In `runIterations`, after the first iteration's response:

```js
// Pseudocode
if (loop.iterationCount === 1 && triggeredBy === 'sei:chat_received' && ownerChat) {
  const calledSay = toolUses.some(u => u.name === 'say')
  if (!calledSay) {
    // Don't append the assistant turn; instead inject a system-noted user turn:
    loop.appendUserTurn([{
      type: 'text',
      name: 'reminder',
      text: 'You did not call say() on the first turn. The owner needs verbal acknowledgement before you start work. Re-issue your tool calls but include a say() that names what you are doing.',
    }])
    continue  // re-call without consuming an iteration
  }
}
```

This costs at most one extra LLM round-trip per first-turn-violation. Once the model learns the pattern in a session it stops happening. Combine with the soft prompt edit so the model gets it right first try most of the time.

**Recommend:** Both. Soft prompt change AND hard orchestrator enforcement. The cost of one extra round-trip on violation is trivial compared to the social cost of the bug. Cite log lines: hunt-sand line 12 (hunt), 19 (oak), 35 (sand), wood-findings line 113 (owner "hello?" ghosted).

## Punctuation post-processor design (D-7 / Theme 2)

Location: `src/llm/orchestrator.js:643-648` in the say() handler (or the `bot.chat(line)` call):

```js
// Strip punctuation from say() output. Internal `text` is exempt.
function postProcessSay(s) {
  return String(s ?? '')
    .toLowerCase()
    .replace(/[.,!?;:—–"`]/g, '')   // strip terminals + dashes + quotes
    .replace(/\s+/g, ' ')             // collapse multi-space
    .trim()
    .slice(0, 256)
}

// In the say() branch:
if (u.name === 'say') {
  const line = postProcessSay(u.input?.text)
  logChatOut(line)
  try { bot.chat(line) } catch {}
  convoMemory.recentChat.pushSelf(config.persona?.name ?? 'sei', line)
  results[i] = { type: 'tool_result', tool_use_id: u.id, content: 'said', is_error: false }
}
```

**Important:** apostrophes (`'`) are kept (contractions: "don't", "what's", "i'm"). The CONTEXT.md spec calls for this explicitly.

**Side effect:** the convoMemory self-buffer stores the *post-processed* line, not the raw LLM output. This is correct — the bot's "memory" of what it said should match what the player saw.

**Don't post-process in `full` mode `[think]` lines** — those are private reasoning relayed for debug visibility. Punctuation strip would change debug output in a confusing way. Keep the strip on the say() path only.

Estimated scope: 8 lines of code in one file.

## Pathfinder fallback strategy (D-E-1 / Theme 4)

No fallback exists today. The bot calls goTo, gets `cant_reach`, and the LLM picks a fresh coordinate next turn — no escalation.

### Cheap, log-driven fallbacks

1. **goTo result includes closest-distance** — when pathfinder gives up, return `cant_reach (closest=8.4m, target was 12m away)`. The LLM can decide whether to retry, give up, or ask for help. **Lives in adapter** (`src/behaviors/pathfind.js`).
2. **Default range for player targets** — when the goTo call destination matches a known player position, set `range: 2`. Fixes D-H-9. **Lives in adapter** (registry's goTo wrapper, or pathfind.js).
3. **Repeat-suppression hint in snapshot** — if the same goTo+args was just attempted and failed in this loop's prior tool_result, add a `last_action_result: cant_reach × N` count (already partly there as `last_action_result`, but no streak count). **Lives in adapter** (snapshot composer) but the system-prompt rule about "ask for help after 2 cant_reach" lives in **brain**.
4. **Explicit "ask for help" prompt rule** — orchestrator.js line 32 already exists ("If you have tried the same approach 2+ times..."). Strengthen with cant_reach being a named trigger. Lives in **brain**.

**Don't add bot-side auto-bridge/dig** — too risky (griefing concern, plus complex cost-benefit for the LLM to pre-judge). The verbal-help-ask path matches the bot's existing personality contract.

## `#N` index instability fix (Themes 3)

The handle table (`src/observers/targeting.js`) already TTLs handles at 10s and replaces the table on every `setHandles` call (per-snapshot replace). The **bug** is that the LLM doesn't know `#1` on turn N is not the same block as `#1` on turn N+1.

### Recommended approach

**(a) Tell the model.** Add to system prompt or to `dig`'s tool description: `"#N references rotate every snapshot — only valid in the SAME turn the snapshot listed them. For repeated digs of the same block type, use {block: "<name>"} which auto-finds nearest each call."`

**(b) Cap parallel digs to 1 in orchestrator.** When `runIterations` sees more than one `dig` tool_use in a single response, execute the first, synthesize `aborted: only one dig per turn allowed; re-issue next turn` for the rest. Saves token blob bloat (theme 3 trim 1).

**(c) Reject coordinate-mismatched `#N`.** When the LLM emits `dig({target: "#5"})` and `#5` doesn't exist in the *current* snapshot's handle table, the result is already "stale target". Ensure the prompt explicitly says "stale target" means the index has rotated and to switch to `{block: "..."}`.

Don't try to make `#N` indices stable across snapshots — too complex (would require persistent block-id tracking and re-numbering on every disappearance), and the per-snapshot TTL already encodes the right semantics. The fix is **prompt clarity + orchestrator cap**.

## Risks and pitfalls

### What the refactor could break

- **Cache breakpoint preservation.** The cached system blocks are byte-stable across (system instructions + persona + capability + primer + tools). Moving the primer from `persona.js` to an adapter-supplied function MUST preserve the exact bytes for minecraft to keep the cache. Verify with a byte-equality assertion in a test.
- **`assertNoMemoryInSystemBlocks` invariant** (orchestrator.js:325-335) — refactor must keep this defensive scan working.
- **Loop._internal.messages shape** is consumed by sessionState's loopHasMutation, convoMemory.synthesizeTitle/Mutations, compaction.serializeMessagesForPrompt. All three are brain-side; refactor doesn't touch them, but the **`name` field on text blocks** (snapshot/event/seed_owner/etc.) is part of the message schema — see `loop.js` D-43 trim algorithm. Don't accidentally break the rebuild-on-call invariant.
- **Cache_control marker placement** — currently on the LAST tool entry. The refactor changes how tools come in (from adapter via `listActions/getActionDescription`). Make sure the LAST tool still gets the marker.
- **Pitfall 4 cache invariant** — OWNER/DIARY content must NEVER appear in cached system blocks. `assertNoMemoryInSystemBlocks` enforces this; refactor must preserve.
- **Import-cycle risk.** brain → adapter for actions, adapter → brain for the memory layer? No — the adapter receives memory side-effects via the AdapterHandlers interface (push direction). The brain never imports adapter; the boot script (`src/index.js` or `src/brain/index.js`) imports both and wires them.

### Migration concerns

- **No data file migration needed.** `OWNER.md`, `DIARY.md`, `config.json` paths stay the same.
- **No config field renames required** but adding `noteToSelf` and an `affectLog` path means a tiny config addition: `memory.affect_md_path` default `./memory/AFFECT.md`.
- **No native-module concerns.** Pure JS refactor. `@electron/rebuild` only matters when better-sqlite3 lands in v2.

### Testing/review concerns

- The CLAUDE.md indicates Electron utilityProcess wraps mineflayer; refactor must keep mineflayer importable only from `src/adapter/minecraft/`, never from anything `src/brain/` reaches. A simple grep test in CI would catch leaks.
- Cache hits should stay at ~5300 tokens per turn after the refactor lands. Verify with one live test session post-refactor.
