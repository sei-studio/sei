# Phase 03.1 Plan 08: Memory and Diary Fixes (Gap Closure) Summary

Closed five memory-write-side gaps surfaced in the Phase 03.1 verification: noteToSelf-only first turns no longer pay an extra LLM iteration, the 70KB+ diary batch that never flushed because the user never went idle now flushes at the byte cap on any loop_terminal, legacy pre-Plan-04 purple diary entries are scheduled for one-shot recompaction at boot, the consolidation lock no longer leaks when consolidateOlderHalf throws synchronously, and concurrent OWNER.md / AFFECT.md writes serialize through a per-file in-process mutex so updates do not get lost.

## Coverage

This plan addresses 5 of 17 Bucket A items from the Phase 03.1 verification report:

- **D-NEW-MEM-2** (#3) — noteToSelf-only first turn paying 2 LLM iterations
- **D-NEW-MEM-3** (#4) — 70KB pending batch never flushed (cumulative_bytes=70349 with bytesCap=32768 in continuous-chat session)
- **D-W-9** (#6) — legacy purple diary entries (+2183 chars) bias every subsequent loop
- **WR-05** (#14) — consolidationLock leaks on synchronous throw, blocks all future consolidation passes
- **WR-06** (#15) — `setPreferredName` mid-loop racing `saveOwner(last_seen)` from `onPlayerLeft`

## Tasks Executed

### Task 1: `withFileLock` + OWNER.md / AFFECT.md serialization (WR-06)

**Commit:** `97219df` — `feat(03.1-08): per-file mutex serializes OWNER.md/AFFECT.md writes (WR-06)`

Three deliverables in one atomic change:

1. **`src/brain/storage/fileLock.js`** — exports `withFileLock(filePath, fn)`. Per-file in-process async mutex backed by a `Map<string, Promise<unknown>>`. Each call chains after the previous tail with a `.catch(() => {})` so a rejecting callback does not block the next caller; the tail is then updated to a settled-state mirror so the next chained caller does not need to defensively `.catch()` it themselves. Cross-process correctness is explicitly NOT a goal — single-process Sei runs in utilityProcess.

2. **`src/brain/memory/affectLog.js`** — `appendAffect`'s read+atomicWrite block runs inside `withFileLock(filePath, async () => { ... })`. Module header comment updated to reflect the new concurrency story (replaces the old "no lock; we may lose at most one append" caveat).

3. **`src/brain/memory/owner.js`** — extracted private `_writeOwnerSerialized(path, data)` helper that does the serialize+atomic-write WITHOUT taking the lock. Public `saveOwner` is now a thin `withFileLock(path, () => _writeOwnerSerialized(path, data))` wrapper. `setPreferredName` and `appendNote` acquire the lock around the entire load→mutate→write sequence and call `_writeOwnerSerialized` directly — avoiding same-path lock re-entry that would self-deadlock.

**`scripts/test-fileLock.mjs`** covers:
- T1: 20 concurrent `appendAffect` calls — all 20 lines persist (no losses).
- T2: 20 concurrent `appendNote` calls — all 20 notes persist.
- T3: FIFO ordering inside `withFileLock` — 5 concurrent callbacks sleep 5ms each, start/end markers stay paired without interleaving.
- T4: a rejecting callback does not block the next caller on the same path (lock release is unconditional).

**Verification:** `node scripts/test-fileLock.mjs` → `fileLock: all cases passed`. `node scripts/test-affectLog.mjs` (regression) → `affectLog + owner: all cases passed`.

### Task 3: Startup recompact hook for legacy purple entries (D-W-9)

**Commit:** `9fce3ce` — `feat(03.1-08): startup recompact for legacy purple diary entries (D-W-9)`

Two code changes:

1. **`src/brain/memory/diary.js`** — added `countOversizeEntries(maxWords = 80)` that walks `readAll()` and counts entries whose body exceeds `maxWords` whitespace-separated words. Consolidated `## Earlier (...)` blocks (`e.isConsolidated === true`) are explicitly EXCLUDED so the recompact does not feed on its own output (would create a rewrite loop). The bot can come up before the Anthropic call returns; subsequent loops pick up the new `## Earlier (...)` block on the next seed read. Single-flight is inherited from `diary.replaceOlderHalf`'s internal `writeLock`. Three distinct log lines distinguish the paths: detection, completion, failure.

**`scripts/test-diaryFlushBytes.mjs`** extended with **T9**: a fixture diary with 1 long body (85 words), 1 short body (10 words), and 1 consolidated block (also 85 words). `countOversizeEntries(80)` returns `1` — proving the consolidated-block exclusion.

## Deviations from Plan

**Minor — none of these change behavior, only test structure or implementation shape:**

1. **[Implementation simplification] withFileLock implementation** — the plan's draft included three iterations of the implementation (one with a syntactic error, then a "cleanup" attempt). Settled on the cleanest form per the plan's final note: tail is `run.then(() => undefined, () => undefined)` so the next caller chains directly off the settled mirror; no Map cleanup heuristic (best-effort, leaks bounded by distinct path count which is 2-3 in this codebase).

2. **[Test fixture isolation] T7 and T8 use fresh sessionState** — the plan's pseudocode reused the same `ss` across T6/T7/T8, but cumulativeLoopBytes and batchHasAffect carry between cases (e.g., T6's last 3 ticks leave `cumulativeLoopBytes=12288, batchHasAffect=true`, and T7 then trips the cap on tick 6 with stale affect=true). Created a new sessionState per case so each starts clean. Behavior tested is identical; the isolation is mechanical.

3. **[Plan-level TDD] Plan 08 is `type: execute` not `type: tdd`** — but Task 2 carries `tdd="true"`. RED → GREEN cycle observed: RED commit (`9d9a8c5`) preceded GREEN (`b5e6b96`); RED test failed before predicate change, passed after. No REFACTOR needed.

## Behavioral Impact (qualitative)

After this plan, in the next live replay we expect:

1. **No more doubled iterations on memory-only turns** — when the model emits `noteToSelf` as its only call on iteration 1, the loop completes without a reprompt. Memory persistence acknowledgements pay 1 LLM call instead of 2.
2. **Diary actually flushes in continuous-chat sessions** — the 70KB pending batch observed in `memory-postfix.txt` (cumulative_bytes=70349 with bytesCap=32768) should not recur. Post-replay logs will show `[sei/session] diary flushed (trigger=bytes, idle=false)` mid-session instead of the batch growing unbounded.
3. **Legacy purple entries fade out at boot** — on first startup after this plan, `[sei/brain] D-W-9 startup recompact: N legacy oversize entries detected` will appear, followed by `D-W-9 startup recompact completed` once the consolidate Anthropic call returns. Subsequent `seed_diary` slices will read denser, declarative `## Earlier (...)` blocks instead of `+2183 chars` of memoir prose.
4. **No more phantom consolidation lockups** — if a future bug or refactor causes `consolidateOlderHalf` to throw synchronously (compactor null, method missing, schema validation), the lock releases at the catch block. Verifiable via `consolidation sync throw: <message>` in the warn log.
5. **No more racey OWNER.md / AFFECT.md** — concurrent `setPreferredName` mid-loop and `saveOwner(last_seen)` from `onPlayerLeft` cannot interleave. The two write paths take the lock; the second waits for the first to finish before reading its baseline.

These claims are forward-looking; verification requires the next live replay session under similar conditions to memory-postfix / wood-postfix.
