<domain>
## Phase Boundary

Two intertwined goals:

1. **Behavior polish** — make Sei feel less like a scripted bot and more like a long-time friend. Fixes are driven by detailed analysis of four observed in-game sessions (logs/), not by speculation.
2. **AI/game decoupling refactor** — split the codebase into a *general AI brain* (persona, prompts, loop, memory, say/think discipline) and a *game adapter* (mineflayer registry, snapshot composition, attack/follow behaviors), such that the action library could be swapped for another game's library with minimal change to the brain.

**Token-efficiency** is a secondary objective threaded through both — Haiku 4.5 is small and easily confused; reducing prompt size and making instructions unambiguous improves quality and cost simultaneously.

**Out of scope:**
- Phase 4 work (Electron GUI, packaging, .dmg/.exe distribution)
- SQLite migration (deferred to V2 per Phase 3 charter)
- Adding a second game adapter (refactor only sets up the seam; we don't ship a second adapter)

</domain>

<decisions>
## Implementation Decisions (LOCKED)

### Behavior — Loop & Say/Think discipline

- **D-1 First-turn `say()` is mandatory and must acknowledge the request.** Not just "anything"; the first turn must verbally acknowledge the action being undertaken (e.g., "alright getting sand", "going hunting"). Currently the bot sometimes just executes silently when a sheep/sand task starts.
- **D-2 Tone shift to "we" framing, peer-to-peer, no punctuation.**
  - Replace "you/me" framing with "we/us" wherever natural — Sei is not handed tasks, she/he is *along for the ride*.
  - Strip punctuation from `say()` output (commas, periods, em-dashes, ellipses). Lowercase preferred. Treat as a post-processor on the say() text — do not ask Haiku to "remember" the rule.
  - Internal `text` (think) is exempt from the no-punctuation rule.
- **D-3 Internal-thought leakage is acceptable for now.**
  - Observed: `text` blocks address the player ("with you", "what's the move?") instead of sounding like inner monologue.
  - Decision: *do not* try to rewrite the persona prompt to enforce inner-monologue style. The user only sees `say()`; forcing Haiku to roleplay private thoughts adds prompt bloat and risks confusion. **Skip this fix.**

### Behavior — Memory & diary recall

- **D-4 Diary recall feels shallow — investigate retrieval, not write side.**
  - User tested in `logs/memory.txt`: had previously told Sei "good job" (presumably stored in DIARY.md), but recall in a later session was generic.
  - Hypothesis: `seed_diary_budget_bytes` evicts older entries before they reach the prompt, OR the seed loader concatenates entries blindly without ranking by relevance/recency.
  - Goal: make Sei feel like a long-time friend who remembers small things from long ago. Plan must inspect the seed-loader's selection algorithm (FIFO? LIFO? topk?) and propose a scheme that keeps a small set of *durable* high-signal entries reachable across sessions, even when newer entries are abundant.
  - Constraint: budget stays small (Haiku context cost matters). Solution should be selection-quality, not budget-expansion.

### Refactor — AI/game decoupling

- **D-5 Establish a `brain/` vs `adapter/` split.**
  - `brain/` (game-agnostic): orchestrator loop, sessionState, convoMemory, compactor, persona prompt assembly, say/think discipline, diary, ownerStore.
  - `adapter/minecraft/` (game-specific): mineflayer bot wiring, registry, snapshot composer, behaviors (follow, combat, autoEat, posHealer, chat, dig, etc.), targeting, lanDiscovery.
  - The brain should consume the adapter via a small interface — actions to call, snapshot to read, events to subscribe to. No mineflayer types or mineflayer imports should leak into `brain/`.
  - Plan should list the exact interface (action descriptors, snapshot shape, event names) and audit existing files to identify what crosses the seam today.
- **D-6 Token efficiency is a first-class goal, not opportunistic.**
  - Audit Haiku's prompt token use across system prefix, snapshot, convoMemory, seed prompts, tool descriptions. Identify removable redundancy.
  - Anthropic prompt caching breakpoints must be respected — don't add tokens that will bust the cache for marginal value.
  - User explicitly offered prompting collaboration — surface specific prompt rewrites for review rather than guessing alone.

### Behavior — Punctuation post-processor

- **D-7 Punctuation strip happens in the `say()` action handler**, not in the prompt. Apply lowercase + strip `[.,!?;:—–"'`]` after the LLM emits text, before chat send. Preserve apostrophes-in-contractions (don't / what's) since stripping breaks readability — strip everything except `'`. Also collapse multi-spaces.

### Logs as the source of truth

- **D-8 Plan must cite specific log lines** for each behavioral fix it proposes. The four logs are first-class evidence; speculation without log support is rejected.

</decisions>

## Specific Ideas

### From user (5 numbered items)

1. **First-loop-turn say() acknowledgement** — bot must verbally acknowledge starting a task. Example failure: bot started killing a sheep / gathering sand without a chat acknowledgement. Fix lives in: orchestrator's per-loop cadence rule + seed prompt for `sei:chat_received`. The 260505-twx work added "mandatory first/last turn"; this item *strengthens* that to "must acknowledge the request, not just say anything."

2. **Tone — "we" framing, no punctuation.** Persona prompt rewrite + say() post-processor for punctuation strip. We are equals; Sei is not "handed tasks."

3. **Text/say still mixed up** — internal `text` reads like player-facing speech. *User's own decision:* skip if it doesn't improve actual play. **Treat as deferred** unless the planner finds a low-cost fix that also reduces tokens.

4. **Diary memory recall** — felt shallow when user expected recall of "good job" praise. Investigate seed loader and budget mechanism; aim for "long-time friend" feeling. Test case in `logs/memory.txt`.

5. **Game/AI decoupling refactor + token efficiency** — biggest item. Establish `brain/` vs `adapter/minecraft/` seam. Audit Haiku prompt tokens. Surface prompt-rewrite proposals for user review.

### From orchestrator's gray-area analysis (delivered to user pre-planning)

- Items 1, 2, 3 are all prompt-tuning tweaks → bundle into ONE prompt-edit sub-task to avoid repeated cache busts.
- Item 5 (refactor) and item 2 (persona prompt) are sequenced: persona prompt lives in the brain, so refactor work touches the same files. Decide order: (a) refactor first, then re-tune persona prompt at the new location, OR (b) re-tune persona prompt first while location is familiar, then refactor. Recommend **(a) refactor first** — moving and rewriting in one pass is cleaner than rewriting then moving.
- Item 4 (diary) may also cross the seam — diary code goes in `brain/`, but seed-loader budget tuning is content-level. Recommend handling diary inside the refactor wave (same file moves, easier review).
