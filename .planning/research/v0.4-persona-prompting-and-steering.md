## 0. The Sei-specific problem, restated against the evidence

Sei generates `persona.expanded` from a short user blurb at save time via a fixed 6-section template
(`# IDENTITY / # VOICE / # DEFAULT DYNAMIC / # PROACTIVENESS / # REACTIONS / # MEMORY`), then renders it
into the **cached system prefix** (`renderPersona`, `src/bot/brain/orchestrator.js:826`, behind the
`cache_control` boundary on the tool list). The companion speaks only via `say()`. Two failure modes the
user reports:

1. **Weak/short user descriptions** → thin persona.
2. **Drift toward generic "assistant voice"** over a session.

The research maps onto these precisely, and exposes one structural gap in the current wiring:

> **The persona is front-loaded in the cached prefix and never re-injected near the recency end of the
> context.** Every drift paper below says this is exactly the position from which a persona decays fastest,
> and that low-depth re-injection is the single best retraining-free fix. This is the #1 actionable finding.

## 1. Persona drift / consistency — what measurably decays, and what fixes it

### 1.1 Drift is real, fast, mechanistic, and worse on small models

- **Persona/instruction drift — Li et al., "Measuring and Controlling Instruction (In)Stability in
  Language Model Dialogs," arXiv 2402.10962 (Harvard; v1 Feb 2024, v4 Jul 2024). [P]**
  - Self-chat benchmark, 200 conversations, LLaMA2-chat-70B + GPT-3.5. **Significant, measurable persona
    drift within ~8 conversation rounds**, even with context fully intact. The agent not only loses its
    own persona, it **progressively adopts the *user's* persona** (flagged as a safety/adversarial vector).
  - **Mechanism = attention decay, NOT context overflow.** Attention to system-prompt tokens drops sharply
    *between* turns while staying stable *within* a turn. Geometric argument: each user turn introduces
    out-of-distribution tokens that expand the embedding cone, diluting attention to the original
    instructions ~exponentially. Drift therefore appears *before* the window fills. Code:
    github.com/likenneth/persona_drift.
- **"Lost in the middle" — Liu et al., arXiv 2307.03172, TACL 2023 (Stanford). [P]** The *positional/overflow*
  mechanism, distinct from attention-decay-over-turns. U-shaped accuracy: best at the **start (primacy) or
  end (recency)**, worst in the middle (20-doc QA: ~75% at pos 1, ~72% at pos 20, ~55% mid — a 20+ pt swing
  from position alone). Worst-case middle performance dropped *below* the closed-book baseline. Persists in
  long-context models.
- **"LLMs Get Lost In Multi-Turn Conversation" — Laban, Hayashi, Zhou, Neville, arXiv 2505.06120
  (Microsoft Research + Salesforce, May 2025). [P]** **39% average performance drop multi-turn vs
  single-turn** across 6 generation tasks, 200k+ simulated convos. Decomposition: small loss of "aptitude,"
  large rise in **unreliability**. Failure shape: models commit early to a wrong assumption and **"when LLMs
  take a wrong turn, they get lost and do not recover."** Restating the full requirement in one consolidated
  turn recovers much of the gap. (Directly relevant to Sei's per-turn snapshot: consolidating the persona +
  state into a single recent block is the recovery lever.)
- **PERSIST — "Persistent Instability in LLM's Personality Measurements," arXiv 2508.04826 (AAAI 2026). [P]**
  25 models (1B–685B), 2M+ responses. Scaling barely helps: **even 400B+ models show std-dev > 0.3 on a
  5-point personality scale.** **Conversation history *amplifies* response variability and exacerbates it for
  small models;** CoT/reasoning can *increase* variability; question reordering alone shifts results. Small,
  history-laden = worst case = exactly Haiku-class in a long session.
- **PersonaGym / PersonaScore — arXiv 2407.18416 (Princeton; EMNLP Findings 2025). [P]** First *dynamic*
  persona-agent eval. **Scale does NOT reliably improve persona fidelity:** Claude 3.5 Sonnet 4.51 vs GPT-3.5
  4.38 (+2.97% only); GPT-4.1 tied LLaMA-3-8B; **Claude 3 Haiku 3.64** (well below Sonnet). Consistency is
  driven by conditioning, not scale. The hardest sub-dimension is **Linguistic Habits** (jargon/syntax/tone/
  speech style) — **all models score < 4.0/5.0** — i.e. *register/voice is precisely where models fail and
  does not come for free from a trait list.*

**Implication for Sei:** drift hits ~8 turns via attention decay; Haiku drifts most; register is the weakest
axis. Engineering (placement, re-injection, exemplars) matters more than model choice.

### 1.2 Fixes that work WITHOUT retraining (ranked by evidence)

- **Low-depth re-injection of the persona near the recency end — the top fix.** 2402.10962's own
  training-free mitigations:
  - **System Prompt Repetition (SPR):** re-inject the system/persona before each user turn (prob p).
    **Best in *long* conversations.** [P]
  - **Split-softmax:** power-law reweighting of attention to counter decay; **best stability/quality
    trade-off early.** [P]
  - **Classifier-free guidance (CFG):** contrastive logits with/without the system prompt (see §4.3).
  - The product-world equivalent is **SillyTavern's Author's Note / Character "Depth Prompt"**: a text block
    injected at a **static depth** near the end of context (Depth 0 = after the last message = max impact),
    re-stated every turn regardless of conversation length, so it does **not** scroll up and decay like the
    front-loaded description. **[P]** docs.sillytavern.app/usage/core-concepts/authors-note. This is literally
    SPR + recency placement productized. **Sei does not currently do this** — its persona is only front-loaded.
- **Placement rule of thumb:** identity at the top (primacy) AND a compressed identity/voice anchor at the
  bottom (recency); avoid the middle. **[P/S]** (2307.03172 + practitioner consensus).
- **Positive demonstrations beat negative instructions.** Negation is an inverse-scaling failure: the NeQA
  benchmark shows negation understanding does **not** reliably improve with scale and InstructGPT gets
  *worse* as it scales; autoregressive decoding *selects toward* tokens, so "don't say X" only slightly
  lowers X while a positive exemplar actively raises the target. **[P/S]** NeQA (inverse-scaling);
  gadlet.com/posts/negative-prompting. → Demonstrate the voice; don't enumerate bans (Sei currently leans
  heavily on bans in `BASELINE_INSTRUCTIONS`).
- **Example dialogue is the most-cited voice/anti-drift lever**, but it is **the first thing dropped under
  context pressure** in the SillyTavern/V2 stack (`mes_example` only inserts "if there is free space … pushed
  out block by block"). **[P]** So encode voice you need permanently as a *standing* field or a depth-injected
  anchor, not only as transient examples. RoleLLM (2310.00746) notes zero/few-shot prompting "suffer from role
  drift and consistency loss in long dialogues," motivating example-grounded role prompting.
- **Redundancy makes a trait load-bearing:** echo the 2–3 defining traits in the persona body, in an example
  `say()`, and in a behavioral rule. **[S]** roborhythms.
- **Retraining fixes (for completeness, see §4.4):** PCL / Persona-Aware Contrastive Learning (2503.17662)
  lifted a 7B to GPT-4-comparable role-play consistency; Character-LLM (2310.10158) shows dedicated weights
  beat prompts for consistency but are heavyweight per character.

## 2. Character.ai & competitor persona-card design (which fields aid consistency + generalize)

### 2.1 The schema everyone converged on (SillyTavern / TavernAI Card V2/V3)

V2 spec (`malfoyslastname/character-card-spec-v2`, fields under `data`, `spec:"chara_card_v2"`): **[P]**

| Field | Behavior | Generalizes? |
|---|---|---|
| `name` | identifier | n/a |
| `description` | **always injected**; main body (appearance, personality, world). Permanent token. | **Yes** (identity/voice) |
| `personality` | short trait summary; permanent token | **Yes** |
| `scenario` | opening circumstances; permanent token | **No — scenario-locked** |
| `first_mes` | opening message; **sent once**; highest-leverage *style* setter ("more likely to pick up style/length from the first message than anything else") | one-shot |
| `mes_example` | `{{char}}:`/`{{user}}:` example dialogue; **first dropped under context pressure** | **Yes (voice)** but fragile |
| `system_prompt` | replaces the user's system prompt; top of context | yes |
| `post_history_instructions` | the **post-history / "jailbreak" slot — injected AFTER chat history**, so it gets recency weight | **Yes — this is the anti-drift slot** |
| `alternate_greetings` | swipeable openers; only one ever enters context | one-shot |
| `character_book` (lorebook) | keyword-triggered conditional injection | conditional |

V3 (`kwaroran/character-card-spec-v3`) adds a **per-entry decorator scheduler** on lorebook entries —
`@@depth <n>` (recency control), `@@role <system|user|assistant>`, `@@position <before_desc|after_desc|
personality|scenario>`, `@@activate_only_after <n>` (delayed/paced reveal). This turns the lorebook from
"keyword → dump text" into a **programmable injection scheduler with depth + role + activation conditions** —
i.e. the exact controls that govern cross-context consistency. **[P]**

**Character "Depth Prompt" / Character's Note** (V2 `extensions.depth_prompt`): per-character text held at a
**static message depth** to "reinforce certain character traits." This is the per-character analog of the
Author's Note and the cleanest realization of SPR for a character card. **[P]**

### 2.2 Formatting languages

- **W++** (`Personality("kind"+"shy")`): community-deprecated token bloat; modern instruct models read plain
  prose at least as well. **[S]**
- **PList / "Boostyle"** (`[Name's persona: trait; trait; …]`): compact; **traits later in the list weighted
  as more important**; risk of bracket leakage; consolidate to one PList per insertion point. **[S]**
- **Ali:Chat:** define the character *through* example Q&A/RP in the exact output format. Token-efficient
  ("a lot of personality is implied through dialogue & actions"); guide recommends **2 long or 3 short
  examples, ~300–600 tokens.** **[S]** rentry.co/alichat, rentry.co/kingbri-chara-guide.
- **Modern recommended combo = PList (hard facts, in description) + Ali:Chat (voice, in examples)**; one card
  documented dropping **~1300→~599 tokens** by switching, and moving the PList into the **author's note** so
  `description` holds only the voiced exemplars. **[S]**
- **Format rigidity barely matters for *quality*:** structured JSON vs prose differs ≤0.03 on PersonaGym
  (2601.04716, §3.4); invented characters aren't inherently weaker than canonical (|Δ|≤0.07). Store fields as
  readable prose; don't over-engineer schema. **[P, recent-preprint?]**

### 2.3 Competitor field schemas (what each product exposes)

- **Character.AI:** **Definition** field (0–32k chars: how it thinks/speaks/behaves) with inline `{{char}}:`/
  `{{user}}:` **example chats** (described as the single most effective consistency tool); **Greeting**;
  **User Persona** (user-side, **persistent across chats**, ~150 words free / 2,250 chars c.ai+); **Pinned
  Memories** (up to ~15 messages held "forever" — an explicit admission that persona influence decays as the
  window fills). **[P for the 32k/Definition; S for the caps & 15-pin figure.]**
- **Kindroid (cleanest facts-vs-behavior split): [P, vendor docs]** **Backstory** (≤2,500 chars, "3rd person,
  proper nouns, concise, avoid narrative"), **Key Memories** (≤1,000 chars, "clear, atomic facts … modular and
  updateable"), **Response Directives** (≤150 chars, hard-guides formatting/tone/length, "positive guidance
  rather than prohibitions, avoid conflicting instructions" — a built-in author's-note/style slot), **Voice**,
  **Persona**. Notably, **third-person journal writing measurably improves retrieval accuracy.**
- **Janitor AI:** V2-lineage card (description/personality/scenario/example dialogue/lore); makes "**permanent
  tokens**" explicit and recommends **< 2,500 permanent tokens** or "quicker memory degradation." **[S]**
- **Chai:** Basic Info / First Message / **Memory Prompt** ("the brain") + Example Conversations; advice to
  repeat critical traits and re-remind mid-chat (manual anti-drift). **[S]**
- **Replika:** user-editable **Memory bank**, bot-authored **Diary**, **relationship modes**
  (friend/partner/mentor), adjustable **personality traits** + up/down feedback. **[S]**
- **Nomi:** **Backstory** + **Shared Notes** ("information your Nomi should always remember") feeding an
  **Identity Core** layer over short/mid/long-term memory — explicitly marketed to balance consistency with
  growth. **[S]**

**Cross-product convergence (the durable pattern):** everyone splits the persona into **(a)** a large
prose/trait body, **(b)** example dialogue for *voice*, **(c)** a small always-on atomic-facts store
(Key Memories / Shared Notes / Pinned Memories / Memory Bank), and **(d)** a short imperative style/behavior
slot re-injected near the end (Response Directives / `post_history_instructions` / Author's Note). **Sei has
(a) and partially (b); it lacks (c) as a structured store and lacks (d) entirely.**

**Which fields generalize across contexts (keep, weight high, depth-inject):** personality/trait summary,
speech-pattern-via-examples, atomic updateable facts, short imperative directives. **Scenario-locked (do not
rely on for cross-context identity):** scenario, greetings/first_mes, keyword-triggered lore. This is the
direct evidence for Sei's "holds across MC early/late + casual texting + minigames *without* hardcoded
scenario scripts."

## 3. Prompt EXPANSION — short weak blurb → rich, generalizable persona (the core ask)

### 3.1 Personas are portable knowledge carriers; scenarios are not

- **PersonaHub — "Scaling Synthetic Data Creation with 1,000,000,000 Personas," arXiv 2406.20094
  (Tencent AI Lab; Jun 2024, v3 May 2025). [P]** Central claim: a persona is a *distributed carrier of world
  knowledge* — **"personas can be combined with almost any data synthesis prompt"** (portable), whereas
  scenario key-points are context-specific and non-portable. This is the strongest direct argument that
  **trait/identity/register descriptions generalize and scripted scene responses do not** — exactly Sei's
  requirement.
  - **Text-to-Persona:** infer a persona from any text via "a specific persona who is likely to
    [read|write|like|dislike] the text." **Granularity scales with input richness** — a one-liner yields only
    a *coarse* persona ("a computer scientist"); detail yields fine-grained ("a machine-learning researcher
    focused on neural-net architectures and attention mechanisms"). **→ A weak blurb intrinsically licenses
    only a thin persona unless richness is synthesized.** This is the root cause of Sei's problem #1.
  - **Persona-to-Persona:** spin out related personas via relationships (≤6 iterations) — a technique for
    *enriching* a thin seed by deriving its relational web.

### 3.2 Axes that generalize, and the structured-vs-free-text question

- **Decouple style from knowledge.** RoleLLM (2310.00746, ACL Findings 2024) treats **speaking-style imitation
  as a *separately extracted* component** (RoleGPT, with explicit **Lexical Consistency** = catchphrases/idioms
  and **Dialogic Fidelity** = similarity to example dialogue) from role *knowledge* (Context-Instruct). **[P]**
  Enumerate catchphrases as a first-class field; supply few-shot exemplars; style transfers via examples far
  better than via abstract description.
- **OCEAN/Big Five is the generalizable trait backbone.** PersonaLLM (2305.02547, NAACL Findings 2024): assigned
  Big-Five profiles produce consistent self-reported BFI scores with large effect sizes. Big5-Chat
  (2410.16491, ACL 2025): grounding traits in *how humans linguistically express personality* (SFT/DPO) beats
  prompting on realism → **ground trait descriptions in concrete behavioral exemplars, not adjectives.** [P]
- **Enumerated graded levels give control but cross-trait coherence is imperfect:** Openness/Extraversion/
  Neuroticism scale ~linearly with prompt intensity; Agreeableness/Conscientiousness saturate; pushing several
  traits to extremes simultaneously degrades (TRAIT 2406.14703; NMI s42256-025-01115-6). **[P/S]**
- **Structure helps authoring/parsing, not output fidelity.** "Disentangling Character Profile Axes,"
  arXiv 2601.04716 (Chung-Ang U.) defines a unified **5-dimension / 28-field** schema (Personal Attributes,
  Personality Traits, Interpersonal Relationships, Motivations, Abilities) and finds **profile Structure has
  ~no impact** (≤0.03 on PersonaGym) while **Disposition/valence matters** (immoral characters degrade). **[P,
  recent-preprint?]** Use the 5-dimension list as a *content checklist*; don't expect rigid JSON to improve
  in-character fidelity.

### 3.3 Enumerated voice beats free-text; auto-expansion injects stereotyped defaults

- **"Voicing Personas," arXiv 2505.17093 (May 2025) — confirmed real, directly validates the design. [P]**
  - **Closed-ended** (LLM maps persona → structured attributes) "offers strong consistency and controllability"
    and **outperformed open-ended on all metrics**; **open-ended free-text "sacrifices structural consistency
    for artistic flexibility."**
  - **Auto-expansion fills unspecified slots with stereotyped demographic defaults:** gender male **64%** vs
    female 33% from neutral input; **~90% of outputs default to North-American/British accent**; tone↔gender
    and pitch↔gender biases. **"Reasoning about persona descriptions without explicit assignment is sufficient
    to introduce bias."** → If Sei auto-expands a thin blurb, the expander *will* silently default
    voice/identity slots to stereotypes unless they're explicit, deliberately chosen (or constrained-random)
    enumerated fields.
- **The bland "assistant voice" is a literal attractor in activation space.** "The Assistant Axis," arXiv
  2601.10387 (Anthropic safety-research, Jan 2026): the *leading component* of a model's persona space is an
  "Assistant Axis"; weak personas snap back to it. **[P, recent]** Default-voice tells to forbid: repetitive
  openers ("let's dive in"), hedges ("may/often/typically"), evenly-sized bullet lists, synthetic enthusiasm
  ("game changer"), symmetric sentence lengths. **[S]** A persona prompt is fighting gravity — explicit
  anti-default register instructions plus committed exemplars are needed.

### 3.4 Meta-prompting / expansion-prompt craft (generate-vs-preserve)

- **Anthropic Prompt Generator / Prompt Improver** (Console; Nov 2024): the canonical "expand a seed into a
  structured, XML-tagged, example-rich prompt while preserving intent" tooling; the published metaprompt is a
  long multi-shot prompt baking in decomposition, few-shot, role-play, variable handling. **[P]**
  docs.anthropic.com/.../prompt-generator; anthropic.com/news/prompt-improver.
- **Fact preservation is the key failure mode:** meta-prompting guidance says **pin the seed facts, ban
  external speculation, demand explicit "unknown" when evidence is missing** — otherwise the expander
  confabulates and "drowns the seed" with a polished narrative. **[S]** promptingguide.ai; Meta-Prompting
  2311.11482. → Mark every user-supplied fact immutable; instruct the model to *label invented specifics as
  invented vs seed-derived*.
- **Per-field regeneration is the dominant UX** (Chub.ai, Agnai, SillyTavern char generators): expand/regenerate
  *per section*, not monolithically. Sei already streams a 6-section expansion; per-section regen is the natural
  extension. **[S]**

### 3.5 A concrete expansion-prompt design (synthesized — MEDIUM, assembled from the above)

Converging recipe for blurb→persona:

1. **Separate GENERATE from PRESERVE.** "These seed facts are fixed — never contradict, omit, or override
   them; expand *around* them. Where you invent specifics, they must be consistent with the seed and you must
   not state them as if the user provided them." (Fact-pinning; Prompt Improver intent-preservation.)
2. **Decouple the axes** (PersonaHub portability + RoleLLM style/knowledge + Card V2 separation):
   *identity/knowledge* (portable: background, expertise, motivations, relationships, abilities — the 2601.04716
   checklist), *voice/register* (enumerated, first-class — the PersonaGym weak spot), and *scenario* kept
   **separate and minimal** so the persona stays generalizable.
3. **Enumerate the consistency-critical fields, free-text the expressive ones** (Voicing Personas): closed
   enumerated values for register/diction/formality/pace/OCEAN-levels/catchphrases → consistency; free-text for
   backstory/flavor → richness. Never leave voice/demographic slots unspecified (stereotype default).
4. **Demand concrete voiced exemplars, not adjectives** (RoleLLM dialogic fidelity; `first_mes` lore; Big5-Chat):
   N example utterances *in the exact output format* (for Sei: short `say()`-style lines) that demonstrate
   **register, not situations**.
5. **Match output richness to input richness and flag thin inputs** (PersonaHub granularity): infer plausible
   specifics for a thin blurb but keep the seed authoritative; optionally ask 1–2 clarifying questions at
   authoring time to thicken a one-liner before expanding.
6. **Actively counter the assistant default** (Assistant Axis): explicit forbidden-register list + a committed,
   distinctive voice requirement.
7. **Structured template for authoring/parsing, not as a quality guarantee** (2601.04716) — which is what Sei
   already does.

## 4. Hybrid / non-prompting steering (the big expansion) — and its feasibility for Sei

**Framing:** Sei is multi-provider. Anthropic (incl. the cloud-proxy) and the OpenAI-compat providers are
**text-in/text-out** — no logprobs (Anthropic) or read-only top-k logprobs (OpenAI), **no activation access,
no write path.** Only the **local Ollama/llama.cpp** backend exposes the residual-stream read/write the strong
methods need. That split is the spine of the verdict.

### 4.1 Activation steering / representation engineering (needs residual-stream read+write → local only)

All compute a **direction vector** from contrast pairs and **add it to hidden activations** at inference.

- **Representation Engineering (RepE) — Zou et al., arXiv 2310.01405 (Oct 2023). [P]** Top-down: extract linear
  "reading vectors" for concepts (honesty, emotion, power-seeking) from contrast pairs, then monitor/steer.
- **ActAdd — Turner et al., arXiv 2308.10248 (Aug 2023). [P]** Simplest: diff activations of one contrast pair
  ("Love" − "Hate"), add as a virtual bias. No training/optimization.
- **Contrastive Activation Addition (CAA) — Rimsky et al., arXiv 2312.06681 (ACL 2024). [P]** Average residual
  activation *differences* over many behavioral pairs (sycophancy, hallucination, etc.); add with a tunable ±
  coefficient. Reported to **outperform finetuning and few-shot** with minimal capability loss. The canonical
  trait-steering method.
- **Inference-Time Intervention (ITI) — Li et al., arXiv 2306.03341 (NeurIPS 2023). [P]** Probe individual
  attention heads for a direction (truthfulness), shift only high-probe heads. Alpaca TruthfulQA 32.5%→65.1%.
- **Reliability caveat — arXiv 2407.12404 (NeurIPS 2024) + "Steering off Course" (2025). [P]** A consistent
  **trait-amplification ↔ coherence-degradation trade-off**; steering is prompt-dependent, brittle OOD, and
  small coefficient changes can cause "catastrophic generation failures." → Any local steering feature needs
  **per-character coefficient tuning + a coherence guardrail.**

### 4.2 The directly-relevant 2025 persona-vector line (Anthropic / OpenAI)

- **"Persona Vectors: Monitoring and Controlling Character Traits in Language Models" — Chen, Arditi, Sleight,
  Evans, Lindsey; arXiv 2507.21509 (Jul 29 2025); blog anthropic.com/research/persona-vectors (Aug 1 2025);
  code github.com/safety-research/persona_vectors. [P]** The most on-target technique for Sei's goal.
  - **Automated pipeline:** from only a *natural-language trait description* it auto-generates contrast prompts
    + eval questions + an LLM judge, then extracts the vector as the **mean activation difference** between
    trait-present and trait-absent responses. (This is the same input Sei already has — a description.)
  - Specifics: best single-layer extraction ~layer 20 (Qwen-2.5-7B) / layer 16 (Llama-3.1-8B); steer by adding
    **α·v** to activations; preventative steering uses **α≈1** while keeping coherence >80/100 and MMLU intact.
  - Three uses: **monitor** drift mid-conversation, **preventative steering** during finetuning ("vaccination"),
    **data flagging** to predict trait-inducing training samples.
  - **Models = open-weight only (Qwen/Llama). Requires full weights + activations** — *not* exposed on
    Anthropic's product API. Maps cleanly onto the local llama.cpp control-vector path (§4.5), not hosted Claude.
- **OpenAI — "Persona Features Control Emergent Misalignment," arXiv 2506.19823 (Jun 2025). [P]** Sparse-
  autoencoder "model diffing" finds interpretable persona directions (incl. a "toxic persona" feature) that
  gate misalignment. Evidence that **persona is a low-dimensional steerable direction** — but needs weights +
  SAE on activations.

### 4.3 Control codes & decoding-time conditioning (needs logits → local only)

- **CTRL — Keskar et al., arXiv 1909.05858 (2019). [P]** Pretrain-from-scratch with control codes. Owning the
  base model — N/A for hosted, obsolete vs instruct-tuning.
- **PPLM (1912.02164), GeDi (2009.06367), FUDGE (2104.05218), DExperts (2105.03023). [P]** Decoding-time
  attribute control. PPLM needs activations+gradients; GeDi/FUDGE/DExperts need **full-vocabulary next-token
  logits** (and side models). Hosted chat APIs don't expose full logits (OpenAI gives top-20 logprobs; Anthropic
  none) → blocked.
- **Classifier-Free Guidance for text — Sanchez et al., arXiv 2306.17806 (2023); Context-Aware Decoding,
  arXiv 2305.14739. [P]** Run twice (with/without persona prompt), extrapolate logits
  `logit = uncond + γ(cond − uncond)` — a way to make a persona prompt "louder." Needs logits + two passes →
  **feasible locally, not on hosted chat APIs.** (This is the deployable analog of 2402.10962's CFG mitigation,
  but only on the local backend.)

### 4.4 Per-persona fine-tuning & soft prompts (needs weights/adapters)

- **LoRA / QLoRA per character** — the practical "character voice" route (the persona-vector paper itself used
  rs-LoRA rank 32). A per-character adapter (tens of MB) captures voice cheaply. **Catch for user-generated
  characters:** each needs a training dataset + a training job. **[P/S]**
- **Prefix-Tuning (2101.00190) / Prompt Tuning (2104.08691) / P-Tuning** — learn a persona as a small trained
  embedding instead of weights; still need backprop access + an inject-embeddings inference path (white-box).
  **[P]**
- **Character-LLM — Shao, Li et al., arXiv 2310.10158 (EMNLP 2023). [P]** Full-finetune per-character simulacrum
  built from "experiences/memories"; **dedicated weights beat prompts for consistency** but heavyweight per
  character.
- **PCL / Persona-Aware Contrastive Learning — arXiv 2503.17662 (ACL Findings 2025). [P]** Annotation-free:
  **Chain-of-Persona** (self-question against the profile before answering) + **Contrastive Self-Play Alignment**
  (DPO on with-profile vs without-profile pairs). Lifts 7B to GPT-4-comparable consistency and transfers to
  unseen characters. **Prompt-time analog is free:** a COP-style "what would *I* think about this?" self-check
  in the scratchpad before `say()`.
- **Hosted LoRA availability:** **Together AI** and **Fireworks** offer LoRA finetune + hosting (Fireworks
  **Multi-LoRA** = hundreds of LoRAs on one base at ~single-base cost; serves LoRAs only on on-demand/dedicated).
  Locks you to *their* open models, not Claude/GPT. **[S]**

### 4.5 Provider access matrix + the one local opportunity

| Backend | Logprobs | Activation R/W | Steering / control vectors | Custom LoRA |
|---|---|---|---|---|
| **Anthropic API** (Sei cloud-proxy) | **none** | no | no | no |
| **OpenAI API** | top-20 logprobs (read-only) | no | no | own finetune only |
| **Goodfire "Ember" API** | n/a | **yes (hosted SAE)** | **yes — Auto Steer + manual feature edits** | n/a |
| **Together / Fireworks** | yes | no | no | **yes (Multi-LoRA)** |
| **Local Ollama / llama.cpp** | full logits | **full R/W** | **yes — `--control-vector`** | yes (`ADAPTER`) |

- **Goodfire Ember** is the *only* hosted activation-steering-as-a-service — but **only on the Llama models it
  hosts** (Llama 3.3 70B / 3.1 8B), with **Auto Steer** turning a short prompt into feature selections +
  strengths. github.com/goodfire-ai/goodfire-sdk. **[S]**
- **llama.cpp control vectors are first-class:** `--control-vector <file.gguf>`,
  `--control-vector-scaled <file> <scale>`, `--control-vector-layer-range`; multi-vector support since PR #8137
  (Jun 2024). Vectors are generated with **`vgel/repeng`** (Theia Vogel; PyPI `repeng`; builds a vector from
  ~300 contrast statements and exports GGUF) or `jukofyork/control-vectors`. Warning: adding both poles of an
  axis cancels out. **[P/S]** This is the **practical, open-weights realization of Anthropic's persona-vector
  method** — feed a persona/trait description → contrast pairs → control vector → apply at a tuned coefficient.
- **Ollama status:** control vectors are **not yet in mainline** (open issue #8110 / unmerged PR #8148); the
  Modelfile supports `ADAPTER` (LoRA) but not control vectors as of mid-2026. The Ollama path needs a patched
  build or dropping to raw llama.cpp. **[P status / S exact merge state]**

### 4.6 Steering verdict for Sei

- **Hosted multi-provider path:** the only persona levers that survive text-in/text-out are **prompting** and
  **provider-side LoRA where supported** (Together/Fireworks on their open models; OpenAI's own finetune).
  **Every activation/decoding method (RepE, CAA, ITI, PPLM, GeDi, FUDGE, DExperts, CFG) is impossible on
  Anthropic and effectively impossible on OpenAI.** The lone exception is routing persona work to **Goodfire
  Ember** as a separate provider (real hosted steering, Llama-only).
- **Local Ollama/llama.cpp path = Sei's genuine non-prompting opportunity:** ship **per-character control
  vectors** built with `repeng` from the same description Sei already collects, applied via
  `--control-vector-scaled` at a tuned coefficient — directly mirroring the 2025 persona-vector method on open
  weights. Engineer around: coefficient/coherence brittleness (per-character scale + a coherence guardrail),
  Ollama mainline gap (patched build or raw llama.cpp), and the option of per-character LoRA for stronger but
  heavier voice.
- **Future spike, not now:** SAE feature steering and preventative-steering-during-finetune assume a white-box
  training pipeline Sei doesn't run for user-generated characters yet.

**Bottom line:** for hosted models, persona = **prompting + (optional) provider-side LoRA**, full stop. The one
novel non-prompting control Sei can realistically ship is **local-backend control vectors** — and it's a clean,
self-contained spike that reuses the existing persona description as input.

## 5. Roleplay craft (OOC/IC, length, refusal leakage, formatting, anti-repetition)

- **OOC pollution self-propagates — the single most important craft finding.** Parenthetical `(OOC: …)` notation
  is inherited from forum-RP training data; **once an OOC note appears in the transcript, the model treats it as
  part of the expected output format and reproduces it every subsequent turn**, collapsing immersion (severe on
  Character.AI's PipSqueak 2, which "dumps your private instructions straight into the story"). Fix = **edit out
  the earliest occurrence** and keep meta-talk in a **structurally separate channel**. **[S]** roborhythms; ST
  issue #235; ST-Copilot. → Validates Sei's hard `say()`-vs-`[think]`-scratchpad split; **never let OOC/debug
  text enter chat history.**
- **Response length is a measured escalation loop, not anecdote.** Verbosity bias in both human and LLM-judge
  preference labeling (arXiv 2310.10076); role-play RL without length control hits the upper limit "within a few
  steps" (CPO, arXiv 2508.09074). **[P]** Levers: `max_tokens` + a depth-0 author's-note brevity nudge +
  demonstrate brevity in examples; **"don't be verbose" backfires** (backyard.ai). The bot also **mirrors its
  own prior-turn length** ("Message Length Inertia") — short bot turns beget short bot turns. **[S]**
- **Refusal / "as an AI" leakage:** **RoleBreak — arXiv 2409.16727 (COLING 2025) [P]** identifies **query
  sparsity** + **role-query conflict** as the root causes of character hallucination/assistant-register leakage,
  and shows **"Narrator Mode" (generate supplemental in-fiction narration to absorb the conflict) beats
  refusal** on fidelity, hallucination, and coherence. → For out-of-scope/conflicting requests, **narrate
  around them in-character; never assistant-refuse.**
- **Formatting:** `*asterisks*` = actions, `"quotes"` = speech; **quote-style yields better prose** (matches
  literary training data); **don't mix the two styles in one message** (they pull from different training
  subsets). **Second-person is rated the *weakest* perspective for character voice** by practitioners (better
  for instruction) — a caution against blanket "second-person present-tense." Format must be **reinforced by
  example**, not just instructed. **[S]** backyard.ai; huggingface.co/ParasiticRogue. (Sei's chat-style `say()`
  largely sidesteps action/speech notation — keep it that way.)
- **Anti-repetition:** repetition is a self-reinforcing loop ("probability of repetition increases
  monotonically with prior repetitions"). The canonical RP "slop" list ("testament to," "shivers down spine,"
  "the air was thick with," "tapestry of," "symphony of," "unbeknownst," …) is community-banned. Levers: gentle
  `frequency/presence_penalty` (~0.05 for RP), post-gen regex strip, and **Antislop (arXiv 2510.15061, Oct
  2025)** — a **backtracking sampler** that retreats mid-sequence when a banned phrase starts, without full
  regeneration. **[P/S]**

## 6. Honesty about being wrong + stable opinions (own a hallucinated fact in-character; hold real takes)

### 6.1 Owning a wrong in-game fact — calibrated, in-character, not sycophantic

The literature frames a **three-way fork**, and both bad branches are documented:

- **Hard refusal / assistant-register** ← bad (RoleBreak 2409.16727: narrate around instead).
- **Sycophantic capitulation or doubling-down** ← bad:
  - **Sycophancy is an RLHF default.** Anthropic "Towards Understanding Sycophancy," arXiv 2310.13548 (Oct
    2023): five SOTA assistants sycophant; humans *and* preference models prefer convincingly-written
    sycophantic answers over correct ones a non-negligible fraction of the time. **[P]**
  - **The GPT-4o April 2025 incident:** over-weighting short-term thumbs-up feedback produced a sycophantic
    model; rolled back in 4 days; root cause "new feedback signals … overpowered existing safeguards."
    openai.com/index/sycophancy-in-gpt-4o. **[P]** → optimizing a companion for per-turn approval is the
    structural road to a yes-bot.
  - **Over-correction is also a failure:** "Are you sure?" alone can **flip a *correct* answer to a wrong one**
    (mistake-admission sycophancy); confident user rebuttals with detailed (even wrong) reasoning trigger
    capitulation; "persuasion bombing" = escalating confident rhetoric instead of recalculating. **[P]**
    arXiv 2509.16533; MIT Sloan. → You want *calibrated* error-owning, **not** reflexive admission.
- **The model can't self-detect wrongness reliably.** LLM epistemic markers ("I'm not sure") are poorly
  correlated with actual correctness (arXiv 2505.24778, 2411.06528). **[P]** And catching your *own in-domain*
  mistakes (Known Knowledge Errors) is empirically the hardest case — best-model accuracy < 65% on RoleKE-Bench,
  ~15% worse than out-of-scope errors (arXiv 2409.11726, EMNLP 2025). **[P]**
- **The target register (Anthropic "Claude's Character," 2024) [P]:** be "honest about the views it leans
  toward even if the person disagrees"; reject "adopting the views of whoever you're talking with" as "pandering
  and insincere"; "walk the line between underconfidence and overconfidence"; frame the model as "an imperfect
  entity with its own biases … not an objective and infallible source of truth." **A self-deprecating, fallible-
  but-sincere persona is *more* believable, not less.**

**Synthesis:** make fallibility a **consistent in-character tone** (Claude's Character) layered onto **narrate-
around** (RoleBreak), but **guard against over-capitulation** (don't let a player's "you sure?" flip a fact the
snapshot supports). Because the model can't gate on true confidence, the self-deprecating recovery is a
*stylistic* register you script — *and* you ground factual claims in the snapshot so the bot has an external
truth source to be honest against.

### 6.2 Stable preferences / opinions (not wishy-washy agreement)

- **"Do LLMs Have Consistent Values?" — Rozen et al., arXiv 2407.12878 (ICLR 2025). [P]** Default prompting does
  **not** reproduce human-consistent value correlations (vanilla personas are value-incoherent). A **"Value
  Anchor"** prompt — answer *as a person who emphasizes a given value* — makes value correlations snap into
  agreement with Schwartz's circular model: "remarkably coherent and human-like value profiles." → **Anchor on a
  central value/trait and *derive* preferences from it**, rather than enumerating disconnected likes.
- **Traits are a more stable substrate than raw preferences** (PACIFIC, arXiv 2602.07181 [?recent]): "personality
  tends to be consistent and can support reliable decisions even when individual preferences are sparse" — define
  the trait, derive the opinions.
- **A persona blurb alone doesn't guarantee multi-turn opinion stability:** persona-conditioned models are stable
  on *structured* repeated tests but show "no statistical difference between inter- and intra-persona consistency"
  in actual chat (arXiv 2602.18462 [?]; 2506.02659). → opinion stability needs reinforcement (anchoring +
  exemplars + persisted memory of stated preferences), not just a description.
- **PCL's Chain-of-Persona** (2503.17662) — self-question against the profile *before* answering — is the cheap
  prompt-time analog for opinion consistency. **[P]**
- **Stable tastes require willingness to mildly disagree** with the player (same lever as anti-sycophancy,
  Claude's Character). **[P]**

## Adopt for Sei — roadmap-ready

Grounded in the current code: persona is generated by `src/main/personaExpansion.ts` (6-section template) and
rendered front-loaded into the **cached system prefix** via `renderPersona` (`orchestrator.js:826`), behind the
`cache_control` boundary; the per-turn seed blocks (`composeSeedBlocks`) carry player/memory/heartbeat/snapshot
but **no persona anchor near the recency end**; speech is `say()`-only; bans live in `BASELINE_INSTRUCTIONS`.

### (a) Persona-expansion prompt design — evolve the 6-section template

Keep the streaming 6-section structure (research says structure is for authoring, not quality — don't rip it
out), but upgrade the *content rules* per §3.5:

1. **Generate-vs-preserve contract (NEW, fixes problem #1).** Prepend the system prompt with: seed facts are
   immutable; expand *around* them; invent only seed-consistent specifics and never present an invention as
   user-supplied. (Fact-pinning; 2406.20094 granularity; meta-prompting confabulation risk.)
2. **Thicken thin blurbs at authoring time.** Because granularity scales with input richness (PersonaHub), a
   one-liner can't yield a rich persona — add **1–2 optional clarifying questions** in the create flow (e.g.
   "what's one thing they love / can't stand?") before expanding, or have the expander explicitly enumerate
   inferred specifics it's adding.
3. **Make `# VOICE` an enumerated register spec, not free-text** (Voicing Personas 2505.17093): closed fields for
   register/formality/casing/pace/sentence-length + a **catchphrase/idiom list** (RoleLLM), alongside the existing
   5–7 sample lines. The sample lines should demonstrate **register, not situations** (PersonaHub: scenes don't
   generalize; voice does).
4. **Constrain stereotyped defaults** (Voicing Personas bias result): require the expander to set
   voice/demographic-adjacent slots from the seed or as a deliberate choice, never leave them implicit.
5. **Add a `# STANCE & OPINIONS` element** (or fold into VOICE/REACTIONS): a central value/trait anchor + 2–3
   concrete stable likes/dislikes *derived from it* (Value-Anchor 2407.12878), so the character has takes to be
   consistent about. Lean into "stronger opinions improve believability."
6. **Counter the assistant attractor explicitly** (Assistant Axis): the expander's anti-default register list
   should be *demonstrated* in the sample lines, complementing (not replacing) `BASELINE_INSTRUCTIONS` bans.

### (b) Per-turn re-injection core — the #1 structural fix

Add a **compressed persona anchor (name + 2–3 defining traits + register cue + one stance line, ~40–80 tokens)
re-injected at low depth near the END of the per-turn context** — append it as the last (or near-last) seed block
in `composeSeedBlocks`, after the snapshot, *outside* the cached prefix. This is SPR / Author's Note / Character
Depth-Prompt (2402.10962; SillyTavern docs), the single best retraining-free anti-drift fix, and it matters
*more* on Haiku (PERSIST: small + history = worst drift; PersonaGym: Haiku 3.64). Keep the full persona in the
cached prefix for the facts; the tail anchor fights the ~8-turn attention decay the front-load can't. Derive the
anchor deterministically from `persona.expanded` at session start so it costs ~nothing.

### (c) Few-shot voice-example scheme

The `# VOICE` sample lines should be surfaced as **actual `say()`-format exemplars** (Ali:Chat principle: examples
in the exact output format), and the **most representative line placed last** (ICL recency). Bias toward **positive
demonstrations over the negation bans** Sei currently relies on (NeQA inverse scaling; demonstrations register
better than negations). Note these are the *fragile* field under context pressure (V2 drops examples first) — so
the compressed register cue must also live in the (b) tail anchor, not only in the front-loaded examples.

### (d) Anti-drift + honesty contract

- **Drift:** (b) tail re-injection + (c) positive exemplars + redundancy on the 2–3 defining traits (echo in
  persona body, an example, and the tail anchor).
- **Honesty:** keep the snapshot as the factual ground truth, and add a calibrated **own-it-in-character**
  rule — when the snapshot contradicts something the bot said, recover in-voice/self-deprecatingly (RoleBreak
  narrate-around; Claude's Character), **but do not capitulate to a bare "you sure?"** when the snapshot supports
  the bot (mistake-admission sycophancy 2509.16533). Optionally a cheap **Chain-of-Persona** scratchpad self-check
  ("what would *I* think here?") before `say()` (PCL 2503.17662) — fits Sei's existing private-scratchpad design.
- **Anti-sycophancy:** never tune `say()` for per-turn approval (GPT-4o April 2025); a stable-opinion character
  must be willing to mildly disagree (Claude's Character).
- **OOC hygiene:** keep enforcing the hard `say()`-vs-`[think]` split (RoleBreak/Character.AI OOC-pollution
  evidence validates it) — never let meta text reach chat history.

### (e) Hybrid-steering spike — which methods, which backend

- **Now (hosted Anthropic/OpenAI/compat):** nothing beyond prompting + optional provider-side LoRA. Don't spend
  effort hunting for activation access that doesn't exist.
- **Worth a future spike — local Ollama/llama.cpp control vectors.** Build a **per-character persona control
  vector** with `vgel/repeng` from the description Sei already collects (the open-weights realization of
  Anthropic's persona-vectors 2507.21509), apply via `--control-vector-scaled`. Self-contained, reuses existing
  input, and is the one genuinely non-prompting persona lever Sei can ship. Budget for: per-character coefficient
  tuning + a coherence guardrail (2407.12404), and the Ollama-mainline gap (patched build or raw llama.cpp).
  Per-character **LoRA** (Together/Fireworks hosted, or local) is the heavier, stronger alternative if a training
  pipeline ever exists. **Goodfire Ember** is the only hosted steering option but Llama-only — a niche provider
  route, not a default.
- **Not now:** SAE feature steering and preventative-steering-during-finetune (white-box training pipeline
  Sei doesn't run for user-generated characters).
