# Research: Non-LLM & Hybrid Minecraft Agents vs. Sei's mineflayer Control Surface

**Stream:** RL / imitation learning / world models / foundation policies, and whether/how
they can be used in hybrid with Sei.
**Author:** research agent
**Date:** 2026-06-25
**For milestone:** v0.4 (Phase 17 — Minecraft Competence is the consumer of this)
**Verdict in one line:** Essentially **none of the marquee RL/IL/world-model Minecraft agents
are runnable inside Sei**, because they are pixels-in / keyboard-mouse-out policies bound to the
rendered Java client, while Sei's bot is a **headless protocol client (mineflayer)** that sees
structured state and acts through a high-level API. What *does* transfer is the **hybrid
architecture pattern** (LLM planner over a small set of robust low-level skills) and the
**classical game-AI control toolkit** (GOAP / behavior trees / utility AI / steering), which run
natively on mineflayer's structured state. Recommend: adopt the planner→skill boundary, implement
micro-controllers as deterministic/heuristic skills (optionally a tiny supervised model for
target selection), and **defer pixel-based RL entirely**.

---

## 0. The load-bearing fact: two incompatible control surfaces

Every technique below must be judged against one axis. There are two fundamentally different ways
to drive a Minecraft agent:

| | **Pixel / human-interface control** | **Protocol / structured control (Sei)** |
|---|---|---|
| Observation | Rendered RGB frame (typically 128×128 to 640×360), the same screen a human sees | Decoded packet state: block grid, entity list with positions/types, inventory, vitals — what `snapshot.js` composes |
| Action | Low-level mouse Δ + keyboard at ~20 Hz (camera pitch/yaw deltas, WASD, click) | High-level intents: `goTo(x,y,z)`, `dig`, `attackEntity`, `craft`, `equip`, pathfinder does the steering |
| Stack | MineRL / Malmo wrapper around the **actual Java game client** + a GPU framebuffer | mineflayer: a Node.js **protocol client**, no renderer, no GPU, no game window |
| Who uses it | VPT, STEVE-1, GROOT, DreamerV3, MineRL comp, MineDojo, and the low-level controllers of Plan4MC / JARVIS-1 / Optimus | Voyager, mineflayer-pathfinder, mineflayer-pvp, **Sei** |

**Consequence that recurs in every verdict below:** a neural policy trained on pixels + mouse/keyboard
cannot be "plugged into" mineflayer. There is no rendered frame to feed it, and its output (a yaw
delta and a left-click) is not what mineflayer consumes (`bot.attack(entity)`,
`bot.lookAt(vec3)`). To use such a policy you would have to **run a real Minecraft Java client with a
rendered framebuffer alongside the bot**, run a ~0.2–0.5 B-parameter transformer at 20 Hz on a GPU,
and translate its synthetic input back into the world — a completely different (and consumer-hostile)
architecture from Sei's headless Electron-forked Node process. That is the central feasibility filter.

Sei's actual surfaces, for grounding the rest of this doc:
- Observation: `src/bot/adapter/minecraft/observers/snapshot.js` — line-oriented text snapshot (pos,
  biome, time, vitals, held, inventory, craftable, nearby entities with `#N` handles, blocks around).
- Actions: `src/bot/adapter/minecraft/registry.js` — closed, Zod-typed registry of ~18 world actions;
  combat is `behaviors/attack.js` (multi-swing, pursue-to-reach, 600 ms swing cadence) and
  `behaviors/combat.js`; navigation is `behaviors/pathfind.js` (mineflayer-pathfinder = A*, wrapped
  with a wall-clock timeout). The roadmap's locked decision #8: **the registry stays closed; no
  code-as-skills**. That rules out Voyager-style code generation as a shipping mechanism.

## 1. RL / IL / world-model / foundation policies — survey + mineflayer verdict

### 1.1 The MineRL competitions (NeurIPS 2019–2022) — the benchmark substrate

- **What:** `MineRLObtainDiamond` and the sample-efficiency competitions, plus the **MineRL-v0**
  dataset of **60M+ state–action pairs** of human play. Agent starts empty and must climb the crafting
  tech tree to a diamond from sparse rewards. ([arXiv:1904.10079](https://arxiv.org/abs/1904.10079),
  [NeurIPS 2020 overview arXiv:2101.11071](https://ar5iv.labs.arxiv.org/html/2101.11071),
  [Diamond 2021 arXiv:2202.10583](https://arxiv.org/abs/2202.10583))
- **BASALT** (2021–2022): the *fuzzy-task* track — no reward function, agents judged by humans on
  open goals like "build a house", "find a waterfall". Forced learning-from-human-feedback.
  ([BASALT 2022 retrospective arXiv:2303.13512](https://arxiv.org/pdf/2303.13512))
- **Obs/action:** pixels (64×64 POV) + low-level mouse/keyboard, via the Malmo/MineRL wrapper around
  the Java client.
- **mineflayer verdict:** **Substrate, not a deliverable.** The *dataset and benchmark* are why VPT
  et al. exist; nothing here is a model you ship. The diamond task itself is *trivial* for a protocol
  client + LLM planner (Voyager and Sei reach iron/diamond tier via structured `craft`/`dig` without
  any of this), which is itself the key insight: **the hard RL problem (pixel control of the tech
  tree) is one Sei sidesteps by construction.**

### 1.2 OpenAI VPT — Video PreTraining (June 2022)

- **What:** A foundation policy trained by **behavior cloning** on **~70k hours of unlabeled YouTube**
  Minecraft footage. The trick: hire contractors to record **~2k hours** of labeled play, train an
  **Inverse Dynamics Model (IDM)** that infers the mouse/keyboard actions behind each video frame, use
  the IDM to pseudo-label the 70k hours, then BC a policy on that. First agent to craft a **diamond
  pickaxe** (after RL fine-tuning); zero-shot it chops trees → planks → crafting table (~1,000 actions,
  ~50 s of play). ([OpenAI/Video-Pre-Training repo](https://github.com/openai/Video-Pre-Training),
  [MarkTechPost 2022-06-27](https://www.marktechpost.com/2022/06/27/openai-introduces-video-pretraining-vpt-a-novel-semi-supervised-imitation-learning-technique/),
  [InfoQ](https://www.infoq.com/news/2022/07/openai-vpt-minecraft/))
- **Obs/action:** 128×128 RGB → mouse/keyboard at 20 Hz. ~0.5 B params (the `foundation-model-2x` /
  `3x` checkpoints).
- **mineflayer verdict:** **Incompatible as a runtime policy.** Pure pixels-in/HID-out. Running it
  needs a rendered client + GPU + 20 Hz inference. **What is reusable conceptually:** VPT's *IDM trick*
  is the canonical example of harvesting cheap supervision from gameplay video — relevant only if Sei
  ever wanted to *train its own* structured-state controller from logs (it would build the structured
  analog of an IDM, not use VPT's weights). Not actionable for v0.4.

### 1.3 DreamerV3 — world-model RL (arXiv Jan 2023; Nature, Apr 2025)

- **What:** Hafner et al., "Mastering diverse control tasks through world models." A single
  hyperparameter config solving 150+ tasks; **first algorithm to collect a diamond in Minecraft from
  scratch with no human data and no curriculum.** Learns a latent world model and "imagines" rollouts
  to train the actor/critic. First diamond at ~**30 M env steps (~17 days of play)**.
  ([Nature s41586-025-08744-2](https://www.nature.com/articles/s41586-025-08744-2),
  [arXiv:2301.04104](https://arxiv.org/abs/2301.04104),
  [danijar.com/dreamerv3](https://danijar.com/project/dreamerv3),
  [GIGAZINE 2025-04-03](https://gigazine.net/gsc_news/en/20250403-google-deepmind-dreamerv3/))
- **Obs/action:** pixels (64×64) + the MineRL discrete action set. Trained tabula rasa with a reward
  shaped over tech-tree milestones.
- **mineflayer verdict:** **Incompatible and irrelevant to ship.** It is a *training-time* result
  about sample efficiency on pixel control. The achievement ("diamond with no human data") is precisely
  the problem mineflayer makes a non-problem. There is no artifact here to embed; the policy is again
  pixel-bound, and 17 days of online RL per behavior is the opposite of a consumer-desktop constraint.
  **Useful only as evidence** that *online RL from scratch is the wrong cost class for Sei.*

### 1.4 STEVE-1 — instructable goal-conditioned policy (NeurIPS 2023)

- **What:** Instruction-tuned VPT. Two stages: (1) fine-tune VPT to follow goals expressed in
  **MineCLIP's latent space** via self-supervised BC + hindsight relabeling; (2) train a CVAE "prior"
  mapping **text → MineCLIP latent**. Follows short-horizon text/visual commands ("get dirt", "chop a
  tree"); completes 12/13 early-game tasks; **trained for ~$60** of compute (because it reuses VPT and
  MineCLIP). ([arXiv:2306.00937](https://arxiv.org/abs/2306.00937),
  [project site](https://sites.google.com/view/steve-1),
  [NeurIPS 2023 PDF](https://proceedings.neurips.cc/paper_files/paper/2023/file/dd03f856fc7f2efeec8b1c796284561d-Paper-Conference.pdf))
- **Obs/action:** inherits VPT — pixels + mouse/keyboard, 20 Hz.
- **mineflayer verdict:** **Incompatible as runtime, but architecturally the most interesting one.**
  STEVE-1 is exactly the "small instructable low-level controller you'd want under an LLM planner" —
  and it is the controller used by hybrid systems below. But it is a **pixel policy**: it cannot read
  mineflayer's entity list and cannot emit `bot.attack`. The *role* STEVE-1 plays (text-conditioned
  micro-controller) is the role Sei should fill with **classical/heuristic skills on structured state**,
  not with STEVE-1's weights.

### 1.5 GROOT / GROOT-2 — foundation policy from video instructions (ICLR 2024 / Dec 2024)

- **What:** GROOT is a goal-conditioned policy learned **self-supervised from gameplay video** with a
  C-VAE: a non-causal transformer encodes a *reference video* into a latent goal, a causal-transformer
  controller decodes it to actions. SOTA on the **Minecraft SkillForge** benchmark; ~70% win rate vs.
  prior generalists; shows emergent goal composition. GROOT-2 (Dec 2024) adds weakly-supervised
  multimodal (text + video) instruction following. ([arXiv:2310.08235](https://arxiv.org/abs/2310.08235),
  [GROOT-2 arXiv:2412.10410](https://arxiv.org/html/2412.10410v1))
- **Obs/action:** pixels + mouse/keyboard, transformer over frame sequences.
- **mineflayer verdict:** **Incompatible** (same pixel/HID binding). Notable only as the current
  frontier of "instruction → low-level behavior" foundation policies; same role/structured-state
  substitution argument as STEVE-1.

### 1.6 MineDojo + MineCLIP — knowledge base + reward model (NeurIPS 2022, Outstanding Paper)

- **What:** A simulation suite (3,142 benchmark tasks) **plus an internet-scale knowledge base** (730k
  YouTube videos w/ transcripts, ~7k wiki pages, ~340k Reddit posts). **MineCLIP** is a contrastive
  video–text model trained on that YouTube corpus; it scores how well a 16-frame clip matches a goal
  string and is used as a **learned dense reward** for RL. ([arXiv:2206.08853](https://arxiv.org/abs/2206.08853),
  [NeurIPS 2022](https://proceedings.neurips.cc/paper_files/paper/2022/hash/74a67268c5cc5910f64938cac4526a90-Abstract-Datasets_and_Benchmarks.html),
  [MineDojo repo](https://github.com/MineDojo/MineDojo))
- **mineflayer verdict:** **MineCLIP itself is a pixel/video model — incompatible.** But two assets are
  indirectly useful: (a) the **wiki/Reddit/YouTube knowledge** is exactly the kind of corpus an LLM
  already internalizes, so Sei's brain effectively gets MineDojo's "knowledge" for free through the base
  model — *no integration needed*; (b) MineDojo's **task taxonomy** is a good source of competence
  targets and acceptance tests for Phase 17. Don't integrate the model; mine the benchmark for test design.

### 1.7 GATO and the "single generalist policy" idea (DeepMind, 2022)

- **What:** A single transformer trained across Atari, captioning, robotics, etc. as one token stream.
  Often cited as the generalist-agent vision. Not Minecraft-specific; tokenizes images + proprioception
  + actions.
- **mineflayer verdict:** **Not applicable.** No Minecraft-on-protocol variant exists; it's a
  multi-task pixel/robotics policy. Mentioned only to close the "foundation policy" set — irrelevant to Sei.

### 1.8 The RL framework toolbox (hierarchical RL, options, goal-conditioned RL)

- **Options framework** (Sutton/Precup/Singh 1999): temporally-extended actions ("skills") with their
  own policy + termination condition over a base MDP. **Hierarchical RL (HRL):** a high-level policy
  selects among options/sub-goals; low-level policies execute them. **Goal-conditioned RL (GCRL):** a
  policy `π(a | s, g)` parameterized by a goal, trained with hindsight relabeling (HER). These are the
  formal underpinnings of every hybrid system in §2 (Plan4MC's "skills" are options; STEVE-1 is a GCRL
  policy; the LLM is the high-level policy).
- **mineflayer verdict:** **The *abstraction* is directly adoptable; the *learning* is not needed.**
  Sei already has the options framework in disguise: each registry action is a temporally-extended
  option with an implicit termination (timeout / success). The LLM is the high-level policy over those
  options. The value for v0.4 is **conceptual vocabulary** ("treat combat-micro as an option with a
  termination condition") rather than any RL algorithm. You get the benefits of HRL without training
  anything because the options are hand-written and the meta-policy is the LLM.

## 2. Hybrid architectures — LLM planner + learned/low-level controllers (the user's main interest)

This is where the field actually converged, and where Sei is *already* sitting. The universal pattern:
**an LLM/foundation model plans high-level sub-goals; a library of low-level controllers executes
them.** The systems differ mainly in (a) what the controllers are and (b) what the observation channel
into the planner is.

### 2.1 Plan4MC (PKU, Mar 2023) — RL skills + LLM-built skill graph

- **Architecture:** Decompose tasks into **basic skills** learned by RL with intrinsic rewards, in
  three classes: **Finding-skills, Manipulation-skills, Crafting-skills**. An LLM is used **offline,
  before execution**, to extract skill dependencies and **build a skill graph**; at run time a graph
  search produces the skill plan. Solves 24 tasks, many needing 10+ sequential skills.
  ([arXiv:2303.16563](https://arxiv.org/abs/2303.16563),
  [Plan4MC repo (PKU-RL)](https://github.com/PKU-RL/Plan4MC), [site](https://sites.google.com/view/plan4mc))
- **Boundary:** LLM = static planner (graph builder), explicitly kept out of the control loop "to avoid
  uncontrollable failures." Controllers = pixel RL policies (MineDojo/MineCLIP-trained).
- **mineflayer relevance:** **The planner/controller *split* is the takeaway; the controllers are not
  portable** (pixel RL). For Sei, the "skill graph" is interesting: an offline, LLM-derived dependency
  graph of competences (smelt needs furnace+fuel+ore; iron-pick needs iron-ingot×3 needs furnace…) could
  make Phase 17's "progress to iron tier unaided" more reliable than per-turn improvisation. But Sei's
  controllers would be the **registry actions**, not RL skills.

### 2.2 DEPS — Describe, Explain, Plan, Select (NeurIPS 2023)

- **Architecture:** An LLM-based **interactive planner** that (D)escribes the current state, (E)xplains
  failures, re-(P)lans, and uses a learned **goal Selector** to order sub-goals by reachability/proximity.
  Improves long-horizon success by closing the loop on execution feedback.
  ([arXiv:2302.01560](https://arxiv.org/pdf/2302.01560))
- **Boundary:** LLM plans + explains; a small trainable selector handles "which sub-goal is closest";
  low-level controllers (again pixel/MineRL) execute.
- **mineflayer relevance:** **The *replan-on-failure* loop maps directly onto Sei.** Sei already feeds
  `lastActionResult` and in-flight status back into the snapshot — that *is* DEPS's "Explain" channel in
  miniature. The "Select" idea (a cheap scorer that picks the nearest viable sub-goal) is a candidate for
  a tiny heuristic/utility function rather than the LLM, reducing token cost on routing decisions.

### 2.3 JARVIS-1 (CraftJarvis, Nov 2023) — memory-augmented multimodal planner

- **Architecture:** Multimodal LM (perceives pixels via a captioner) as **high-level planner** →
  **low-level instruction-following controllers** (STEVE-1/VPT-style) execute. Adds a **multimodal
  memory** of past successful trajectories that is retrieved to improve planning. Completes 200+ tasks;
  **5× more reliable than prior SOTA on ObtainDiamondPickaxe.**
  ([arXiv:2311.05997](https://arxiv.org/abs/2311.05997), [site](https://craftjarvis-jarvis1.github.io/))
- **Boundary:** LM planner (with retrieved memory) → text/visual goal → pixel controller.
- **mineflayer relevance:** **Its memory mechanism rhymes with Sei's MEMORY.md/experience retrieval**
  (Phase 16). The controller half is again pixel-bound and not portable. JARVIS-1 is the strongest
  evidence that *the planner+memory+controller decomposition scales to long horizons* — validating Sei's
  overall shape, while confirming the learned controllers don't transfer to protocol control.

### 2.4 Optimus-1 / Optimus-2/3 (2024) — hierarchical knowledge graph + experience

- **Architecture:** A **Hybrid Multimodal Memory** module: experience organized as a **hierarchical
  directed knowledge graph** plus an abstracted experience pool, feeding situation-aware planning. Excels
  at long-horizon tasks; uses visual observation directly as planning condition (vs. JARVIS-1's
  caption-to-text). ([Optimus-1 arXiv:2408.03615](https://arxiv.org/html/2408.03615v2))
- **mineflayer relevance:** Same story — **the knowledge-graph-as-memory idea is portable** (and
  overlaps Sei's roadmap memory work), the controller is not. Reinforces that the differentiator among
  hybrids is *planning + memory quality*, which is squarely Sei's LLM-brain strength.

### 2.5 Voyager (NVIDIA, May 2023) — the one hybrid that *is* mineflayer-native

- **Architecture:** GPT-4 + (1) automatic curriculum, (2) an **ever-growing skill library of executable
  JavaScript using the mineflayer API**, (3) iterative prompting with execution-error / self-verification
  feedback. No model fine-tuning. 3.3× more unique items, tech-tree milestones up to 15.3× faster than
  prior SOTA; skills transfer to new worlds. ([arXiv:2305.16291](https://arxiv.org/abs/2305.16291),
  [Voyager repo](https://github.com/minedojo/voyager), [site](https://voyager.minedojo.org/))
- **Boundary:** No learned controller at all — the LLM *writes code* against mineflayer; the "controller"
  is whatever JS it emits, accumulated as a retrievable library.
- **mineflayer relevance:** **This is the proof that high-level structured control + LLM is sufficient
  for the full tech tree without any pixel RL.** BUT Voyager's mechanism (LLM generates+stores executable
  code) **directly violates Sei's locked decision #8 (closed registry, no code-as-skills).** So Voyager
  is a *negative template*: take its evidence (mineflayer is enough), reject its method (code generation).
  The closed-registry equivalent of a "skill library" is **a curated set of parameterized skills the LLM
  composes**, not generated code.

### 2.6 Where the planner/controller boundary actually gets drawn (synthesis)

Across all hybrids the boundary sits at the same place: **the planner reasons in symbols/sub-goals at
~0.2–2 Hz; the controller reacts in the world at ~20 Hz.** The split exists because:
1. **Latency:** an LLM cannot run a 20 Hz reactive loop (a single Sei brain turn is on the order of a
   1–20 s round-trip; combat/dodge needs sub-100 ms reactions).
2. **Reliability:** keeping the LLM offline/out-of-loop (Plan4MC) or feedback-gated (DEPS) prevents
   per-tick hallucination.
3. **Cost:** tokens per tick at 20 Hz is economically impossible; controllers are cheap/local.

**For Sei the boundary is forced by the same latency wall.** Today, the LLM brain *is* both planner and
(too-slow) controller — it dispatches `attackEntity` and then waits a full turn. The mismatch shows up
exactly in the failure modes Phase 17 names: **creeper deaths and weak combat** are sub-second control
problems that a per-turn LLM cannot solve. **The right move is to push reactive micro-control below the
LLM**, into fast local skills the registry exposes — which is the hybrid pattern, implemented with
classical control rather than pixel RL.

**What a Sei controller consumes/emits (concrete):**
- *Consumes* (from mineflayer state already in `snapshot.js`/observers): self position+velocity, target
  entity position+velocity+type, health/food, held item & reach, line-of-sight (`observers/lineOfSight.js`),
  nearby hostile list.
- *Emits* (mineflayer API): `bot.lookAt` / `bot.setControlState('forward'|'back'|'left'|'right'|'jump'|
  'sprint', bool)`, `bot.attack(entity)`, `bot.activateItem()` (shield), pathfinder goals. No pixels, no
  yaw deltas — structured intents at the rate mineflayer's physics ticks (20 Hz) allow.
- *Runs* in the bot's own Node event loop, between LLM turns — the LLM says "engage that skeleton", a
  `combat-micro` skill takes over strafing/blocking/retreating until the engagement ends or the LLM
  preempts it (which the FSM's P1_CHAT preemption already supports).

**Prior art that this is feasible on mineflayer:** **`mineflayer-pvp`** (PrismarineJS) already implements
melee micro — approach, strafe, jump for critical hits, shield timing — purely on structured state, and
**mineflayer-pathfinder** is the A* nav Sei already wraps. These are existence proofs that the
"controller" tier works at the protocol layer without a single learned weight.

## 3. Small / embeddable models on a consumer desktop — feasibility

The question: can Sei run a *learned* low-level policy or distilled controller locally inside the
Electron-forked Node bot?

**Runtime options that exist:**
- **onnxruntime-node** (ONNX Runtime, Node binding) — CPU by default, optional CoreML/DirectML/CUDA EP.
  Sei already ships native modules and rebuilds them via `electron-builder install-app-deps`, so adding
  one ONNX native dep is mechanically in-budget (it's asar-unpacked like the rest of `node_modules/**`).
- **transformers.js / onnxruntime-web (WASM)** — pure-JS/WASM, no native build, but slower; fine for
  tiny MLPs.
- **TensorFlow.js** — similar tier.

**Footprint/latency reality for the models that would actually help:**
- A **tiny MLP** for threat assessment / target selection / aim correction over ~20–50 structured
  features is **<1 MB** and runs in **well under 1 ms on CPU** — trivially embeddable, no GPU. This is the
  *only* class of learned model that is realistic here.
- A **VPT/STEVE-1/GROOT-class transformer** (0.2–0.5 B params, vision encoder, 20 Hz) needs a GPU, a
  rendered framebuffer, and ~tens of ms/step. **Not viable** in a background Node utilityProcess on an
  arbitrary consumer machine, and pointless without pixels anyway. Hard no.

**When does a tiny supervised model beat hand-tuned heuristics?** Only when the decision boundary is
genuinely high-dimensional, non-obvious, and you have labeled data. For Sei's structured features the
honest answer is **rarely**:
- **Aim / lead-the-target, creeper-dodge distance, retreat threshold** — low-dimensional, physically
  interpretable; a heuristic (or a closed-form intercept calc) is *better* than a model: explainable,
  zero training data, zero footprint, tunable by hand. A learned model here is over-engineering.
- **Target selection among many simultaneous threats** (which of 5 mobs to engage/flee, weighing type,
  distance, your gear, the player's safety) — this is the one place a small utility-scored or
  tiny-learned classifier *might* edge out hand-tuned weights, because the trade-off surface is messy.
  Even then, **utility AI (hand-weighted scoring) is the 90% solution** and should be tried first; a
  learned scorer is a later optimization only if you collect engagement-outcome data.

**Distillation angle:** you can't distill VPT/STEVE-1 into mineflayer heuristics directly (different
I/O), but you *can* distill *observed good play* (your own logged engagements, or scripted-expert
rollouts) into a small structured-state policy — i.e., behavior-clone on `(structured features →
mineflayer control)` pairs. That is a real path, but it is a **research/data-collection project**, not a
v0.4 deliverable. For this milestone, heuristics win on every axis that matters (ship date, footprint,
debuggability).

## 4. Specialized non-LLM decision-making that pairs with an LLM planner

This is the toolkit that is actually shippable, because it runs natively on mineflayer's structured
state and has decades of game-AI pedigree.

### 4.1 GOAP — Goal-Oriented Action Planning
- **What:** Give a planner the current world state, a goal state, and a set of actions with
  preconditions+effects+cost; A*-search the action space for a least-cost plan to the goal. Descends from
  **STRIPS (1971)**; popularized by **Jeff Orkin for F.E.A.R. (2005)**, where the AI was GOAP + a 3-state
  machine (GoTo / Animate / UseSmartObject). ([Game Developer: Building the AI of F.E.A.R.](https://www.gamedeveloper.com/design/building-the-ai-of-f-e-a-r-with-goal-oriented-action-planning),
  [GOAP overview](https://tonogameconsultants.com/game-ai-planning/))
- **Fit for Sei:** GOAP *is the closed-registry skill-graph idea made executable*. Model registry
  actions as GOAP operators (precondition: `has(iron_ore) ∧ near(furnace) ∧ has(fuel)`; effect:
  `has(iron_ingot)`), and a local planner can chain "progress to iron tier" deterministically — fast,
  offline, no tokens, no hallucination. This is the strongest classical match for Phase 17's
  "unaided progression to iron tier" and for robust multi-step crafting/smelting. **The LLM proposes the
  goal; GOAP guarantees a valid action sequence.**

### 4.2 Behavior Trees (BTs)
- **What:** Hierarchical tree of fallback/sequence/decorator/leaf nodes ticked at a fixed rate; the modern
  industry standard for reactive NPC behavior (replaced sprawling FSMs). Composable and inspectable.
  ([Nez AI docs: FSM/BT/GOAP/Utility](https://anshuman-kumar.gitbook.io/nez-doc/ai-fsm-behavior-tree-goap-utility-ai))
- **Fit for Sei:** Ideal for **reactive safety/combat micro** running between LLM turns: a "survival" BT
  ticked at 20 Hz — `if creeper within N and fuse-risk → retreat; else if hostile in reach → strafe+hit;
  else idle` — directly fixes "only detects mobs after being hit" and "destroyed by creepers." Each BT is
  a single closed-registry skill the LLM can arm/disarm. Mature JS libs exist (`behaviortree`,
  `mineflayer-statemachine` for the mineflayer-native variant).

### 4.3 Utility AI
- **What:** Score each candidate action with a utility function over world features (curve-shaped
  considerations), pick the max each tick. Great for graded trade-offs ("heal because hp<30%" vs "flee"
  vs "fight"). Modern engines blend Utility + GOAP + BT.
  ([GOBT synergy paper, J. Multimed. Inf. Syst.](https://www.jmis.org/archive/view_article?pid=jmis-10-4-321))
- **Fit for Sei:** Best tool for **target selection and engage/flee decisions** among multiple threats
  (the §3 "messy trade-off" case) — hand-weighted first, learnable later. Cheap, local, tunable.

### 4.4 FSM, steering behaviors, classical search
- **FSM:** Sei's brain already has an explicit FSM (`src/bot/brain/fsm.js`, P0–P3 priority queue). Combat
  states (Engage/Strafe/Retreat/Reposition) slot in as a sub-FSM under a single combat skill.
- **Steering (Reynolds):** seek/flee/pursue/evade/separation vectors are the textbook primitive for
  creeper-evasion and kiting; ~10 lines on `bot.entity.position`/velocity, emit `setControlState`.
- **A\*/pathfinding:** already present via mineflayer-pathfinder. Mob-threat classification can be a
  flat rule table (entity type → hostile/neutral/passive from `minecraft-data`) — no model needed.

**Composition pattern (the recommended stack):** LLM (intent/persona/social) **→** GOAP (valid
multi-step plans for progression/crafting) **→** Behavior Tree / Utility (reactive safety + combat micro
at 20 Hz) **→** steering + pathfinder (motor primitives) **→** mineflayer API. This is the same
planner→controller hierarchy as the research hybrids, with **every tier implemented in classical,
local, structured-state code** instead of pixel neural nets.

## 5. What's realistic vs. research-only for this milestone

| Technique | Status for a shipping v0.4 |
|---|---|
| VPT, STEVE-1, GROOT/-2, DreamerV3, MineCLIP, GATO | **Research-only / out of scope.** Pixel + HID control; need a rendered client + GPU; I/O-incompatible with mineflayer. Defer indefinitely. |
| Online RL from scratch (DreamerV3-style) | **Out of scope.** Days of compute per behavior; wrong cost class for a desktop app. |
| Distilling a small *structured-state* BC policy from logs | **Research spike, not v0.4.** Real but needs data pipeline + training; revisit post-milestone if heuristics plateau. |
| Tiny embeddable MLP (target selection) via ONNX | **Feasible but optional.** Only worth it after utility-AI heuristics exist and you have engagement data. |
| Hybrid *architecture* (LLM planner over fast local controllers) | **Realistic and recommended now.** It's the field consensus and matches Sei's FSM. |
| GOAP / Behavior Tree / Utility AI / steering on structured state | **Realistic now; the right tool.** Native to mineflayer, no GPU, debuggable, respects the closed registry. |
| Voyager-style code-as-skills | **Excluded by locked decision #8.** Use as evidence (mineflayer suffices), not as method. |
| mineflayer-pvp / mineflayer-statemachine / pathfinder plugins | **Realistic now.** Proven protocol-layer controllers; adopt/adapt rather than build from zero. |

## Hybrid options for Sei, ranked by feasibility

**Ranked most → least feasible. The top tier is what Phase 17 should actually pursue.**

1. **Reactive safety/combat micro-controller as a closed-registry skill (Behavior Tree + steering).**
   A 20 Hz survival+combat loop running in the bot process between LLM turns: proactive hostile
   detection (read entity list every tick), creeper-evasion via flee/evade steering before the fuse,
   melee strafing + jump-crits + shield timing. The LLM arms it ("defend me", "kill that zombie") and the
   FSM's existing P1 preemption disarms it. **Directly fixes the named Phase-17 failures** (post-hit
   detection, creeper deaths, weak combat). **Verdict: HIGH feasibility, HIGH value.** Cost: medium
   (one new skill module + a tick loop; can crib from `mineflayer-pvp`). No model, no GPU, no tokens
   in-loop. *This is the single highest-leverage item.*

2. **GOAP / skill-dependency-graph planner for tech-tree progression (offline, local).**
   Encode registry actions as operators with preconditions/effects; a local A* planner turns "reach iron
   tier" or "smelt this" into a guaranteed-valid action sequence, with the LLM choosing the goal. Mirrors
   Plan4MC's skill graph and DEPS's selection, minus the RL. **Fixes "progress to iron tier unaided" and
   robust furnace/crafting chains.** **Verdict: HIGH feasibility, MEDIUM-HIGH value.** Cost: medium
   (operator definitions + a small planner). Pairs naturally with the closed registry.

3. **Utility-AI scorer for engage/flee/target-selection.**
   Hand-weighted utility over (mob type, distance, your gear, player exposure, hp/food) to pick the
   action each decision point; replaces brittle if-ladders and offloads a routing decision from the LLM.
   **Verdict: MEDIUM-HIGH feasibility, MEDIUM value.** Cost: low. Start hand-tuned; it's the staging
   ground for an optional learned scorer later.

4. **Feedback-driven replan loop (DEPS-lite) in the brain.**
   Formalize what Sei half-does: surface `lastActionResult` + failure cause, and have the planner
   explain-then-replan rather than retry blindly. **Verdict: MEDIUM feasibility, MEDIUM value.** Cost:
   low-medium; mostly prompt + orchestrator plumbing, no new runtime. Synergizes with Phase 16 memory.

5. **Tiny embeddable supervised model (ONNX MLP) for target selection / aim.**
   Only after #1 and #3 exist and you've logged engagement outcomes to train on. <1 MB, sub-ms CPU,
   ships via onnxruntime-node. **Verdict: MEDIUM feasibility, LOW-to-SPECULATIVE value** — heuristics
   likely match it; justify only with data showing the trade-off surface beats hand weights. Cost:
   high (data pipeline + training) for uncertain gain. **Defer past v0.4.**

6. **Distill a structured-state BC controller from gameplay logs (the "IDM-trick analog").**
   Behavior-clone `(structured features → mineflayer control)` from your own expert/scripted rollouts —
   the only legitimate way VPT's *idea* reaches mineflayer. **Verdict: LOW feasibility for now /
   research spike.** Real but a project unto itself; revisit only if classical control plateaus.

7. **Pixel-based foundation policies (VPT / STEVE-1 / GROOT / DreamerV3) embedded in Sei.**
   **Verdict: DEFER ENTIRELY — control-surface mismatch.** Would require running an actual rendered
   Minecraft Java client + GPU framebuffer + 20 Hz transformer inference alongside the protocol bot, then
   translating mouse/keyboard back to the world: a different, consumer-hostile product. **Do not pursue.**
   Keep only as a reference for *what good play looks like* when designing heuristics and tests.

**Bottom line for the roadmap:** the research frontier (VPT→GROOT→DreamerV3) is impressive and almost
entirely **unusable inside Sei** because Sei chose the protocol control surface — which is a *feature*,
not a limitation: it sidesteps the hard pixel-RL problems those papers solve. The portable insight is the
**hybrid hierarchy**, and the portable *implementation* is the **classical game-AI stack (BT + GOAP +
utility + steering)** running locally on mineflayer's structured state, with the LLM as the high-level
planner it already is. Items 1–3 above are the concrete, in-budget Phase-17 deliverables.
