## 0. CRITICAL: `bot.activateItem()` is broken on MC 1.21+ ‚Äî verify Sei's blast radius first

**Confirmed.** [PrismarineJS/mineflayer#3742](https://github.com/PrismarineJS/mineflayer/issues/3742)
("use_item packet / activateItem not working on Minecraft 1.21.8"), filed
2025-09-20, labels `possible bug` / `Stage1`, **state CLOSED / NOT_PLANNED**.

Root cause (from the issue body): the `use_item` packet definition in
protodef / minecraft-data was not updated for the 1.21 wire format.

- Pre-1.21: `use_item` carried `rotation: vec2f (x, y)`.
- 1.21 officially splits this into separate `yaw: float` + `pitch: float` fields.
- Sending the old shape throws `SizeOf error for undefined` during
  serialization; the packet never reaches the server, so **no server-side
  interaction fires**.

A reproduction in the thread (`bot.setQuickBarSlot(0)` then `bot.activateItem()`)
shows: on 1.20.4 the held item opens its window; on 1.21.8 nothing happens.

**Why "NOT_PLANNED" does not mean "safe to ignore":** the maintainers closed it
without a mineflayer-side fix. The real fix has to land in
`minecraft-data`/`node-minecraft-protocol` packet schemas. **Do not assume your
pinned transitive protocol version has it.** Treat this as a live regression to
re-verify against Sei's actual target build before relying on any right-click
behavior.

**Blast radius inside Sei** ‚Äî every right-click-with-item path routes through
`bot.activateItem()` somewhere:

| Sei behavior | File | Uses | 1.21 risk |
|---|---|---|---|
| `activateItem` action | `behaviors/activate.js` | `bot.activateItem()` directly | **Directly broken** |
| `consumeItem` (eat) | `behaviors/consume.js` | `bot.consume()` | `consume()` internally calls `activateItem()` then waits for the eat ack ‚Äî **at risk** |
| auto-eat plugin | `behaviors/autoEat.js` (`mineflayer-auto-eat ^5`) | plugin calls the same use-item path | **at risk** |
| future: bow, fishing rod, shield raise, bucket, ender pearl, potion | n/a yet | all use `activateItem` | broken until protocol fix |

**This is the single highest-leverage thing to verify for v0.4** ‚Äî eating is
iron-tier-critical and currently flows through the broken path.

**Workarounds, in order of preference:**

1. **Verify-then-pin.** Connect to a real 1.21.x LAN world and call
   `bot.consume()` + `bot.activateItem()` with console open. If you see
   `SizeOf error for undefined`, you have the bug. The fix may already exist in a
   newer `minecraft-data` ‚Äî bump the transitive dep and re-test. ([#3731 "Buckets
   don't work"](https://github.com/PrismarineJS/mineflayer/issues/3731) is the
   same family of regression ‚Äî track both.)
2. **Send the packet by hand** with the correct 1.21 field names. mineflayer
   exposes the raw protocol via `bot._client.write('use_item', { hand: 0, sequence,
   rotation: { pitch, yaw } })` ‚Äî patch the field shape to whatever your protocol
   version actually expects (inspect
   `node_modules/minecraft-data/minecraft-data/data/pc/1.21*/protocol.json`
   ‚Üí `packet_use_item`). Wrap in a `seiActivateItem(bot)` shim so every call site
   (consume, activate, future bow) goes through one patch point.
3. **Prefer block/entity interactions** where possible ‚Äî `bot.activateBlock()`
   and `bot.activateEntity()` use the `block_place` / `interact` packets, NOT
   `use_item`, so they are **unaffected** by #3742. (This is why doors/buttons in
   ¬ß5 are safe and eating is not.)
4. **Fall back to `@nxg-org/mineflayer-auto-eat`** if the PrismarineJS plugin's
   internal use-item path is the blocker ‚Äî the nxg fork sometimes carries its own
   packet shims.

> **Action item:** add a `seiActivateItem()` shim now (even if it just wraps
> `bot.activateItem()` today) so the eventual protocol patch is a one-file change.

## 1. Furnace / smelting

**API:** `bot.openFurnace(furnaceBlock)` ‚Üí `Promise<Furnace>`. `Furnace` extends
`prismarine-windows` `Window`. ([api.md](https://github.com/PrismarineJS/mineflayer/blob/master/docs/api.md))

Methods / props:
- `furnace.putInput(itemType, metadata, count)` ‚Üí `Promise<void>`
- `furnace.putFuel(itemType, metadata, count)` ‚Üí `Promise<void>`
- `furnace.takeOutput()` ‚Üí `Promise<Item>` (also `takeInput()`, `takeFuel()`)
- `furnace.inputItem()` / `furnace.fuelItem()` / `furnace.outputItem()` ‚Üí `Item|null`
- `furnace.fuel` ‚Äî 0..1 fuel remaining; `furnace.progress` ‚Äî 0..1 smelt progress
- event `furnace.on('update', ‚Ä¶)` ‚Äî fires when `fuel`/`progress` change

`itemType` is the **numeric item id** (`mcData.itemsByName[name].id`), `metadata`
is almost always `null` on modern versions.

**Implementation sketch** (`behaviors/smelt.js`):

```js
import mcDataLib from 'minecraft-data'

// fuel preference: avoid burning tools/planks the player needs; prefer
// dedicated fuels. value = items smelted per unit.
const FUEL_PRIORITY = ['coal', 'charcoal', 'coal_block', 'lava_bucket',
  'blaze_rod', 'dried_kelp_block', 'oak_planks' /* ‚Ä¶any *_planks last resort */]

export async function smeltAction(args, bot, config) {
  const signal = config?.signal
  const mcData = mcDataLib(bot.version)

  // 1. resolve a furnace block in reach (reuse resolveBlock from targeting.js)
  const furnaceBlock = await resolveBlock(args.against ?? { block: 'furnace' }, bot)
  if (!furnaceBlock) return 'no furnace in reach'
  // pathfinder: GoalGetToBlock so we end up adjacent, then open
  await goTo(bot, furnaceBlock.position.x, furnaceBlock.position.y,
             furnaceBlock.position.z, 2, config.pathfinder_timeout_ms, signal)

  const furnace = await bot.openFurnace(furnaceBlock)
  try {
    const inputId = mcData.itemsByName[args.input]?.id
    if (inputId == null) return `unknown smeltable: ${args.input}`
    const have = bot.inventory.count(inputId)
    if (have === 0) return `no ${args.input} to smelt`
    const want = Math.min(args.count ?? have, have)

    // 2. fuel math BEFORE inserting input. Each coal smelts 8 items.
    if (!furnace.fuelItem() && (furnace.fuel ?? 0) <= 0) {
      const fuel = pickFuel(bot, mcData, want)        // see FUEL_PRIORITY
      if (!fuel) return `no fuel for smelting (need coal/charcoal)`
      await furnace.putFuel(fuel.type, null, fuel.count)
    }
    await furnace.putInput(inputId, null, want)

    // 3. await completion: poll progress / count output, bounded by wall clock.
    //    ~10s per item in vanilla; cap by config.smelt_timeout_ms (e.g. 200s)
    const out = await awaitSmelt(furnace, want, args.timeout_ms ?? 60000, signal)
    return `smelted ${out} ${args.input}`
  } finally {
    furnace.close()   // ALWAYS close ‚Äî leaking windows breaks later container ops
  }
}
```

**`awaitSmelt`**: resolve when `furnace.outputItem()?.count >= want` OR when input
is exhausted AND output stable for ~1s; race against a wall-clock timeout and the
abort signal. Take output incrementally (`takeOutput()` empties the slot) so a
long smelt that gets aborted still banks partial progress.

**Edge cases / known bugs:**
- **Window leak**: forgetting `furnace.close()` leaves `bot.currentWindow` set;
  the next `openContainer`/`openFurnace` may reject or desync. Always `finally`-close.
- **Smelting is slow** (10s/item, blast furnace 5s). A naive `await` blows past
  any reasonable action timeout ‚Äî you MUST either poll-and-return-early ("started
  smelting 16 iron, ~160s") or run it as a long-runner like `follow`. Recommend:
  return immediately after loading input+fuel ("smelting 8 iron_ore in the
  furnace") and let a later `withdrawSmelted`/`takeOutput` action collect; this
  fits Sei's single-layer loop far better than blocking 160s.
- **Blast furnace / smoker** are separate block types but the same `Furnace`
  class ‚Äî `resolveBlock` should accept `furnace|blast_furnace|smoker`.
- Fuel choice must **never auto-burn the player's good stuff** ‚Äî bound
  `FUEL_PRIORITY` to coal/charcoal/blaze_rod/dried_kelp_block by default; only
  fall to planks if `args.allowWoodFuel`.
- `putInput` rejects if the input slot already holds a different item type.

**Zod arg-shape:**
```js
z.object({
  input: z.string().min(1),                       // loose term or exact id ("iron_ore", "raw_iron")
  fuel: z.string().optional(),                    // override; else auto-pick
  count: z.number().int().min(1).max(64).optional(),
  against: TargetShape.optional(),                // which furnace; else nearest
  allowWoodFuel: z.boolean().optional().default(false),
})
```
Pair with a `collectSmelted` (or reuse `withdrawItem` against a furnace) action
to pull finished output.

## 2. Proactive hostile-mob awareness (threat-safety spine)

This is the biggest behavioral gap. Today Sei is **reactive only** ‚Äî combat
starts on the `entityHurt` event (`behaviors/combat.js`), i.e. *after* damage.
Iron-tier survival needs **pre-damage** awareness: spot the hostile, judge
distance + line-of-sight, and react (warn / flee / pre-emptively engage) before
the first hit. Creepers especially must be handled *before* fuse range.

**API surface (all already in mineflayer core):**
- `bot.entities` ‚Äî map of all tracked entities; iterate `Object.values`.
- entity fields: `e.name` (e.g. `'creeper'`), `e.type` (`'mob'|'hostile'|'player'`),
  `e.position`, `e.velocity`, `e.kind` (`'Hostile mobs'` on some versions).
- `e.position.distanceTo(bot.entity.position)` ‚Äî range.
- Line-of-sight: Sei already has `observers/lineOfSight.js` ‚Äî reuse it; mineflayer
  also ships `bot.canSeeBlock` and you can raycast with
  `bot.world.raycast(from, dir, range)`.
- Reuse the existing `HOSTILE_MOBS` set in `combat.js` (already comprehensive:
  zombie/skeleton/creeper/spider/witch/blaze/‚Ä¶ piglin_brute/zoglin).

**Implementation sketch** ‚Äî a `startThreatWatch(bot, config)` ticker wired in
`fsmWires.js`, emitting a `sei:threat` event at **P0_SAFETY** priority:

```js
const CREEPER_FUSE_RANGE = 3      // creeper ignites within ~3 blocks
const ALERT_RANGE = 16            // start tracking
const DANGER_RANGE = 6            // react now

export function startThreatWatch(bot, config) {
  const throttle = createThrottle(config?.threat_throttle_ms ?? 2500) // reuse debounce.js
  setInterval(() => {
    const me = bot.entity?.position
    if (!me) return
    let nearest = null, nearestD = Infinity, creeperClose = null
    for (const e of Object.values(bot.entities)) {
      if (!HOSTILE_MOBS.has(e?.name) || !e.position) continue
      const d = e.position.distanceTo(me)
      if (d > ALERT_RANGE) continue
      // LOS gate: a hostile behind a wall is not (yet) a threat ‚Äî cuts false alarms
      const visible = hasLineOfSight(bot, e)        // observers/lineOfSight.js
      if (e.name === 'creeper' && d <= CREEPER_FUSE_RANGE + 2) creeperClose = e
      if (visible && d < nearestD) { nearestD = d; nearest = e }
    }
    if (creeperClose) {
      bot.emit('sei:threat', { kind: 'creeper', entity: creeperClose,
        distance: creeperClose.position.distanceTo(me), action: 'flee' })
    } else if (nearest && nearestD <= DANGER_RANGE) {
      throttle.throttle('threat', { kind: nearest.name, entity: nearest,
        distance: nearestD }, p => bot.emit('sei:threat', p))
    }
  }, config?.threat_tick_ms ?? 500)
}
```

**Reactions (dynamic retreat):**
- **Creeper within fuse range** ‚Üí *flee*, don't fight. Compute a retreat vector
  `away = me.minus(creeper.position).normalize()`, set
  `bot.pathfinder.setGoal(new GoalXZ(me.x + away.x*8, me.z + away.z*8))` (or
  `GoalInvert(new GoalNear(creeper, 8))`). Skeletons/ranged ‚Üí break LOS behind a
  block. Melee mobs ‚Üí either fight (if armed + healthy) or pillar up / wall off.
- **Snapshot surfacing**: add a `threats:` line to the per-turn snapshot
  (`observers/snapshot.js`) so the LLM *sees* `threats: creeper #3 @4m (LOS),
  zombie #5 @11m` every turn and can pre-empt verbally ("creeper! back up!").
  This is cheaper and more in-character than a hard autopilot, and matches Sei's
  "invite/involve the player" design note.

**Edge cases / bugs:**
- **Phantom entities**: `bot.entities` retains stale entries briefly after death;
  guard with `e.isValid !== false` and a position freshness check.
- **Spider/zombie on hard difficulty** can be neutral in daylight ‚Äî `e.metadata`
  is unreliable for hostility; the name-set approach is more robust.
- **LOS cost**: a raycast per hostile per 500ms tick is fine for <~20 entities;
  cap the scan to the nearest N by cheap distance first, raycast only those.
- **Don't preempt-storm.** `combat.js` already learned this lesson
  (`_seiAttackThrottle`, windowMs 3500): one reaction per window, leading-edge.
  Mirror it for `sei:threat`.
- **Iron Golems / wolves / players** are not in `HOSTILE_MOBS` ‚Äî keep it that way
  (no auto-aggro on neutrals; matches the Auto-PvP out-of-scope rule).

**Zod arg-shape** (the LLM-callable side ‚Äî a `fleeFrom` action):
```js
z.object({
  target: EntityTarget.optional(),     // "#N" handle or mob name; else nearest hostile
  distance: z.number().min(4).max(32).default(12),
})
```

## 3. Combat (`mineflayer-pvp ^1.3.2`)

**Plugin: `mineflayer-pvp`** (PrismarineJS, actively accepting PRs).
([repo](https://github.com/PrismarineJS/mineflayer-pvp) ¬∑
[api.md](https://github.com/PrismarineJS/mineflayer-pvp/blob/master/docs/api.md))

Full API:
- `bot.pvp.attack(entity)` ‚Äî start attacking; auto-pathfinds into range and swings.
- `bot.pvp.stop()` ‚Äî stop (lets pathfinder finish current move).
- `bot.pvp.forceStop()` ‚Äî stop + force pathfinder halt.
- `bot.pvp.target` ‚Äî read-only current target entity.
- `bot.pvp.movements` ‚Äî the `Movements` config used while pursuing.
- `bot.pvp.followRange` ‚Äî how close to approach.
- `bot.pvp.viewDistance` ‚Äî distance at which it loses interest.
- `bot.pvp.attackRange` ‚Äî max distance it will still swing from.
- `bot.pvp.meleeAttackRate` ‚Äî the solver that decides attack cadence (handles
  the **1.9+ attack-cooldown / crit timing** ‚Äî this is what you get "for free"
  vs. Sei's current hand-rolled 250ms `setInterval` in `combat.js`).
- events: `bot.on('startedAttacking')`, `bot.on('stoppedAttacking')`,
  `bot.on('attackedTarget')` (fires each melee swing ‚Äî good hook for "took a
  swing" speech / weapon-wear tracking).

**What mineflayer-pvp does NOT do:** weapon selection, shield raising, kiting,
or bow logic. You wire those around it:

- **Weapon switch**: before `attack`, pick best sword/axe via `mineflayer-tool`
  (¬ß8) or a manual scan of `bot.inventory.items()` by attack damage, then
  `bot.equip(weapon, 'hand')`.
- **Shield**: equip `shield` to `'off-hand'`, then **raise it with
  `bot.activateItem(true)`** (the `true` = offhand) ‚Äî **but this is the broken
  use_item path (¬ß0)** on 1.21. Until the protocol fix, shield blocking is
  effectively unavailable; rely on armor + kiting instead. Track this as blocked
  on #3742.
- **Kiting / strafing vs. melee**: between swings, strafe with control states ‚Äî
  `bot.setControlState('left'|'right'|'back', true)` pulsed, or set a
  `GoalFollow(target, idealRange)` that keeps you at sword reach while circling.
  For ranged enemies (skeleton), close distance fast; for slow melee (zombie),
  hold reach and back-pedal.
- **Flee-vs-fight decision**: gate on `bot.health`, armor points, and whether a
  weapon exists. Below ~6 health or unarmed ‚Üí route to ¬ß2 `fleeFrom` instead of
  `bot.pvp.attack`.
- **Bow / ranged**: `bot.hawkEye` plugin
  ([`minecrafthawkeye`](https://www.npmjs.com/package/minecrafthawkeye)) provides
  aim-prediction: `bot.hawkEye.oneShot(target, 'bow')` /
  `autoAttack`. It draws the bow via the use-item path too ‚Äî **also blocked by
  #3742** for the draw step; verify before committing. Alternative:
  `@nxg-org/mineflayer-custom-pvp` bundles `BowPvP` (projectile prediction) +
  `SwordPvP` (criticals, strafing, **shield management**) and may carry its own
  packet handling.

**Migration recommendation for Sei:** replace the hand-rolled `startAttacking`
`setInterval(‚Ä¶250ms)` loop in `combat.js` with `bot.pvp.attack(target)`. You keep
your `resolveAttacker`/throttle/`entityHurt` logic for *deciding who to fight*,
but hand the *how to fight* (range management, cooldown-correct swing timing,
crits) to the plugin. This directly fixes the "Sui frozen in fights" class of
bugs because pvp's `meleeAttackRate` won't fire faster than the server accepts.

**Edge cases / bugs:**
- mineflayer-pvp drives the pathfinder ‚Äî if Sei's `goTo`/`follow` also drive it,
  they fight over the goal. Establish a single owner: when `bot.pvp.target` is
  set, suppress the follow tick (Sei's `combat.js` already `stopFollow()`s ‚Äî keep
  that discipline).
- Knockback produces transient non-finite velocity (Sei already guards this in
  `combat.js`; #3887 "bot freezes mid-air after knockback on 1.21.11" is a live
  related issue ‚Äî keep the finite-guards).
- pvp won't switch targets mid-fight; call `attack(newTarget)` to retarget.

**Zod arg-shape** (`fight` action, complementing the existing `attackEntity`):
```js
z.object({
  target: EntityTarget.optional(),   // "#N" or mob name; else nearest hostile in view
  entity: z.string().optional(),
  weapon: z.string().optional(),     // override; else auto best melee
}).refine(a => a.target || a.entity, { message: 'need target or entity' })
```
Keep the existing `attackEntity` (discrete N-swing) for scripted single hits;
add `fight`/`stopFighting` as the sustained-combat pair backed by pvp.

## 4. Reading signs

**API:** `block.getSignText()` ‚Äî returns `[frontText, backText]` (array of
strings; each may contain `\n`-joined lines). This is the stable accessor ‚Äî use
it **instead of** poking `block.blockEntity` NBT directly (which is version-
fragile). ([api.md](https://github.com/PrismarineJS/mineflayer/blob/master/docs/api.md);
older `block.signText` was front-only and unreliable ‚Äî
[#675](https://github.com/PrismarineJS/mineflayer/issues/675),
[#1489](https://github.com/PrismarineJS/mineflayer/issues/1489).)

```js
const block = bot.blockAt(new Vec3(x, y, z))
const [front, back] = block.getSignText()   // e.g. ["Front\nHello", "Back\n..."]
```

Hanging signs (1.20+) share the same `front_text`/`back_text` block-entity schema
([Minecraft Wiki ‚Äî Sign](https://minecraft.wiki/w/Sign)), so `getSignText()`
covers standing signs, wall signs, and hanging signs uniformly.

**Implementation sketch** (`readSign` action): resolve the block via the existing
`resolveBlock(TargetShape)`, confirm `block.name.includes('sign')`, call
`getSignText()`, return both faces (drop empty back). If the sign isn't in a
loaded chunk, the block-entity may be `null` ‚Äî `getSignText()` returns empty
strings; surface "sign is blank or not loaded, move closer."

**Edge cases:**
- Chunk must be loaded with block-entity data ‚Äî freshly-rendered chunks can lag a
  tick; a "move within ~4 blocks then read" pattern is most reliable.
- Color/format codes (¬ß sequences / JSON text components) may appear; strip with
  a `replace(/¬ß./g, '')` (or parse the message JSON if you want clean text).
- Empty faces return `''` not `null`.

**Zod arg-shape:**
```js
TargetShape   // reuse: { block?, target?, x/y/z, maxDistance } ‚Äî resolves the sign block
```
Could also fold sign text into the `find`/snapshot pipeline so nearby signs are
surfaced automatically.

## 5. Doors / gates / trapdoors / buttons / levers

**API:** `bot.activateBlock(block, direction?, cursorPos?)` ‚Üí `Promise<void>`.
Uses the `block_place`/interact packet (NOT `use_item`), so it is **unaffected
by #3742**. ([api.md](https://github.com/PrismarineJS/mineflayer/blob/master/docs/api.md))
- `direction` defaults to `new Vec3(0, 1, 0)`; `cursorPos` defaults to block
  center `(0.5, 0.5, 0.5)`.
- Works for doors, fence gates, trapdoors, buttons, levers, note blocks.

**Implementation sketch** (`useBlock`/`toggleBlock` action):
```js
const block = await resolveBlock(args.against, bot)
if (!block) return 'no such block in reach'
// must be within reach (~4.5 blocks); pathfind closer if needed
await bot.activateBlock(block)
return `toggled ${block.name}`
```
A door's open/closed state is in `block.getProperties().open` (or
`block._properties`) ‚Äî read it to report or to ensure idempotent open.

**Passing through doors during navigation:** `mineflayer-pathfinder`'s
`Movements` has **`canOpenDoors` (default `false`)** which the readme describes as
"Enable feature to open Fence Gates." Set `movements.canOpenDoors = true` so the
pathfinder opens gates en route. **Caveat:** historically this option is partial
(gates yes, wooden doors inconsistently; iron doors need a button/lever and
pathfinder won't press it). For reliable door traversal, an explicit `useBlock`
action the LLM can call is more dependable than relying on pathfinder.

**Edge cases:**
- **Iron doors/trapdoors** have no hand interaction ‚Äî they need a button/lever
  (`activateBlock` the lever, not the door).
- Buttons/levers: `activateBlock` toggles; buttons auto-release after their tick.
- Reach: `activateBlock` fails silently if out of reach ‚Äî pathfind to within ~3
  blocks first (`GoalGetToBlock`).

**Zod arg-shape:**
```js
z.object({
  against: TargetShape,                          // the door/lever/button
  // optional: ensure a target state rather than blind toggle
  state: z.enum(['open', 'close', 'toggle']).optional().default('toggle'),
})
```

## 6. Precise placement & simple structures

**API:** `bot.placeBlock(referenceBlock, faceVector)` ‚Üí `Promise<void>`. Sei
already wraps this well in `behaviors/place.js` (reference-block + face-vector,
near-bot autoplacement fallback, FACE_PRIORITY ordering) and `behaviors/build.js`
(cuboid, 256-cell cap). The placement model is correct; the gaps are
**reliability** and **structure recipes**.

**Common `placeBlock` failure modes (and Sei's current handling):**
- **No solid reference face** ‚Äî placing into open air with no neighbor. Sei's
  `refFaceForCell`/`autoPlacementNearBot` already handle this; it's the #1 cause
  of the historical "placeBlock timed out 5√ó then used build()" bug noted in
  `place.js`.
- **Bot is standing in the target cell** ‚Äî server rejects. Need to step aside
  first (pathfinder to an adjacent cell) or place at head-ring.
- **`placeBlock` hangs** ‚Äî mineflayer waits for a `blockUpdate` ack that may
  never come if the placement was illegal; Sei already races a 4s timeout. Keep
  it.
- **Must look at the face** ‚Äî mineflayer auto-looks, but if `bot.look` is mid-
  flight from another action it can mis-aim; serialize placements.
- **Need the block in hand** ‚Äî Sei equips first. Good.

**Structure recipes worth adding** (built on `build`/`placeBlock`):
- **Pillar up** (`allow1by1towers` analog): jump + place beneath feet, N times ‚Äî
  `bot.setControlState('jump', true)` then `placeBlock(blockBelow, (0,1,0))` at
  the apex. Reliable emergency escape from mobs (see ¬ß2).
- **Emergency shelter / wall-off**: place a 1-wide, 2-high wall on the threat
  side, or seal a 1√ó1√ó2 pocket (place 4 walls + ceiling around self). A
  `panicShelter` action is high-value for threat-safety.
- **Bridge** (`scaffold`): place blocks forward while sneaking at the edge ‚Äî
  pathfinder does this when `Movements.scafoldingBlocks` is populated (note the
  upstream misspelling `scafoldingBlocks`).

Sei's `build` cuboid already covers walls/floors/boxes within the 256-cell cap;
the missing primitive is the **self-relative emergency** (pillar/shelter) where
coords are computed from `bot.entity.position` at call time, not pre-specified.

**Zod arg-shape** (new `pillarUp` / `shelter`):
```js
// pillarUp
z.object({ block: z.string(), height: z.number().int().min(1).max(8).default(3) })
// shelter (seal self in)
z.object({ block: z.string().optional() /* else best available cobble/dirt */ })
```
Harden existing `placeBlock`: add a "step aside if standing in target cell" guard.

## 7. Navigation (`mineflayer-pathfinder ^2.4.5`)

Sei already uses pathfinder with a 12s wall-clock timeout (Pitfall #1) and
abort-signal plumbing ‚Äî the architecture is right. The gap is **`Movements`
tuning** and **goal-class variety**.
([readme](https://github.com/PrismarineJS/mineflayer-pathfinder))

**`Movements` tunables (with defaults) ‚Äî set these on the shared movements
instance:**

| Property | Default | Use for Sei |
|---|---|---|
| `canDig` | `true` | leave on for mining; consider **off near the player's base** to avoid griefing terrain |
| `digCost` | `1` | raise to bias toward walking around vs. tunneling |
| `placeCost` | `1` | raise to discourage scaffold spam |
| `allow1by1towers` | `true` | pillaring ‚Äî keep on |
| `allowFreeMotion` | `false` | straight-line walk when terrain allows |
| `allowParkour` | `true` | gap jumps ‚Äî keep on (but a source of falls) |
| `allowSprinting` | `true` | faster follow |
| `allowEntityDetection` | `true` | route around mobs/players (good for threat-safety) |
| `entityCost` | `1` | raise to give mobs a wider berth |
| `canOpenDoors` | `false` | set **true** for door traversal (¬ß5) |
| `maxDropDown` | `4` | lower (e.g. 3) to reduce fall damage |
| `infiniteLiquidDropdownDistance` | `true` | drops into water ignore maxDropDown |
| `liquidCost` | `1` | **raise high** to avoid swimming through lava-adjacent water |
| `dontCreateFlow` | `true` | don't break blocks touching liquid ‚Äî prevents lava/water floods |
| `dontMineUnderFallingBlock` | `true` | gravel/sand safety ‚Äî keep on |
| `blocksToAvoid` | set | **add lava, fire, magma_block, cactus, sweet_berry_bush, powder_snow** |
| `blocksCantBreak` | set | add player-valuable blocks (chests, beds) to avoid griefing |
| `scafoldingBlocks` | `[dirt,cobble‚Ä¶]` | populate with cheap blocks for bridging (note misspelling) |

**Lava avoidance** is not a single flag ‚Äî combine `blocksToAvoid` (lava/magma),
high `liquidCost`, and `dontCreateFlow`. Pathfinder has **no innate "fear of
falling into lava while parkouring"**, so also consider lowering `allowParkour`
near known lava, or gating it on a vision check.

**Goal classes:**
- `GoalNear(x,y,z,range)` ‚Äî Sei's default; stop within `range`.
- `GoalBlock(x,y,z)` ‚Äî stand *in* that exact block (foot level).
- `GoalGetToBlock(x,y,z)` ‚Äî stand *adjacent* (correct for chests/furnaces/doors ‚Äî
  use this for ¬ß1 furnace and ¬ß5 doors so you don't try to stand inside them).
- `GoalFollow(entity,range)` ‚Äî Sei's follow tick should use this directly.
- `GoalXZ(x,z)` ‚Äî long-range, Y-agnostic exploration / fleeing.
- `GoalY(y)` ‚Äî dig to a depth (mining to diamond level y=-59).
- `GoalInvert(goal)` ‚Äî **flee**: `GoalInvert(new GoalNear(creeper, 8))` (¬ß2).
- `GoalCompositeAny([‚Ä¶])` ‚Äî "reach any of these" (nearest of several ores).

**Vision-assisted nav:** Sei renders frames (`prismarine-viewer`, `visualize`/
`look`). When a render exists, the VLM can pick a heading the pathfinder then
executes ‚Äî but pathfinder already has full world geometry, so vision's real value
is **semantic** (avoid that lava lake, head toward the village) not geometric.
Keep vision for high-level "which way" and let pathfinder own the metric path.

**Edge cases / bugs:**
- **Silent hangs** ‚Äî already guarded by Sei's 12s timeout (mandatory; Pitfall #1).
- **`GoalNear` with range 0** thrashes (Sei already bumps player-coords to range
  2 ‚Äî see `isCoordsAtKnownPlayer`).
- Goal churn from competing owners (follow vs. pvp vs. goTo) ‚Äî single-owner
  discipline (see ¬ß3).
- Knockback / mid-air freeze [#3887](https://github.com/PrismarineJS/mineflayer/issues/3887)
  on 1.21.11 ‚Äî keep finite-guards.

## 9. Classic / pre-LLM bot tech worth copying

- **Baritone** (the canonical Minecraft pathfinding bot):
  - **Goal system** ‚Äî Baritone's `Goal` interface (`GoalBlock`, `GoalNear`,
    `GoalXZ`, `GoalYLevel`, `GoalGetToBlock`, `GoalComposite`, `GoalInverted`,
    `GoalRunAway`) is *exactly* the vocabulary mineflayer-pathfinder copied.
    **`GoalRunAway(blocks, ‚Ä¶positions)` is the missing primitive for ¬ß2 flee** ‚Äî
    replicate it (= `GoalInvert(GoalNear)` with a min distance).
  - **Process model** ‚Äî Baritone layers independent "processes" (MineProcess,
    FollowProcess, BuilderProcess, ExploreProcess) under a priority arbiter; the
    highest-priority process that *wants control* owns the path. **This is
    structurally identical to Sei's P0‚ÄìP3 FSM** ‚Äî validate that combat/threat
    (P0/P1) always preempt mine/follow (P2). Baritone got right: *one* path
    owner, priority-arbitrated, never two goals at once.
  - **Pathfinding** ‚Äî A* with movement-cost tuning (break/place/fall costs) ‚Äî the
    same knobs as `Movements` (¬ß7). Baritone's lesson: **cost-tune, don't
    hard-code rules** (e.g. high lava cost beats a "never go near lava" boolean).
- **MineRL / MineDojo** ‚Äî skill decomposition for learning agents:
  - Decompose long tasks into a **tech-tree of subgoals** (log‚Üíplanks‚Üítable‚Üí
    wooden_pickaxe‚Üístone‚Üífurnace‚Üíiron). MineDojo's task suite is literally an
    iron-tier progression ladder. **Copy the dependency graph as the LLM's
    crafting target list** ‚Äî Sei's `observers/progression.js` already gestures at
    this; the iron-tier spine is: wood ‚Üí tools ‚Üí stone tools ‚Üí furnace ‚Üí smelt
    iron ‚Üí iron tools/armor.
  - **Primitive granularity**: MineDojo/Voyager expose mid-level skills
    (`mineBlock`, `craftItem`, `smeltItem`, `combatEntity`, `placeBlock`) ‚Äî *not*
    raw key presses. Sei's closed registry is already at this altitude; the gaps
    in this report (smelt, fight, flee, useBlock, readSign) are exactly the
    MineDojo/Voyager primitive set. Voyager's skill list is a good completeness
    checklist.
  - What they got right: **idempotent, verifiable skills** ‚Äî each skill checks
    its own success post-condition (did the item appear in inventory?) and
    returns a result string. Sei already does this (result strings, snapshot
    readback). Keep it.

## Concrete adapter additions for Sei

> Prioritized to reach the **iron-tier spine** (wood‚Üístone‚Üífurnace‚Üíiron tools/
> armor) + **threat-safety**. Names follow Sei's camelCase action convention;
> `TargetShape`/`EntityTarget`/`Vec3Shape` are the existing shared shapes in
> `registry.js`.

### P0 ‚Äî Threat-safety spine (highest leverage, mostly new wiring)

1. **`startThreatWatch` ticker (not an LLM action ‚Äî a wire)** in `fsmWires.js`,
   emits `sei:threat` at P0; add `threats:` line to `observers/snapshot.js`.
   Reuse `HOSTILE_MOBS` + `lineOfSight.js` + `debounce.js` throttle. *This is the
   single most important addition ‚Äî Sei is blind to danger until hit.*
2. **`fleeFrom`** ‚Äî `z.object({ target: EntityTarget.optional(), distance:
   z.number().min(4).max(32).default(12) })`. Impl: `GoalInvert(new
   GoalNear(threat, distance))` / `GoalRunAway`-style retreat; special-case
   creeper within fuse range (immediate, no LOS gate).
3. **Migrate combat to `mineflayer-pvp`** ‚Äî keep `resolveAttacker`/throttle/
   `entityHurt` in `combat.js`, replace the 250ms `setInterval` swing loop with
   `bot.pvp.attack/stop`. Fixes "frozen in fights"; gets correct cooldown/crit
   timing free. Add **`fight`** (`z.object({ target/entity, weapon? })`) +
   **`stopFighting`** (`z.object({})`) actions; auto-equip best melee weapon
   before attacking.

### P1 ‚Äî Iron-tier crafting spine

4. **`smelt`** ‚Äî `z.object({ input, fuel?, count?, against?: TargetShape,
   allowWoodFuel: default false })`. `bot.openFurnace` ‚Üí `putFuel`/`putInput`,
   **return early** ("smelting N‚Ä¶"), always `finally` close. The keystone for
   iron: no smelt = no iron tools/armor.
5. **`collectSmelted` / extend `withdrawItem`** to pull furnace output
   (`takeOutput`). Pairs with `smelt`.
6. **Adopt `mineflayer-tool`** and harden `dig`/`gather`/`mineVein` to call
   `bot.tool.equipForBlock(block, { requireHarvest: true })` before mining ‚Äî
   ensures iron ore is mined with a stone+ pickaxe (else no drop) and saves
   durability. *Iron ore needs ‚â•stone pickaxe or it drops nothing ‚Äî this is a
   silent iron-tier blocker today.*

### P2 ‚Äî Interaction & survivability

7. **`useBlock`** ‚Äî `z.object({ against: TargetShape, state:
   z.enum(['open','close','toggle']).default('toggle') })`. `bot.activateBlock`
   (unaffected by #3742). Also set `Movements.canOpenDoors = true`.
8. **`readSign`** ‚Äî `TargetShape`. `block.getSignText()` ‚Üí both faces. Optionally
   auto-surface nearby signs in the snapshot.
9. **`pillarUp`** ‚Äî `z.object({ block, height: 1..8 default 3 })` and
   **`shelter`** ‚Äî `z.object({ block? })`. Self-relative emergency builds;
   compose with `fleeFrom` (pillar/wall when can't outrun).
10. **Adopt `mineflayer-armor-manager`** ‚Äî auto-equip best armor on pickup;
    raises effective HP for free, supports threat-safety.

### Harden existing actions

- **`activateItem` + `consumeItem` + auto-eat (¬ß0, BLOCKING):** add a
  `seiActivateItem()` shim; verify eating works on the real 1.21.x target. If
  broken, hand-write the `use_item` packet with 1.21 `yaw/pitch` fields or bump
  `minecraft-data`. **Eating is iron-tier-critical and currently on the broken
  path ‚Äî fix or confirm before anything else.**
- **`placeBlock`:** add "step aside if standing in target cell" guard; already
  has good face-fallback.
- **`goTo`/`follow`:** use `GoalGetToBlock` for container/furnace/door
  approaches (don't try to stand inside them); ensure single goal-owner with pvp.
- **`Movements` tuning:** add lava/fire/cactus/powder_snow to `blocksToAvoid`,
  raise `liquidCost`, lower `maxDropDown` to ~3, populate `scafoldingBlocks`,
  consider `canDig=false` near the player's base.

### Explicitly do NOT adopt
- `mineflayer-statemachine` (stale ~3 yrs; Sei's `brain/fsm.js` is better).
- Auto-PvP against players (out-of-scope per REQUIREMENTS; threat watch stays
  mob-only ‚Äî players never auto-aggro'd).
- `mineflayer-web-inventory` in shipped builds (dev debug only).
