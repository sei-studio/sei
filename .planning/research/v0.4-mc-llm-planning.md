# LLM + Symbolic Planning for Minecraft Agents — Research for Sei v0.4

**Purpose.** Engineering roadmap research for Sei, an LLM Minecraft companion: a
mineflayer (Node) bot in a forked process, **single-layer loop** (ONE LLM call
reasons *and* emits tool calls — no separate planner/dispatcher), default
Haiku-class model, **closed Zod-typed action registry** (the model CANNOT
generate or execute code, cannot hot-load skills), ~18 world actions today,
running on a consumer desktop. Every finding below is filtered through the
question: *what transfers to a closed-registry single-layer bot?*

**Date of research:** 2026-06-25. Citations include repos, papers, and dates.

---

## 0. TL;DR for the impatient

The famous Minecraft-agent papers (Voyager, GITM, DEPS, JARVIS-1, Plan4MC,
Odyssey, Optimus) almost all assume one of two things Sei does **not** have:
(a) the LLM **writes JavaScript skills** (Voyager, Mindcraft), or (b) there is a
**separate planner module** plus a **learned RL controller** (DEPS, Plan4MC,
GITM, Optimus, JARVIS-1). Sei has neither: it has a fixed action vocabulary and a
single LLM turn.

So the transferable value is **not their control stacks** — it is their **data
structures and conventions**:

1. **A Minecraft progression/dependency graph as static data** (Plan4MC's "skill
   graph", GITM's recursive sub-goal tree, the crafting recipe DAG) that the
   single LLM call *consults* as context, instead of a planner that *executes* it.
2. **Self-verification as a tool-result convention** (Voyager's critic, DEPS's
   "Describe + Explain") collapsed into Sei's already-existing
   `last_action_result:` / `craftable:` / inventory-delta snapshot lines — i.e.
   make the environment, not a second LLM, do the verifying.
3. **An environment-aware curriculum / "what's next" hint** (Voyager's
   curriculum queries game state before proposing the next goal) reduced to a
   single computed "next milestone" line in the snapshot.

These three fit the single-layer, closed-registry, Haiku-class budget. The rest
of this document derives them.

## 1. Sei today (grounding the constraints)

From the codebase (`src/bot/adapter/minecraft/registry.js`,
`src/bot/config.js`, `src/bot/adapter/minecraft/observers/snapshot.js`):

- **Action set (registered tools), ~21 today:** `goTo`, `dig`, `explore`,
  `find`, `gather`, `placeBlock`, `build`, `equip`, `craft`, `attackEntity`,
  `follow`, `unfollow`, `consumeItem`, `look` (VLM-gated), `dropItem`,
  `activateItem`, `sleep`, `openContainer`, `depositItem`, `withdrawItem`, plus
  brain tools `remember`/`forget`/`say`/`end_loop`/`setGoal`. All are **Zod-typed
  closed schemas**.
- **Snapshot already surfaces** (the key insight for everything below): `pos`,
  `biome`/`time`, `hp`/`food`/`xp`, `holding`, `inventory (K/36 slots)`,
  **`craftable:`** (what can be made *right now*, table-gated), `terrain at
  feet`, `nearby blocks` with `#N` handles, `nearby entities`, `follow_target`,
  `owner` whereabouts, **`last_action_result:`**, a loud **`*** INVENTORY JUST
  CHANGED ***`** delta line, and **`recent_events:`** (deaths, hp loss). This is
  already a structured world-state feed — exactly the substrate the planning
  papers feed their planners.
- **Loop:** event-sourced FSM, single LLM call per turn, `iteration_cap` 300
  backstop, Haiku 4.5 default, 12s timeout. `HEARTBEAT.md` holds active goals
  across turns (`setGoal`/`clearGoal`); `MEMORY.md` is LLM-written long-term
  memory.
- **Gaps named in the brief:** furnace/smelting, proactive mob awareness,
  combat, signs, doors.

The leverage point is therefore **the prompt context + the snapshot + a few new
closed actions**, never a new control architecture.

## 2. System-by-system deep dive

For each: architecture · action/skill abstraction · long-horizon planning ·
self-verification/reflection · **what transfers to Sei**.

### 2.1 Voyager (NVIDIA / Caltech / Stanford et al.) — May 2023

- **Paper:** "Voyager: An Open-Ended Embodied Agent with Large Language Models",
  arXiv:2305.16291 (2023-05-25, NeurIPS 2023 workshop / spotlight).
  <https://arxiv.org/abs/2305.16291>
- **Repo:** <https://github.com/MineDojo/Voyager> · site
  <https://voyager.minedojo.org/>. Built **on Mineflayer** (relevant to Sei).
- **Architecture (three modules around GPT-4, blackbox, no fine-tuning):**
  1. **Automatic curriculum** — proposes the next task to maximize exploration.
     Crucially **environment-aware**: it queries current game state (inventory,
     nearby, biome, completed tasks) before proposing, so it "never attempts
     goals its current loadout makes impossible."
  2. **Skill library** — an ever-growing store of **executable JavaScript**
     (mineflayer code) keyed by a natural-language/embedding description;
     retrieved by query and composed.
  3. **Iterative prompting** — generates code, runs it, feeds back **environment
     feedback + execution errors + self-verification** until it passes.
- **Self-verification:** a **separate GPT-4 critic agent** is handed the current
  state + the task and asked "did the program achieve this? if not, critique
  it." Described in the paper as *more comprehensive than self-reflection*
  because it both checks success and suggests the fix. No hand-coded success
  checkers.
- **What transfers / what does NOT:** The **skill library = generated code** is
  exactly what Sei's closed registry forbids — do **not** copy it. What *does*
  transfer: (a) the **environment-aware curriculum** idea — compute the next
  reasonable goal *from current state* and surface it; (b) the **self-verify via
  state, not vibes** idea — but collapse the second GPT-4 call into a
  deterministic check against the snapshot (Sei can't afford a second Haiku call
  per action on desktop latency/credits). Voyager is the canonical "why a closed
  registry is different" reference.

### 2.2 Ghost in the Minecraft / GITM (OpenGVLab / Shanghai AI Lab) — May 2023

- **Paper:** arXiv:2305.17144 (2023-05-26). Repo:
  <https://github.com/OpenGVLab/GITM>
- **Architecture:** **LLM Decomposer → LLM Planner → LLM Interface.** The
  Decomposer recursively breaks a goal into a tree of sub-goals; the Planner
  emits **structured actions** (text); the Interface maps structured actions to
  keyboard/mouse. Backed by **text-based knowledge + memory** (reference text
  about recipes/tech tree, plus memory of past plans).
- **Action/skill abstraction:** **"structured actions"** — a *fixed, closed
  vocabulary* of textual actions (e.g. *craft*, *mine*, *equip*) that the LLM
  composes. This is the **closest match to Sei's closed registry** of any paper
  here: GITM deliberately rejects free-form code and raw control in favor of a
  small set of structured actions, exactly Sei's posture.
- **Long-horizon planning:** **recursive goal decomposition into a sub-goal
  tree**, ordered by a reference tech-tree. First agent to obtain **all
  Overworld tech-tree items**; **+47.5% absolute** success on ObtainDiamond over
  prior RL. Runs on **CPU only** (32 cores, no GPU) — it is a pure
  LLM+knowledge+memory system, no learned controller.
- **Self-verification/reflection:** structured-action feedback + memory of which
  decompositions worked; replans on failure.
- **What transfers:** **The strongest single reference for Sei.** Its
  "structured actions over a tech-tree, recursive sub-goal decomposition,
  text-knowledge of recipes, all on CPU" *is* Sei's design space. Adopt: (a) ship
  a **text/data tech-tree** the model can consult; (b) frame multi-step goals as
  **recursive decomposition into known tool calls** (Sei does the recursion
  across *turns* via `HEARTBEAT.md`, not in one call). GITM proves a
  closed-vocabulary LLM agent can finish the whole Overworld tree without code
  generation or RL.

### 2.3 DEPS — Describe, Explain, Plan and Select (PKU / BIGAI) — Feb 2023

- **Paper:** "Describe, Explain, Plan and Select: Interactive Planning with LLMs
  Enables Open-World Multi-Task Agents", arXiv:2302.01560 (2023-02-03, NeurIPS
  2023). <https://arxiv.org/abs/2302.01560>
- **Architecture:** event-triggered **Descriptor** → LLM as **Explainer +
  Planner** → **goal Selector** → goal-conditioned controller.
  - **Describe:** convert execution outcome into text the LLM can read.
  - **Explain:** when a plan fails, the LLM **self-explains why** (which sub-goal
    was infeasible) before replanning — this is the key error-correction trick.
  - **Plan:** produce an ordered sub-goal sequence.
  - **Select:** a **trainable horizon-predictor** ranks *parallel* candidate
    sub-goals by estimated steps-to-complete, so the plan picks the **closest
    achievable** sub-goal first (e.g. "gather wood you can reach now" before "go
    find iron"). This addresses plan *efficiency*, not just correctness.
- **Long-horizon:** first **zero-shot** agent to robustly do **70+** Minecraft
  tasks; ~2× prior performance.
- **Self-verification/reflection:** the **Describe→Explain** loop *is* the
  self-correction mechanism — it closes execution feedback back into the planner
  with an explicit "explain the failure" step before retrying.
- **What transfers:** Two ideas, both prompt-level for Sei:
  1. **"Explain before replan"** — when an action fails, don't just retry; have
     the model state *why* it failed and what precondition is missing. In a
     single-layer loop this becomes a **tool-result that names the missing
     precondition** (see §4) so the model's next turn is informed.
  2. **The Selector's "pick the nearest-achievable sub-goal"** — for Sei,
     approximate with a **cheap heuristic ordering** in the progression data
     (prefer sub-goals whose inputs are already in inventory / nearby), surfaced
     as the "next milestone" hint. Sei cannot train a horizon predictor, but it
     can *sort candidate next-steps by how many of their inputs are already
     satisfied* — a static analog.

### 2.4 JARVIS-1 (PKU / UCLA / BIGAI — CraftJarvis) — Nov 2023

- **Paper:** "JARVIS-1: Open-World Multi-task Agents with Memory-Augmented
  Multimodal Language Models", arXiv:2311.05997 (2023-11-10). Repo:
  <https://github.com/CraftJarvis/JARVIS-1>
- **Architecture:** multimodal LLM (sees pixels + reads instructions) → plans →
  embodied control via a learned controller (STEVE-1-class). Key novelty is a
  **multimodal memory**: stores `(situation, plan, outcome)` triples and
  **retrieves successful past plans** for similar situations to seed new
  planning ("self-instruct + self-improve" over a growing experience store).
- **Action/skill abstraction:** high-level sub-goals dispatched to a low-level
  controller; not a closed text registry.
- **Long-horizon:** 200+ tasks; **5× more reliable** on the diamond-pickaxe
  long-horizon task than prior SOTA. The reliability gain comes mostly from
  **memory retrieval of plans that previously worked**.
- **Self-verification/reflection:** self-check + memory; replans using retrieved
  prior successes.
- **What transfers:** the **"remember plans that worked and retrieve them"**
  pattern maps directly onto Sei's existing `MEMORY.md` and per-world memory
  segmentation. Concretely: when Sei completes a multi-step goal, write a terse
  **procedure** to memory ("to get iron: dig down to y=-50, find iron_ore,
  smelt") so future turns retrieve a known-good recipe instead of re-deriving it.
  This is the cheapest high-value idea: it needs **no new architecture**, just a
  `remember()` convention for procedures.

### 2.5 Plan4MC (PKU-RL) — Mar 2023

- **Paper:** "Plan4MC: Skill Reinforcement Learning and Planning for Open-World
  Minecraft Tasks", arXiv:2303.16563 (2023-03-29, NeurIPS 2023 FMDM workshop).
  Repo: <https://github.com/PKU-RL/Plan4MC>
- **Architecture:** **skills (learned by RL) + planning over a skill graph.**
  Three skill types: **Finding**, **Manipulation**, **Crafting**. RL with
  intrinsic rewards trains each basic skill; an **LLM builds a skill *graph***
  offline (relationships/dependencies between skills); at task time a **graph
  search** walks the skill graph to produce the skill sequence.
- **The transferable artifact — the SKILL GRAPH:** a directed graph where nodes
  are skills/items and edges are "skill A's output is skill B's precondition".
  Built *once, in advance, by an LLM*, then **consulted by cheap graph search**,
  not the LLM, at runtime. This is the single most important data-structure idea
  in the whole literature for Sei (see §3).
- **Long-horizon:** 40 tasks, many requiring **>10 sequential skills**; most
  sample-efficient demonstration-free RL on the tech tree at the time.
- **Self-verification:** each RL skill has a success criterion; the planner
  re-walks the graph on failure.
- **What transfers:** **The skill-graph-as-precomputed-data pattern.** Sei
  cannot do the RL half, but the **dependency graph built once and consulted by
  graph search** is *exactly* a closed-registry-friendly planning structure:
  ship it as JSON, walk it deterministically to compute "to reach goal X, the
  ordered prerequisites are [...]," and inject the next step as context. The LLM
  picks tools; the graph supplies the spine.

### 2.6 Odyssey (Zhejiang Univ — VIPA) — Jul 2024

- **Paper:** "Odyssey: Empowering Minecraft Agents with Open-World Skills",
  arXiv:2407.15325 (2024-07-22). Repo: <https://github.com/zju-vipa/Odyssey>
- **Architecture:** Voyager-lineage but **open-world** (not just diamond). Three
  parts: (1) a **skill library of 40 primitive + 183 compositional skills** (all
  **mineflayer** based), (2) a **fine-tuned LLaMA-3** trained on **390k+ QA pairs
  derived from the Minecraft Wiki** (domain knowledge baked into weights), (3) a
  benchmark (long-term planning, dynamic-immediate planning, autonomous
  exploration).
- **Action/skill abstraction:** primitive skills + **compositional skills**
  (named macros that chain primitives). This is a **closed, curated skill set**
  (unlike Voyager's open code-gen) — closer to Sei.
- **What transfers:** Two things. (1) The **primitive vs compositional split** —
  Sei's ~21 actions are "primitives"; a small number of **named compositional
  macros** (e.g. `makeAndUseFurnace`, `setUpBase`) as *new closed registry
  actions* would mirror Odyssey's 183 compositions without code-gen. (2) The
  **"bake Minecraft Wiki knowledge in"** idea — Sei can't fine-tune Haiku, but it
  can ship the equivalent as **static reference data** (recipes, smelting
  inputs, mob danger table) in the prompt or a retrieved snippet. Odyssey's QA
  dataset is itself a mineable source of compact recipe/knowledge facts.

### 2.7 Mindcraft & mindcraft-ce — 2024–2025 (the mineflayer-native one)

- **Repos:** original <https://github.com/kolbytn/mindcraft> /
  <https://github.com/mindcraft-bots/mindcraft> ("Minecraft AI with
  LLMs+Mineflayer"); community fork **mindcraft-ce**
  <https://github.com/mindcraft-ce/mindcraft-ce> (site mindcraft-ce.com). CE
  arose because the original went stale; CE moves faster, merges community PRs.
- **Architecture (most directly comparable to Sei — same stack):** four parts:
  (1) a **mindserver** launching/managing agents (web UI at :8080), (2) a **main
  agent loop** handling player + inter-agent messages, (3) a **library of
  high-level action commands + observation queries**, (4) a **prompting/model
  layer** over arbitrary LLMs. Classes: `MindcraftAgent`, `History`, `Command`,
  `Prompter`, `Coder`, `Model`.
- **Action/skill abstraction — the important parallel:** **parameterized
  commands**, e.g. `!collectBlocks(material, number)`, `!goToPlayer`,
  `!craftRecipe` — a **closed, named, parameterized command set** invoked from
  chat/LLM output. This is **almost exactly Sei's closed Zod registry**, just
  expressed as `!command(args)` text instead of tool-calls. CE explicitly added
  the gaps Sei lists: **doors, fence gates, better swimming/movement**. It also
  has an **optional `Coder` (JS code-gen) gated behind `allow_insecure_coding`,
  off by default** — i.e. even the mineflayer-native projects treat code-gen as a
  dangerous opt-in, validating Sei's closed-registry stance.
- **CE extras worth noting:** **RAG + realtime Minecraft Wiki search** (lookup
  recipes/mob info on demand), **tool-provider repositories** (community command
  packs), **Andy-4** a local fine-tuned model for keyless users, Docker images.
- **What transfers (high, because same stack):**
  - Sei's furnace/door/sign/combat gaps are **solved-in-public** in mindcraft-ce
    — read their behavior implementations for the **mineflayer mechanics**
    (opening doors, fence gates, furnace smelting via `bot.openFurnace()`,
    placing/reading signs) rather than re-deriving them.
  - **Parameterized commands ≈ Sei tools.** Their command catalog is a checklist
    for "what actions a companion needs."
  - **On-demand Wiki RAG** is a lighter alternative to baking knowledge in —
    Sei could retrieve a recipe snippet only when the model is about to craft
    something it doesn't know.
  - Confirms **code-gen should stay off** for a consumer product (CE defaults it
    off for safety).

### 2.8 Optimus-1 (NeurIPS 2024) & Optimus-2 (CVPR 2025) — HUST/JiuTian-VL

- **Optimus-1:** "Hybrid Multimodal Memory Empowered Agents Excel in
  Long-Horizon Tasks", arXiv:2408.03615 (2024-08-07, NeurIPS 2024). Repo:
  <https://github.com/JiuTian-VL/Optimus-1>. Two memory structures:
  (1) a **Hierarchical Directed Knowledge Graph (HDKG)** encoding world knowledge
  (item/recipe dependencies) the planner consults, and (2) an **Abstracted
  Multimodal Experience Pool (AMEP)** of past experience for in-context
  reference. Components: **Knowledge-guided Planner** + **Experience-Driven
  Reflector** (the reflector decides *when* to replan). Near human-level on many
  long-horizon tasks.
- **Optimus-2:** "Multimodal Minecraft Agent with Goal-Observation-Action
  Conditioned Policy (GOAP)", arXiv:2502.19902 (2025-02, CVPR 2025). Adds a
  **learned low-level policy** π(aₜ | o₁:ₜ, a₁:ₜ₋₁, g) with an action-guided
  behavior encoder; MLLM planner on top. Trained on the **MGOA** dataset (25k
  videos, ~30M observation-goal-action pairs).
- **What transfers:** Optimus-1's **HDKG is another vote for "ship the tech tree
  as a directed knowledge graph the planner reads"** (§3). Its **Experience-Driven
  Reflector** — *a cheap module that decides whether the current plan is still on
  track and triggers replan* — is a good model for Sei: rather than reflecting
  every turn, **only re-plan on a detected discrepancy** (action failed,
  inventory didn't change, hp dropping). Optimus-2's GOAP is the **hybrid
  LLM-planner + learned-controller** pattern (see §5); not directly adoptable but
  the canonical example of where a learned controller slots under an LLM planner.

### 2.9 Altera "Project Sid" / PIANO — Nov 2024 (multi-agent)

- **Paper:** "Project Sid: Many-agent simulations toward AI civilization",
  arXiv:2411.00114 (2024-11-01), Altera.AL. Repo:
  <https://github.com/altera-al/project-sid>
- **Architecture — PIANO (Parallel Information Aggregation via Neural
  Orchestration):** runs **multiple cognitive modules concurrently** (memory,
  planning, social reasoning, motor control, goal generation) that all feed a
  **central decision/cognitive-controller** module which keeps outputs coherent
  across streams. Designed for **real-time** interaction with humans + other
  agents while several slow modules think in parallel. Demonstrated 10–1000+
  agents developing roles (farmer/artist), following + amending laws by voting,
  cultural/religious transmission.
- **What transfers:** Sei is single-agent today but the brief notes
  multi-character (one bot per character) sessions. PIANO's lesson for Sei is
  **decouple the fast "say/react" path from the slow "plan" path** so the bot can
  stay socially responsive while a longer goal proceeds — Sei already does a soft
  version (P1_CHAT preempts P2_MOVEMENT in the FSM; `say` emitted up front). If
  Sei grows multi-bot coordination, PIANO's **central-controller-reconciles-
  parallel-modules** is the reference architecture, and its **emergent role
  specialization from identical agents via social signals** is the design target
  for companion teams. Not a single-layer-loop idea — it is the opposite (many
  modules) — so adopt only the *concurrency/coherence* principle, not the module
  explosion.

### 2.10 Low-level controllers (hybrid hooks — sibling agent covers RL depth)

- **VPT** (OpenAI, 2022): behavior-cloned foundation policy from 70k hrs of
  video; the substrate most controllers fine-tune.
- **STEVE-1** (NeurIPS 2023, arXiv:2306.00937): text/visual-instructable VPT;
  raw mouse/keyboard from pixels; **$60 to train**; completes 12/13 early-game
  tasks. Used as the controller under JARVIS-1-style planners.
- **GROOT, MineCLIP/MineDojo, Optimus-2's GOAP, MrSteve** (What-Where-When
  memory, arXiv:2411.06736): the controller layer that LLM planners delegate to.
- **Relevance to Sei:** Sei's "low-level controller" is **mineflayer's
  pathfinder + behaviors**, not a neural net — and that is *fine*. The hybrid
  pattern (LLM picks high-level goal → non-LLM executes precise control) is
  **already how Sei works**: the LLM calls `goTo`/`gather`/`dig` and mineflayer
  does the physics. Sei is the symbolic-controller version of the same
  LLM-planner-plus-non-LLM-controller split. The takeaway is *don't* try to make
  the LLM emit low-level control; keep widening the **closed action vocabulary**
  (the symbolic controller) instead.

## 3. Planning structures worth copying **as DATA** (not code)

The unifying insight across GITM (sub-goal tree + tech-tree), Plan4MC (skill
graph), Optimus-1 (HDKG), and the crafting-graph community work
(e.g. Minecraft crafting tree in Neo4j, pablissimo.com; redblobgames crafting
graphs) is the same: **encode Minecraft progression once as a directed graph /
table, and have a cheap deterministic walk supply the next step.** The LLM is the
flexible policy *on top of* a rigid spine; it is bad at remembering exact recipe
quantities and tech ordering, and great at adapting once told what's next.

### 3.1 The progression spine as a dependency DAG

Model Minecraft's early-to-mid game as nodes = **capabilities/items**, edges =
**"requires"**. Each node carries: the inputs it consumes, the tool/station it
needs, and how to obtain it (which Sei action). Example (abbreviated):

```
logs --(craft)--> planks --(craft)--> crafting_table
planks --(craft)--> sticks
sticks + planks --(craft @table)--> wooden_pickaxe
wooden_pickaxe --(mine)--> cobblestone --(craft @table)--> stone_pickaxe + furnace
stone_pickaxe --(mine, y<16)--> iron_ore --(smelt @furnace)--> iron_ingot
iron_ingot --(craft @table)--> iron_pickaxe, bucket, shield, armor
iron_pickaxe --(mine, y<16)--> diamond ...
food spine:   any_meat/crops --(smelt/eat)--> restore hunger
shelter spine: planks/dirt --(place/build)--> walls + door + bed --(sleep)--> skip night
```

**Why a DAG, not a tree:** items have multiple parents (cobblestone feeds both
`stone tools` and `furnace`), and the agent should pick whichever prerequisite it
already has progress on — DEPS's "nearest achievable sub-goal" needs the shared
structure of a DAG. Quantities live on edges (`requires {qty: 2}`, `outputs
{n: 4}`) exactly as the Neo4j/Factorio crafting-graph work models them.

### 3.2 How a closed-registry single-layer bot *uses* it

The bot does **not** run a planner. Instead, each turn:

1. A **pure function** `nextMilestone(inventory, nearby, stations, goal)` walks
   the DAG backward from the current goal (or a default "tech up" goal),
   returns the **first unsatisfied prerequisite whose own inputs are satisfied**
   (a topological "ready frontier"), and the **single action** that advances it.
2. That becomes **one extra snapshot line**, e.g.:
   `next: to get a stone_pickaxe you still need 3 cobblestone — mine cobblestone
   (you have a wooden_pickaxe). after: craft furnace, then mine iron at y<16.`
3. The LLM is free to ignore it (player asked for something else) — it's a
   **hint, not a command** — preserving the single-layer "model decides"
   contract. This mirrors GITM's tech-tree-ordered decomposition and Plan4MC's
   graph walk, but the *walk is deterministic code*, not an LLM call, so it costs
   nothing and never hallucinates a recipe.

### 3.3 Curriculum / "what do I need before X"

Voyager's **environment-aware automatic curriculum** reduces, for Sei, to the
same `nextMilestone()` function used in "proactive" mode: when idle and
proactiveness ≥ Agentic, the bot can adopt the ready-frontier node as a
self-set goal ("I'll go get us some iron"). The DAG **is** the curriculum —
walking it forward from current state yields the next sensible objective, and it
is automatically "achievable now" because the frontier is defined as
inputs-satisfied. This is the cheapest way to get Voyager-style self-directed
progression without code-gen or a curriculum LLM.

### 3.4 Knowledge as static reference (recipes, smelting, mob danger)

Mirror Odyssey's wiki-QA and CE's wiki-RAG as **small static tables shipped with
the bot** (or retrieved on demand):

- **Smelting table:** `iron_ore→iron_ingot (fuel: coal/charcoal/planks)`,
  `raw_food→cooked_food`, `sand→glass`, `cobblestone→stone`. Directly fills the
  **furnace gap**: a `smelt(input, fuel, count)` action + this table is a
  complete furnace feature.
- **Mob danger / behavior table:** hostile vs passive, day/night spawn, ranged
  vs melee, drops. Fills the **proactive mob awareness + combat** gaps: the
  snapshot's `nearby entities` can be annotated `zombie (hostile, melee)` and the
  bot told to react.

These are data, not architecture — they slot into the existing prompt/snapshot.

## 4. Self-verification & recovery in a single-layer loop

The papers verify with either a **second LLM critic** (Voyager) or a
**Describe→Explain** text loop (DEPS) or a **Reflector that decides when to
replan** (Optimus-1). Sei cannot afford a second Haiku call per action, and
*does not need one* — **the environment already returns ground truth**. Sei's
edge is that its tools return real results and its snapshot diffs state.

### 4.1 Make tool results self-verifying (the central recommendation)

Adopt a **uniform structured tool-result convention** so every action's
`last_action_result:` answers "did it work, and if not, what's missing?" Sei
already does this in spots (`unfollow` reads back the follow target; `craft`
reports what was actually made; `find` returns `{found:false, reason}`).
Generalize it to a small typed shape every behavior returns:

```
{ ok: true|false,
  effect: "mined 3 cobblestone",          // what actually changed (verifiable vs inventory delta)
  reason: "no crafting_table in reach",   // WHY it failed (DEPS "explain")
  fix: "place a crafting_table first" }    // the missing precondition / next step
```

Rendered into one snapshot line:
`last_action_result: craft stone_pickaxe FAILED — needs a crafting_table in
reach. fix: place crafting_table, then retry.`

This collapses Voyager's critic and DEPS's explainer into **zero extra LLM
calls**: the *action handler* (which knows the mineflayer failure) emits the
critique. The single-layer model reads it next turn and self-corrects. This is
the highest-leverage, lowest-cost idea in the whole report.

### 4.2 Verify against state deltas, not the model's word

Sei already emits `*** INVENTORY JUST CHANGED ***` and `recent_events:`. Lean on
them as the **success oracle**: an action claims `mined 3 cobblestone`; the
inventory delta confirms `+3 cobblestone` or the model sees the mismatch
(`said it mined but inventory unchanged` → infer drop not collected / wrong
block). This is the JARVIS-1 / Optimus "check outcome against observation"
discipline, done with diffs Sei computes anyway.

### 4.3 Recovery: precondition checks + bounded retry, not open reflection

- **Precondition gating (cheapest):** before an action can fail expensively,
  the **handler** checks the obvious precondition and returns the `fix` string
  (no LLM). E.g. `craft` needing a 3×3 recipe returns "need crafting_table"
  immediately — the `craftable:` line already encodes this; extend it to *all*
  station-gated actions (furnace, anvil).
- **Bounded retry with explanation:** on failure, the next turn's context
  carries the `reason`/`fix`. Cap retries of the *same failing action on the
  same target* (a small per-target counter) and, on exhaustion, inject "this
  isn't working — try a different approach" (mirrors DEPS's replan trigger and
  Optimus-1's reflector firing only on discrepancy). Sei already has analogous
  nudges (follow-stuck, idle "look then explore") — generalize them.
- **Reflect only on discrepancy** (Optimus-1 Reflector): do not add per-turn
  self-critique (kills latency/credits and Haiku goes mute under heavy
  reflection per Sei's own `say`-tool finding). Trigger a "step back and replan"
  turn only when a discrepancy is detected (action failed twice / inventory not
  advancing / hp dropping / goal stalled N turns).

## 5. Hybrid hooks: LLM-as-planner + non-LLM controller

| System | High level (LLM) | Low level (non-LLM controller) |
|---|---|---|
| DEPS | LLM planner + selector | goal-conditioned RL controller |
| Plan4MC | graph search over skill graph | RL skills (Find/Manipulate/Craft) |
| GITM | LLM decomposer/planner | structured-action → mouse/keyboard interface (no RL) |
| JARVIS-1 | multimodal LLM planner + memory | STEVE-1-class learned controller |
| Optimus-2 | MLLM planner | **GOAP** learned policy π(a\|o,a,g) |
| **Sei** | **single LLM turn picks a closed tool** | **mineflayer pathfinder + behaviors (symbolic)** |

**Key observation:** Sei *already is* an instance of the dominant pattern —
LLM-as-high-level-planner over a non-LLM controller. The difference is Sei's
controller is **classical (mineflayer)** rather than **learned (RL/VPT)**. For a
consumer desktop product this is the right call: deterministic, debuggable, no
GPU, no training. The roadmap implication is **invest in the symbolic
controller's vocabulary and robustness** (more closed actions: smelt, useDoor,
placeSign, fight; tighter precondition/timeout wrappers like the existing
pathfinder wall-clock guard) rather than chasing learned controllers. A learned
controller (STEVE-1/GOAP) would only matter if Sei ever needed pixel-level
combat or parkour the pathfinder can't express — explicitly out of scope for a
Haiku-class desktop companion. (Sibling agent covers RL depth.)

## 6. Adopt for Sei (closed-registry, single-layer) — ROADMAP-READY

### 6.1 Recommended progression/curriculum data structure

Ship a **static progression DAG as JSON** (`progression.json`, loaded by the bot
adapter), plus a **pure `nextMilestone()` walker**. Node schema:

```jsonc
{
  "id": "stone_pickaxe",
  "kind": "item",                       // item | station | capability
  "requires": [                          // edges, with quantities
    { "id": "cobblestone", "qty": 3 },
    { "id": "stick", "qty": 2 },
    { "id": "crafting_table", "station": true }
  ],
  "obtain": { "action": "craft", "args": { "item": "stone_pickaxe" } },
  "unlocks": ["furnace", "stone_axe", "iron_tier"]
}
```

- Covers the **spine the brief names**: wood → tools → food → shelter → iron →
  (diamond). Include parallel food and shelter sub-DAGs so "survive the night"
  and "tech up" are both walkable.
- `nextMilestone(state, goal)` = topological walk returning the **ready frontier
  node** (all inputs satisfied) nearest the goal, plus the single Sei action to
  advance it. **Deterministic, no LLM, no hallucinated recipes.** This is
  Plan4MC's skill-graph-walk + GITM's tech-tree ordering + Optimus-1's HDKG, in
  the form a closed-registry single-layer bot can actually consume.
- Surface as **one snapshot line** (`next:`), advisory not mandatory, preserving
  the model-decides single-layer contract. In proactive/idle mode, the frontier
  node becomes a self-set goal (Voyager curriculum, for free).

### 6.2 Self-verification tool-result convention

Standardize **every** behavior's return to `{ ok, effect, reason, fix }` and
render it into `last_action_result:` as
`<action> <OK|FAILED> — <effect|reason>. <fix?>`. Pair with **handler-side
precondition checks** that emit `fix` *before* an expensive failure (extend the
existing `craftable:` station-gating to furnace/anvil/bed). Verify claimed
effects against the **inventory/hp deltas Sei already computes**. Net result:
Voyager's critic + DEPS's explainer + Optimus's reflector, at **zero extra LLM
calls** — the environment is the verifier. Reflect/replan only on a **detected
discrepancy** (failed-twice / no-progress / hp-drop), never every turn (protects
Haiku latency and avoids the mute-under-reflection failure Sei already
documented).

### 6.3 The 3–5 highest-leverage planning ideas that fit the constraints

1. **Static progression DAG + `nextMilestone()` walker → one `next:` snapshot
   line.** (Plan4MC/GITM/Optimus-1.) Gives long-horizon coherence and a free
   curriculum without a planner module or code-gen. **Highest structural
   leverage.**
2. **`{ok,effect,reason,fix}` self-verifying tool results + precondition
   gating.** (Voyager critic / DEPS explain, collapsed into handlers.) Turns
   blind retries into informed self-correction at zero added cost. **Highest
   cost/benefit.**
3. **Procedural memory: `remember()` known-good recipes after a multi-step
   success.** (JARVIS-1 plan retrieval, on Sei's existing `MEMORY.md`.) "How I
   got iron last time" is retrieved instead of re-derived — reliability
   compounds across sessions, fits per-world memory segmentation already in
   place.
4. **Close the named gaps as new closed actions + static knowledge tables, not
   code-gen.** Add `smelt`/furnace (+ smelting table), `useDoor`/`useGate`,
   `placeSign`/`readSign`, and combat upgrades (proactive hostile annotation in
   `nearby entities` via a mob-danger table; reactive `attackEntity`/flee).
   **Borrow mineflayer mechanics directly from mindcraft-ce** — these are
   solved-in-public on the identical stack (doors, fence gates, furnace,
   swimming). This is Odyssey's "primitive + a few compositional skills" and CE's
   parameterized-command catalog. **Highest feature leverage; directly clears the
   brief's gaps.**
5. **Discrepancy-triggered replan + bounded per-target retry.** (Optimus-1
   Reflector / DEPS replan / DEPS selector "nearest-achievable".) A small
   per-target failure counter + "this isn't working, change approach" nudge, plus
   ordering candidate next-steps by inputs-already-satisfied. Generalizes Sei's
   existing stuck-nudges into a uniform recovery policy. **Highest robustness
   leverage.**

**Explicitly do NOT adopt:** Voyager/Mindcraft **code-generation skill
libraries** (violate the closed registry; even mineflayer-native CE defaults
code-gen *off* for safety); **learned RL/VPT controllers** (no GPU/training on a
desktop; mineflayer's symbolic controller is the right substrate); **PIANO's
many-module concurrency** (opposite of single-layer — revisit only if multi-bot
coordination becomes a milestone, and then only the fast-react/slow-plan
decoupling, which Sei's FSM already approximates).
